# Spring、SpringBoot、SpringCloud 三者区别

---

## 🧠 联想记忆法

### 记忆口诀/联想

**核心口诀：「地基 → 精装房 → 小区生态」**

| 框架 | 口诀映射 | 一句话要点 |
|------|---------|-----------|
| **Spring** | 毛坯房的地基和框架 | 提供 IoC 容器（控制反转）和 AOP（面向切面编程），但需手动配置每一项 |
| **Spring Boot** | 精装公寓，拎包入住 | 自动配置（Auto-Configuration）+ 起步依赖（Starter），内嵌 Tomcat，零 XML |
| **Spring Cloud** | 智慧小区全套配套设施 | 注册中心、配置中心、网关、负载均衡、熔断降级——微服务一站式解决方案 |

### 记忆原理

该口诀利用 **建筑类比法（Architecture Analogy）** 这一认知心理学中的强记忆锚点（Memory Anchor）：

- **Spring 是地基与框架**：如同建造一栋建筑，Spring 提供了最核心的承重结构——IoC 容器（相当于"梁柱"）和 AOP 切面（相当于"管线预埋"）。但地基之上的一切（墙体、门窗、水电）都需要开发者自己搭建，反映到工程中就是大量的 XML 和 Java Config 配置。
- **Spring Boot 是精装房**：精装房在交付时已完成硬装和软装——墙面刷好了（自动配置）、水电入户了（内嵌 Tomcat）、家电齐全了（Starter 依赖）。开发者作为"住户"只需搬入家具（业务代码），无需从零装修（零配置开发）。
- **Spring Cloud 是小区生态**：单一的精装房（Spring Boot 单体应用）住起来舒适，但当一个"小区"（分布式系统）需要多栋楼（微服务）协同运转时，就需要基础设施——物业中心（服务注册与发现 Nacos/Eureka）、自来水公司（配置中心）、快递站（API 网关）、电网调度（负载均衡 LoadBalancer）、备用发电机（熔断降级 Sentinel）。

**三层递进关系**：精装房不能脱离地基独立存在，小区生态不能脱离精装房独立存在——这正是 Spring Boot 依赖 Spring、Spring Cloud 依赖 Spring Boot 的传递关系。

### 关联知识

- **Maven/Gradle 的传递依赖（Transitive Dependency）**：Spring Boot Starter 的本质是 Maven 的传递依赖打包。理解 Maven 的依赖传递机制，就能理解为何引入一个 `spring-boot-starter-web` 就能同时获得 Spring MVC、Jackson、内嵌 Tomcat 等数十个依赖。
- **传统 Java Web 部署流程**：回忆"编写 Servlet → 配置 web.xml → 打包 WAR → 部署到外部 Tomcat"的繁琐流程，对比 Spring Boot 的 `java -jar app.jar` 一键启动，深刻理解"约定大于配置"（Convention over Configuration）的设计哲学。
- **分布式系统理论 CAP/BASE**：Spring Cloud 的组件选型背后是分布式系统理论——Nacos 支持 AP（可用性+分区容忍性）和 CP（一致性+分区容忍性）双模式切换；Eureka 采用 AP 架构放弃强一致性。将这两者关联，面试中可展示系统性的知识深度。

## 📖 深度解答

### 一、核心概念：三者的定位和关系（不是替代而是递进）

Spring（Spring Framework）、Spring Boot、Spring Cloud 三者**不是替代关系（Replacement），而是递进关系（Progression）**，分别对应 Java 企业级开发的不同抽象层次：

| 维度 | Spring | Spring Boot | Spring Cloud |
|------|--------|-------------|--------------|
| **英文定位** | Foundation Framework | Rapid Development Scaffold | Microservice Solution Ecosystem |
| **核心价值** | 解耦（Decoupling） | 自动化（Automation） | 治理（Governance） |
| **关注层次** | 对象管理、依赖管理 | 项目搭建、配置管理 | 服务治理、分布式协调 |
| **配置方式** | XML + 注解 + Java Config | application.yml + 自动配置 | bootstrap.yml + 配置中心 |
| **部署方式** | WAR 部署到外部容器 | Fat JAR 内嵌容器，java -jar | Docker 容器化 + 服务编排 |
| **适用范围** | 单体应用 | 单体 / 微服务均可 | 分布式微服务集群 |
| **启动入口** | XML/AnnotationConfigApplicationContext | @SpringBootApplication | @EnableXxx 注解激活组件 |

**核心结论**：Spring 是做"能力抽象"的——将对象管理、事务管理、数据访问等能力抽象为统一接口（IoC Container, PlatformTransactionManager, JdbcTemplate 等）。Spring Boot 是做"体验封装"的——将 Spring 的使用体验从"复杂"变为"简单"。Spring Cloud 是做"生态整合"的——将分布式系统的共性难题以统一的方式解决。

**依赖关系链条**：`Spring Cloud Application` -> `Spring Boot` -> `Spring Framework`。Spring Cloud 的每个微服务实例本质上就是一个 Spring Boot 应用，而 Spring Boot 底层完全依赖 Spring Framework 的 IoC 容器和 AOP 能力。

### 二、底层原理

#### 2.1 Spring 核心：IoC 容器 + DI + AOP + Bean 生命周期管理

**IoC（Inversion of Control，控制反转）** 是 Spring 最核心的设计思想，也是整个 Spring 生态体系的基石。传统编程模式下，对象主动创建和管理自己的依赖（`UserService userService = new UserServiceImpl()`），对象的创建权和生命周期完全由开发者控制。IoC 将这个权力"反转"给了容器——开发者只需声明依赖关系，容器负责对象的创建、组装和管理。

```java
// 传统方式：程序员主动控制对象创建和依赖管理
public class UserController {
    private UserService userService;
    
    public UserController() {
        // 程序员手动创建依赖
        this.userService = new UserServiceImpl();
    }
}

// Spring IoC 方式：容器代为管理，开发者声明依赖关系
@RestController
public class UserController {
    @Autowired  // 容器自动注入依赖
    private UserService userService;
}
```

**DI（Dependency Injection，依赖注入）** 是 IoC 的具体实现手段。Spring 提供三种依赖注入方式：
- **构造器注入（Constructor Injection）**：Spring 官方推荐，保证依赖不可变（Immutable）和 Bean 的完整初始化状态。适用于必需依赖（Mandatory Dependency）。
- **Setter 注入（Setter Injection）**：通过 Setter 方法注入，适用于可选依赖（Optional Dependency）。
- **字段注入（Field Injection）**：通过 `@Autowired` 直接注入字段，代码最简洁但不推荐在生产中使用——无法进行单元测试（无法 Mock 私有字段），且隐藏了依赖关系。

**AOP（Aspect-Oriented Programming，面向切面编程）** 将横切关注点（Cross-Cutting Concerns）从业务逻辑中剥离并模块化。典型的横切关注点包括：事务管理（`@Transactional`）、方法耗时监控、权限校验、日志记录。AOP 的底层实现依赖于 **代理模式（Proxy Pattern）**：
- **JDK 动态代理（JDK Dynamic Proxy）**：目标类实现了接口时使用，基于 `java.lang.reflect.Proxy`，运行时生成代理类。
- **CGLIB 代理（CGLIB Proxy）**：目标类未实现接口时使用，通过字节码生成（Bytecode Generation）创建目标类的子类作为代理。

**Bean 生命周期（Bean Lifecycle）** 是 Spring 容器管理对象的核心流程：

```
BeanDefinition加载 → 实例化(Instantiation)
    → 属性赋值(Populate Properties)
    → Aware 接口回调(BeanNameAware, ApplicationContextAware等)
    → BeanPostProcessor 前置处理(postProcessBeforeInitialization)
    → @PostConstruct / InitializingBean.afterPropertiesSet()
    → BeanPostProcessor 后置处理(postProcessAfterInitialization) → 【AOP代理在此生成】
    → Bean 就绪(Ready for Use)
    → 容器关闭 → @PreDestroy / DisposableBean.destroy()
```

面试加分点：**BeanPostProcessor 的后置处理阶段**是 AOP 代理的生成时机——`AbstractAutoProxyCreator` 在此阶段检查 Bean 是否需要被代理，如果需要则返回代理对象而非原始对象。

#### 2.2 Spring Boot 核心：约定大于配置 + 自动配置 + 起步依赖

**约定大于配置（Convention over Configuration）** 是 Spring Boot 最核心的设计哲学。Spring Boot 预置了大量合理的默认值（Sensible Defaults），开发者只需在偏离默认行为时才提供配置：

| 配置项 | Spring Boot 默认值 | 自定义方式 |
|--------|-------------------|-----------|
| 静态资源路径 | `classpath:/static/` | `spring.web.resources.static-locations` |
| 视图模板前缀 | `classpath:/templates/` | `spring.thymeleaf.prefix` |
| 服务器端口 | 8080 | `server.port` |
| 数据源类型 | HikariCP | `spring.datasource.type` |

**自动配置（Auto-Configuration）** 是 Spring Boot 最核心的底层机制。其完整实现链路如下：

1. `@SpringBootApplication` 是一个组合注解（Composed Annotation），包含：
   - `@EnableAutoConfiguration`：启动自动配置机制
   - `@ComponentScan`：扫描当前包及其子包的 `@Component` 及其衍生注解
   - `@SpringBootConfiguration`：等同于 `@Configuration`

2. `@EnableAutoConfiguration` 通过 `@Import(AutoConfigurationImportSelector.class)` 导入配置选择器。`AutoConfigurationImportSelector` 的 `getCandidateConfigurations()` 方法扫描 classpath 下所有 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件（Spring Boot 3.x 使用此文件；Spring Boot 2.x 使用 `META-INF/spring.factories`）。

3. 读取到的自动配置类（Auto-Configuration Class）经过 **条件注解（Conditional Annotation）** 的过滤，只有满足条件的配置才会生效：

```java
@Configuration
@ConditionalOnClass(DataSource.class)               // 类路径存在 DataSource
@ConditionalOnMissingBean(DataSource.class)          // 容器中没有自定义 DataSource
@ConditionalOnProperty(prefix = "spring.datasource", name = "url")  // 配置了数据源URL
@EnableConfigurationProperties(DataSourceProperties.class)
public class DataSourceAutoConfiguration {
    
    @Bean
    @ConditionalOnMissingBean
    public DataSource dataSource(DataSourceProperties properties) {
        // 根据 application.properties 中的配置自动创建数据源
        HikariDataSource dataSource = properties.initializeDataSourceBuilder()
            .type(HikariDataSource.class).build();
        return dataSource;
    }
}
```

关键条件注解及其含义：

| 注解 | 条件判断逻辑 |
|------|-------------|
| `@ConditionalOnClass` | 指定类在 classpath 中存在 |
| `@ConditionalOnMissingClass` | 指定类在 classpath 中不存在 |
| `@ConditionalOnBean` | 容器中存在指定 Bean |
| `@ConditionalOnMissingBean` | 容器中不存在指定 Bean（用户自定义优先级高于自动配置） |
| `@ConditionalOnProperty` | 配置文件中存在指定属性 |
| `@ConditionalOnWebApplication` | 当前环境为 Web 应用 |
| `@ConditionalOnExpression` | SpEL 表达式结果为 true |

**起步依赖（Starter）** 是 Spring Boot 项目构建的核心创新。每个 Starter 是一个 Maven POM 模块，打包了一组功能相关的依赖，并利用 Maven 的传递依赖（Transitive Dependency）机制自动引入所有子依赖：

```xml
<!-- pom.xml：引入一个 Starter，获得全套 Web 开发能力 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
<!-- 等效于手动管理以下依赖的版本：
  - spring-webmvc (Spring MVC 框架)
  - tomcat-embed-core (内嵌 Tomcat 服务器)
  - jackson-databind (JSON 序列化/反序列化)
  - hibernate-validator (参数校验)
  - spring-boot-autoconfigure (自动配置核心)
  - spring-web (HTTP 客户端和服务端)
  - spring-boot-starter (Spring Boot 基础)
-->
```

**常见 Starter 分类**：

| Starter | 功能 |
|---------|------|
| `spring-boot-starter-web` | Web 应用（Servlet 栈，内嵌 Tomcat） |
| `spring-boot-starter-webflux` | 响应式 Web 应用（WebFlux 栈，内嵌 Netty） |
| `spring-boot-starter-data-jpa` | JPA + Hibernate 数据访问 |
| `spring-boot-starter-data-redis` | Redis 数据访问 |
| `spring-boot-starter-security` | Spring Security 安全框架 |
| `spring-boot-starter-test` | 测试框架（JUnit 5 + Mockito + AssertJ） |
| `mybatis-spring-boot-starter` | MyBatis 数据访问（第三方） |

**内嵌 Servlet 容器（Embedded Servlet Container）**：Spring Boot 将应用打包为可执行 JAR（Fat JAR / Uber JAR），将 Tomcat、Jetty 或 Undertow 等 Servlet 容器内嵌到应用中，通过 `java -jar` 直接启动，无需部署 WAR 到外部容器。内嵌容器的自动配置由 `ServletWebServerFactoryAutoConfiguration` 完成，默认使用 Tomcat，可通过引入 `spring-boot-starter-jetty` 或 `spring-boot-starter-undertow` 切换。

#### 2.3 Spring Cloud 核心：微服务一站式解决方案

Spring Cloud 构建在 Spring Boot 之上，提供分布式微服务的全套基础设施（Infrastructure）。其核心组件体系如下：

**（1）服务注册与发现（Service Registry and Discovery）**

服务注册中心是微服务体系的核心组件，负责维护所有服务实例的元信息（IP、端口、健康状态），允许服务消费者动态发现服务提供者的地址，实现服务间的解耦。

```yaml
# application.yml — 服务提供者注册到 Nacos
spring:
  application:
    name: order-service           # 服务名，服务消费者通过此名称调用
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848  # Nacos 服务器地址
        namespace: dev                # 命名空间隔离（环境隔离）
        group: DEFAULT_GROUP
server:
  port: 8081
```

```java
// 服务消费者 — 通过 OpenFeign 声明式调用远程服务
@SpringBootApplication
@EnableFeignClients(basePackages = "com.example.client")
public class ConsumerApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConsumerApplication.class, args);
    }
}

// 声明式远程调用接口 — 像调用本地方法一样调用远程服务
@FeignClient(
    name = "order-service",           // 对应服务提供者的 spring.application.name
    path = "/order",                  // 统一前缀
    fallbackFactory = OrderFallbackFactory.class  // 熔断降级工厂
)
public interface OrderFeignClient {
    @GetMapping("/{id}")
    Result<Order> getOrderById(@PathVariable("id") Long id);
    
    @PostMapping("/create")
    Result<Long> createOrder(@RequestBody OrderDTO orderDTO);
}
```

**（2）配置中心（Configuration Center）**

配置中心将配置从微服务实例中剥离，集中管理并支持动态刷新。生产环境推荐使用 Nacos Config 替代 Spring Cloud Config：

```yaml
# bootstrap.yml — 应用启动时优先从配置中心拉取配置
spring:
  cloud:
    nacos:
      config:
        server-addr: 127.0.0.1:8848
        file-extension: yaml
        namespace: dev
        group: DEFAULT_GROUP
        refresh-enabled: true         # 支持配置动态刷新
  config:
    import: nacos:application.yaml    # 导入配置中心的配置
```

```java
// 配置动态刷新 — 不重启应用即可更新配置
@RestController
@RefreshScope  // 配置刷新时重新创建 Bean
public class ConfigController {
    
    @Value("${order.timeout:5000}")
    private Integer orderTimeout;
    
    @GetMapping("/config/timeout")
    public Integer getTimeout() {
        return orderTimeout;
    }
}
```

**（3）负载均衡（Load Balancing）**

Spring Cloud LoadBalancer 是 Ribbon 的官方替代品（Ribbon 已进入维护模式）。通过 Feign 的 `@FeignClient` 注解声明远程调用时，自动结合 LoadBalancer 实现客户端负载均衡。

**（4）API 网关（API Gateway）**

Spring Cloud Gateway 基于 WebFlux（Reactor 模型，非阻塞 I/O），提供路由转发、鉴权、限流、请求聚合等功能。它的性能显著优于基于 Servlet 阻塞 I/O 的 Zuul 1.x：

```yaml
# application.yml — Spring Cloud Gateway 路由配置
spring:
  cloud:
    gateway:
      routes:
        - id: order-service-route
          uri: lb://order-service            # lb:// 表示启用负载均衡
          predicates:
            - Path=/api/order/**              # 路径匹配断言
            - Method=GET,POST                 # HTTP 方法限定
          filters:
            - StripPrefix=1                   # 去除 /api 前缀
            - AddRequestHeader=X-Gateway-Source, gateway
            - name: RequestRateLimiter        # 限流过滤器
              args:
                redis-rate-limiter:
                  replenishRate: 10           # 令牌桶填充速率
                  burstCapacity: 20           # 令牌桶容量
```

**（5）熔断降级（Circuit Breaking and Degradation）**

Sentinel（Alibaba 出品）是 Hystrix 的替代方案，以流量为切入点，提供实时监控、熔断降级、系统自适应保护、热点参数限流等功能：

```java
@RestController
public class OrderController {
    
    @PostMapping("/order/create")
    @SentinelResource(
        value = "createOrder",                    // 资源名称（唯一标识）
        blockHandler = "createOrderBlockHandler", // 限流/熔断后的降级方法
        fallback = "createOrderFallback"           // 业务异常后的 fallback
    )
    public Result<Long> createOrder(@RequestBody OrderDTO order) {
        return Result.success(orderService.create(order));
    }
    
    // 限流/熔断降级方法 — 方法签名必须与原始方法一致，增加 BlockException 参数
    public Result<Long> createOrderBlockHandler(OrderDTO order, BlockException ex) {
        // 记录限流日志
        log.warn("createOrder 被限流，参数：{}", order);
        return Result.error(429, "系统繁忙，请稍后重试");
    }
    
    // 业务异常 fallback — 不需要 BlockException 参数
    public Result<Long> createOrderFallback(OrderDTO order, Throwable ex) {
        return Result.error(500, "服务异常：" + ex.getMessage());
    }
}
```

#### 3.1 演进关系图（文字描述）

```
2014+  ┌──────────────────────────────────────────────────────┐
       │                 Spring Cloud Ecosystem              │
       │  ┌──────────┐ ┌────────┐ ┌────────┐ ┌──────────┐  │

```

**演进驱动力**：每一次演进都解决了上一阶段的核心痛点：
- **Spring** 解决了 EJB（Enterprise JavaBean）的过度复杂性问题，提出轻量级 IoC 容器
- **Spring Boot** 解决了 Spring 项目配置繁琐（XML Boilerplate）和部署复杂的问题
- **Spring Cloud** 解决了单体架构难以扩展、服务间通信复杂、分布式治理困难的问题

#### 3.2 实践对比：同样功能的配置工作量

**场景**：搭建一个 RESTful Web 服务 + MySQL 数据库访问

**Spring 时代（纯 XML + 注解混合配置）：**

```xml
<!-- web.xml — 必须的 Servlet 容器配置 -->
<web-app xmlns="http://java.sun.com/xml/ns/javaee" version="3.0">
    <listener>
        <listener-class>org.springframework.web.context.ContextLoaderListener</listener-class>
    </listener>
    <context-param>
        <param-name>contextConfigLocation</param-name>
        <param-value>/WEB-INF/applicationContext.xml</param-value>
    </context-param>
    <servlet>
        <servlet-name>dispatcher</servlet-name>
        <servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
        <init-param>
            <param-name>contextConfigLocation</param-name>
            <param-value>/WEB-INF/dispatcher-servlet.xml</param-value>
        </init-param>
        <load-on-startup>1</load-on-startup>
    </servlet>
    <servlet-mapping>
        <servlet-name>dispatcher</servlet-name>
        <url-pattern>/</url-pattern>
    </servlet-mapping>
</web-app>
```

```xml
<!-- dispatcher-servlet.xml — Spring MVC + 数据源配置 -->
<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xmlns:context="http://www.springframework.org/schema/context"
       xmlns:mvc="http://www.springframework.org/schema/mvc"
       xsi:schemaLocation="...">
    
    <mvc:annotation-driven/>
    <context:component-scan base-package="com.example"/>
```

### 四、深入思考

#### 4.1 版本对应关系（Spring Boot 2.x/3.x vs Spring Cloud）

Spring Boot 与 Spring Cloud 之间存在 **严格的版本兼容性约束**。Spring Cloud 早期版本使用伦敦地铁站名作为版本号（按字母顺序递增），从 2020.0.x 开始改用日历版本（Calendar Versioning）：

| Spring Boot 版本 | JDK 要求 | Spring Cloud 版本 | 代号 | 关键变更 |
|-----------------|----------|-------------------|------|---------|
| 2.0.x | 8+ | Finchley | F | 首个基于 Boot 2.x 的 Cloud 版本 |
| 2.1.x | 8+ | Greenwich | G | 支持 Spring Boot 2.1 新特性 |
| 2.2.x | 8+ | Hoxton.SR12 | H | Netflix 组件标记为维护模式，建议迁移 |
| 2.3.x | 8+ | Hoxton.SR+ | H | 最后支持 Java 8 的 Cloud 系列 |
| 2.4.x ~ 2.5.x | 8+ | 2020.0.x (Ilford) | I | **改为日历版本命名**，兼容 Spring Boot 2.4.x/2.5.x |
| 2.6.x ~ 2.7.x | 8+ | 2021.0.x (Jubilee) | J | **移除了已弃用的 Netflix 组件**（Ribbon、Hystrix 等） |
| **3.0.x** | **17+** | **2022.0.x (Kilburn)** | K | **重大版本断崖**：JDK 17 基线、Jakarta EE 9+ 命名空间迁移、GraalVM 原生镜像 |
| 3.1.x | 17+ | 2022.0.x | K | 支持虚拟线程（Virtual Threads, Project Loom） |
| 3.2.x | 17+ | 2023.0.x (Leyton) | L | 继续演进，优化 AOT 编译 |
| 3.3.x | 17+ | 2023.0.x | L | 安全更新和 Bug 修复 |
| 3.4.x | 17+ | 2024.0.x (Mays) | M | 最新长期支持版本 |

**版本管理——BOM（Bill of Materials）：**

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-dependencies</artifactId>
            <version>2023.0.0</version>  <!-- 对应 Spring Boot 3.2.x -->
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

**重要提示**：
1. Spring Boot 3.x 要求 **JDK 17+**，不再支持 JDK 8/11，这是 Java 生态近十年最大的版本断崖
2. Spring Boot 3.x 使用 **Jakarta EE 命名空间**（`javax.*` → `jakarta.*`），升级时需全局替换
3. Spring Boot 2.x 的 Cloud 组件（Eureka、Hystrix、Ribbon、Zuul）已全面进入维护/弃用状态，新项目应直接采用 **Spring Boot 3.x + Nacos + Sentinel + Gateway + OpenFeign** 的技术栈

#### 4.2 面试追问点与深度扩展

**追问 1：Spring Boot 自动配置的完整加载链路是什么？**
> 回答要点：`@SpringBootApplication` -> `@EnableAutoConfiguration` -> `@Import(AutoConfigurationImportSelector)` -> `SpringFactoriesLoader.loadFactoryNames()` (Boot 2.x) 或 `AutoConfigurationImportSelector.getCandidateConfigurations()` (Boot 3.x 从 `AutoConfiguration.imports` 文件读取) -> 读取 `AutoConfiguration` 类列表 -> 经过 `@Conditional` 链式条件判断 -> 符合条件的配置类被加载 -> 执行 `@Bean` 方法创建自动配置的 Bean。

**追问 2：谈谈 Spring Boot 3.x 相对 2.x 的主要变化和升级注意事项**
> 回答要点：JDK 17 基线、Jakarta EE 命名空间迁移（`javax.persistence.*` 变为 `jakarta.persistence.*`）、`spring.factories` 被 `AutoConfiguration.imports` 替代、原生镜像（Native Image / AOT）支持、虚拟线程（Virtual Threads）预览、Observability（可观测性）的 Micrometer 化。升级时需注意：JDK 版本先升级到 17+ → 替换 Java EE 依赖为 Jakarta EE → 替换 `spring.factories` 为 `AutoConfiguration.imports` → 验证第三方依赖的兼容性版本。

**追问 3：Spring Cloud 组件选型——为什么从 Netflix 迁移到 Alibaba？**
> 回答要点：Eureka 2.x 已闭源不再迭代，Nacos 同时提供注册中心和配置中心且生态活跃；Hystrix 进入维护模式且功能单一，Sentinel 提供流量控制、熔断降级、系统保护、热点限流等更丰富的功能；Zuul 1.x 基于 Servlet 阻塞 I/O 性能有限且不再维护，Gateway 基于 WebFlux 非阻塞 I/O 性能显著提升。阿里巴巴在中文互联网场景下经过大规模生产验证（双十一），成熟度和社区活跃度更高。

**追问 4：Spring Boot 自定义 Starter 的实现步骤？**
> 回答要点：① 创建自动配置类（`@Configuration` + `@ConditionalOnClass`/`@ConditionalOnMissingBean` + `@EnableConfigurationProperties`） ② 创建配置属性绑定类（`@ConfigurationProperties(prefix = "xxx")`） ③ 在 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 中注册自动配置类（Spring Boot 3.x） ④ 提供 spring-boot-autoconfigure-processor 生成配置元数据（需引入 `spring-boot-configuration-processor`）

**追问 5：微服务拆分粒度的原则是什么？过度拆分会带来什么问题？**
> 回答要点：依据 **DDD（Domain-Driven Design，领域驱动设计）** 的**限界上下文（Bounded Context）** 进行拆分。每个微服务应该对应一个独立的业务子域（Subdomain）。过度拆分会导致"分布式单体"（Distributed Monolith）——拆分过细导致服务间通信激增、分布式事务次数增加、部署协调成本急剧上升，最终你虽然形式上有了微服务，但本质上却是一个需要更多基础设施的单体应用。

### 常见误区

1. **"Spring Boot 取代了 Spring"** —— 错误。Spring Boot 基于 Spring Framework，两者是增强关系（Enhancement），而非替代关系（Substitution）。Spring Boot 项目中引用的 `spring-boot-starter-*` 依赖会传递引入 Spring Framework 的核心 JAR。
2. **"Spring Cloud 是 Spring Boot 的升级版"** —— 错误。两者解决的问题不同，Spring Cloud 是微服务治理生态，Spring Boot 是应用快速开发脚手架。Spring Cloud 的每个微服务节点本身就是一个 Spring Boot 应用。
3. **"Spring 就是 Spring MVC"** —— 概念混淆。Spring MVC 是 Spring 框架中 Web 层的一个模块，Spring 还包括 IoC 容器、AOP、事务管理、JDBC 抽象等数十个模块。
4. **"Spring Boot 应用不需要配置"** —— 片面理解。Spring Boot 通过"约定大于配置"减少了大部分配置，但仍然需要通过 `application.properties/yml` 提供必要的配置（数据源、Redis 地址等），只是不需要 XML 了。
5. **"用了 Spring Cloud 就是微服务了"** —— 错误认知。微服务不仅仅是引入技术框架，还需要领域驱动设计（DDD）进行服务拆分、CI/CD 流水线、容器化部署（Docker + K8s）、全链路监控、日志集中管理等。Spring Cloud 只是解决了服务间通信和治理的问题。

### 过渡话术

**从 Spring 过渡到 Spring Boot：**
> "以上就是 Spring 框架的核心能力——IoC 容器负责对象管理，AOP 负责横切关注点。但在实际工程中，我们会发现搭建一个 Spring 项目需要大量重复的 XML 配置，效率很低。这就引出了 Spring Boot——它在 Spring 之上封装了自动配置能力，通过约定大于配置的设计哲学，把我们从繁琐的配置工作中解放了出来。核心体现在三个方面：自动配置（Auto-Configuration）、起步依赖（Starter）和内嵌容器。"

**从 Spring Boot 过渡到 Spring Cloud：**
> "Spring Boot 很好解决了单体应用的开发效率问题，一个注解加几个配置参数就能快速启动一个 Web 服务。但当我们面对大规模分布式系统时，单体架构的局限性就暴露出来了——部署耦合、扩展困难、服务间通信复杂。Spring Cloud 正是在 Spring Boot 的基础上，提供了一整套微服务治理的解决方案，包括服务的注册与发现、配置的集中管理、API 网关的路由转发、以及服务间调用时的负载均衡和熔断降级。"

**回答中的表述提升点：**
> "Spring Framework 提供了最底层的 **能力抽象**（Ability Abstraction），Spring Boot 在其基础上完成了 **体验封装**（Experience Encapsulation），而 Spring Cloud 则将这一切升级为 **生态整合**（Ecosystem Integration）——这三个层次恰好对应了 Java 企业级开发从单体到分布式演进的完整路径。"

**收尾总结：**
> "总结一下，从 Spring 到 Spring Boot 再到 Spring Cloud，体现了一条清晰的演进路线：Spring 解决了 EJB 的复杂性，Spring Boot 解决了配置的繁琐性，Spring Cloud 解决了分布式治理的困难性。在实际项目中，我建议根据团队规模和技术积累选择合适的技术栈——对于初创项目，Spring Boot 足以应对；当业务规模增长到需要拆分服务时，再逐步引入 Spring Cloud 组件，避免过度设计。"
