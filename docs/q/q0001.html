请解释MySQL的索引底层数据结构，为什么选用B+树而不是红黑树或Hash？

磁盘IO 数据结构 MySQL B+树 索引

请解释MySQL的索引底层数据结构，为什么选用B+树而不是红黑树或Hash？

联想记忆法

记忆口诀: "胖矮 B+ 树，叶子串珍珠"
记忆原理: 视觉意象——"胖矮"对应树高度低(非叶子只存键，分支多)、"串珍珠"对应叶子节点双向链表
关联知识: 操作系统多级页表（层级索引思想）、文件系统 inode（索引节点管理数据块）

深度解答

核心概念（是什么）

B+ 树是 MySQL InnoDB 的默认索引结构。非叶子节点只存键值不存数据，所有数据存在叶子节点，叶子节点通过双向链表连接。它解决了海量数据的高效范围查找和顺序扫描问题。

底层原理（为什么）

磁盘 I/O 特性是 B+ 树设计的核心驱动力：
- 磁盘寻道是瓶颈: 一次随机 I/O 约 5-10ms
- 顺序读比随机读快 10-100 倍
- 磁盘按页读取，默认 4KB/16KB

为什么不用红黑树: 红黑树是二叉的，百万数据约 40 层 → 40 次磁盘 I/O。B+ 树一个节点可存几百个键，百万数据只需 2-3 层 → 2-3 次 I/O。

为什么不用 Hash: Hash 等值查询 O(1) 但无法范围查询、无法排序、无法最左前缀匹配。MySQL 作为关系数据库，范围查询和排序是核心需求。

实践应用（怎么用）

  • 聚集索引：叶子存完整行数据，每表仅 1 个
  • 辅助索引：叶子存主键值，查询需回表
  • 覆盖索引：查询列全在索引中，无需回表
  • 自增主键：避免 B+ 树页分裂导致的数据移动
  • 最左前缀原则：联合索引 (a,b,c) 只有用 a 或 a,b 前缀才能利用索引

深入思考（注意事项）

  • 不是索引越多越好——每次 INSERT/UPDATE/DELETE 需维护所有索引
  • 联合索引字段顺序极其重要
  • ICP（索引下推）：MySQL 5.6+ 将 WHERE 条件推到存储引擎层过滤，减少回表
  • 页分裂会产生磁盘碎片，影响性能
  • 面试官可能追问：B 树 vs B+ 树的区别（B 树非叶子也存数据）
