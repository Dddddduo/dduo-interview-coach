📖 深度解答

第一层：核心概念

1.1 消息幂等性（Message Idempotency）

消息幂等性 是指消费者对同一条消息执行多次与执行一次所产生的业务效果完全相同。在分布式消息队列（Distributed Message Queue, MQ）体系中，幂等性是保障数据一致性的关键防线。

1.2 问题的本质：为什么要幂等？

问题的根源在于 MQ 的 至少一次语义（At-Least-Once Delivery）。为了保证消息不丢失，消息队列普遍采用以下可靠性机制：

机制 英文 说明 导致重复的原因
生产者重试 Producer Retry 发送超时或异常时重新发送 消息实际已到达 Broker，但 ACK 丢失
Broker 重试 Broker Redelivery 未收到消费者 ACK 时重新投递 消息已消费但 ACK 未到达 Broker
消费者宕机 Consumer Crash 消费后未提交 Offset 就宕机 重启后从上次 Offset 开始重新拉取

这些机制叠加使得重复消息是分布式系统的常态而非异常。MQ 本身无法去重 —— 去重的责任不可避免地落在了消费端（Consumer Side）。

1.3 消息投递语义（Delivery Semantics）

语义 英文 特点 工程代价
至多一次 At-Most-Once 消息可能丢失，但绝不重复 低，无需幂等
至少一次 At-Least-Once 消息不丢失，但可能重复 中，需要幂等
精确一次 Exactly-Once 消息不丢不重 高，分布式事务 + 幂等

业界主流 MQ（Kafka、RocketMQ、RabbitMQ）默认提供 At-Least-Once 语义。达到 Exactly-Once 的唯一路径就是幂等消费端 + 至少一次投递。

第二层：底层原理

2.1 幂等的数学本质

幂等（Idempotence）的数学定义：f(f(x)) = f(x)。在业务层面，这意味着无论执行多少次，系统最终状态是一致的。对于写操作而言，关键在于将"操作"转化为"声明"——不是"把金额加 100"，而是"如果这是一条未处理的通知，则执行加 100 操作"。

2.2 去重的核心逻辑

所有幂等方案的本质都可以抽象为三步：

收到消息 → 查询是否已处理 → 已处理则跳过 / 未处理则执行并标记

不同的方案仅在于"查询"和"标记"这两个操作使用的存储介质和原子性保证不同。

2.3 一致性与性能的权衡

去重状态存储的位置决定了方案的特性：
- 同库同事务（消息表法、唯一键约束）：强一致性，但性能受限于数据库
- 异库异步（Redis 去重）：高性能，但存在最终一致的时间窗口
- 业务内嵌（状态机、乐观锁）：无需额外存储，但适用范围有限

第三层：实践应用 —— 五大方案详解（含 Java 代码）

方案一：数据库唯一键约束（Unique Key Constraint）

原理：利用数据库唯一索引（Unique Index），将消息的唯一标识（如 order_id + msg_id）作为约束列。重复插入时抛出 DuplicateKeyException，直接捕获并视为消费成功。

适用场景：订单、支付、转账等有明确业务单号（Business Key）的场景。

@Service
public class OrderConsumer {

    @Autowired
    private OrderMapper orderMapper;

    @Transactional
    public void consumeOrderMessage(OrderMessage msg) {
        try {
            OrderEntity order = new OrderEntity();
            order.setOrderId(msg.getOrderId());
            order.setMsgId(msg.getMsgId());
            order.setStatus(msg.getStatus());
            order.setAmount(msg.getAmount());
            orderMapper.insert(order);
        } catch (DuplicateKeyException e) {
            log.warn("Duplicate message ignored, orderId={}, msgId={}",
                     msg.getOrderId(), msg.getMsgId());
        }
    }
}

DDL 关键:

CREATE TABLE `t_order` (
  `id` BIGINT AUTO_INCREMENT,
  `order_id` VARCHAR(32) NOT NULL,
  `msg_id` VARCHAR(64) NOT NULL,
  `status` VARCHAR(16),
  `amount` DECIMAL(10,2),
  UNIQUE KEY `uk_order_msg` (`order_id`, `msg_id`)
);
优势 劣势
绝对可靠，不丢失去重状态 强依赖数据库，性能瓶颈
天然支持事务一致性 受限于业务表必须可加唯一索引
无额外中间件成本 高并发下数据库压力大

方案二：Redis 去重（Redis Deduplication）

原理：消费前，通过 SET key NX 命令尝试将消息 ID 写入 Redis。写入成功则说明未处理；写入失败（已存在）则说明已处理过。设置 TTL 以避免内存无限增长。

@Component
public class RedisDeduplication {

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String DEDUP_PREFIX = "dedup:msg:";
    private static final long TTL_SECONDS = 7 * 24 * 3600;

    public boolean tryProcess(String msgId) {
        Boolean success = redisTemplate.opsForValue()
                .setIfAbsent(DEDUP_PREFIX + msgId, "1", TTL_SECONDS, TimeUnit.SECONDS);
        return Boolean.TRUE.equals(success);
    }
}

@Service
public class PaymentConsumer {

    @Autowired
    private RedisDeduplication dedup;
    @Autowired
    private PaymentService paymentService;

    public void consume(PaymentMessage msg) {
        if (!dedup.tryProcess(msg.getMsgId())) {
            return; // 已处理
        }
        paymentService.processPayment(msg);
    }
}
优势 劣势
性能极高（纯内存操作） TTL 到期后可能重复消费
无数据库压力 需额外维护 Redis 集群
实现简单 与业务事务不在同一介质

方案三：业务状态机（State Machine）

原理：将业务状态流转定义为有向图，状态转换是单向不可逆的。已到达终态的消息直接返回成功。

@Component
public class OrderStateMachine {

    @Transactional
    public boolean handleState(String orderId, OrderEvent event) {
        Order order = orderMapper.selectByOrderId(orderId);
        OrderState currentState = order.getStatus();
        OrderState targetState = event.getTargetState();

        if (currentState.ordinal() >= targetState.ordinal()) {
            return true; // 已处理，幂等返回
        }

        if (!StateGraph.isValidTransition(currentState, targetState)) {
            throw new IllegalStateException("Invalid transition");
        }

        order.setStatus(targetState);
        orderMapper.updateById(order);
        return true;
    }
}

enum OrderState {
    INIT(0), PAID(1), SHIPPED(2), CONFIRMED(3);
    private final int level;
    OrderState(int level) { this.level = level; }
    public int getLevel() { return level; }
}
优势 劣势
业务语义最自然 仅适用于有状态流转的业务
无需额外去重状态存储 状态图需前期充分设计
与业务事务完全一致 不适用于无状态操作

方案四：乐观锁（Optimistic Lock with Version）

原理：为数据表增加版本号字段，每次更新时 WHERE version = oldVersion 且 SET version = version + 1。

@Service
public class AccountService {

    @Transactional
    public boolean transferWithVersion(TransferMessage msg) {
        Account account = accountMapper.selectByAccountId(msg.getAccountId());

        int affected = accountMapper.updateBalanceByVersion(
            account.getId(),
            account.getBalance().add(msg.getAmount()),
            account.getVersion(),
            account.getVersion() + 1
        );

        return affected > 0;
    }
}

Mapper:

<update id="updateBalanceByVersion">
    UPDATE t_account SET balance = #{newBalance}, version = #{newVersion}
    WHERE id = #{id} AND version = #{oldVersion}
</update>
优势 劣势
天然支持并发无锁 仅适用于更新操作
无需额外去重中间件 高冲突场景大量回滚

方案五：消息表法（Message Table）

原理：单独建消息消费记录表，在同一事务中先插入消费记录再执行业务。

@Service
public class MessageTableConsumer {

    @Transactional
    public void consume(Message msg) {
        try {
            MsgConsumeRecord record = new MsgConsumeRecord();
            record.setMsgId(msg.getMsgId());
            record.setBizKey(msg.getBizKey());
            msgConsumeMapper.insert(record);
        } catch (DuplicateKeyException e) {
            return; // 已消费
        }
        businessService.process(msg);
    }
}

消息表:

CREATE TABLE `t_msg_consume` (
  `msg_id` VARCHAR(64) NOT NULL PRIMARY KEY,
  `biz_key` VARCHAR(64) NOT NULL,
  `consume_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `status` TINYINT DEFAULT 0
) ENGINE=InnoDB;
优势 劣势
同库同事务强一致 每次多一次写操作
永不丢失去重状态 需定期清理历史数据
通用性强 消息表可能成为热点

4.2 生产建议：混合方案

Redis 前置去重（快速过滤 99% 重复）
  + 消息表/唯一键后置兜底（强一致保底）

public void hybridConsume(Message msg) {
    if (!redisDedup.tryProcess(msg.getMsgId())) return;        // 第一层：Redis
    try {
        msgConsumeMapper.insert(new MsgConsumeRecord(msg.getMsgId()));
    } catch (DuplicateKeyException e) { return; }              // 第二层：DB 兜底
    businessService.process(msg);
}

4.3 面试亮点

  1. 指出幂等不是 MQ 的职责，是消费端的职责，体现分布式系统的责任边界划分
  2. 区分业务幂等（Business Idempotency） 与技术幂等（Technical Idempotency）
  3. Kafka 0.11+ 的幂等生产者通过 Producer ID + Sequence Number 在 Broker 端去重，但消费端仍需幂等
  4. Any 持久化方案都要考虑数据清理策略
