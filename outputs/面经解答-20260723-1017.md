## 第1题：CountDownLatch和Semaphore区别，配合使用

### 🧠 联想记忆法

**记忆口诀**: "CountDown是倒计时门闩，Semaphore是信号灯令牌"

**记忆原理**: 想象一个宴会大厅——CountDownLatch像大门上的门闩，等人到齐了（count归零）门才打开，所有人都能进；Semaphore像停车场的计位牌，牌子发完了就得等有人出来才能进。一个是"等够数就放行"，一个是"限制同时进入的数量"。

**关联知识**: 联系生活中的场景——演唱会等所有人都坐好才开始（CountDownLatch），图书馆座位有限，出来一个才能进一个（Semaphore）。

### 📖 深度解答

**1. 核心概念（是什么）**

CountDownLatch（倒计时门闩）和 Semaphore（信号量）是 Java 并发包（java.util.concurrent）中的两种同步工具类（Synchronizer），都用于线程间的协调控制，但解决的问题完全不同：

- **CountDownLatch**：允许一个或多个线程等待，直到在其他线程中执行的一组操作完成。它维护一个计数器，线程调用 `await()` 等待计数器归零，其他线程调用 `countDown()` 递减计数器。
- **Semaphore**：控制同时访问特定资源的线程数量。它维护一组许可证（permits），线程通过 `acquire()` 获取许可证，使用完毕后通过 `release()` 归还。

**2. 底层原理（为什么）**

CountDownLatch 基于 AQS（AbstractQueuedSynchronizer，抽象队列同步器）实现。AQS 是 Java 并发包的基石，它维护了一个 volatile 的 int 类型 state 变量和一个 FIFO 的等待队列（CLH 锁变体）。CountDownLatch 将 state 初始化为构造时传入的 count 值，`countDown()` 通过 CAS（Compare-And-Swap，比较并交换）操作尝试将 state 减 1，当 state 变为 0 时，AQS 会唤醒所有在等待队列中的线程。关键点是：CountDownLatch 的计数器只能被使用一次，一旦归零就无法重置。

Semaphore 同样基于 AQS 实现，但它的工作机制更灵活。它维护一个 state 表示可用许可证数量。`acquire()` 时通过 CAS 将 state 减 1（如果 state > 0），如果 state == 0，当前线程会进入 AQS 的等待队列挂起。`release()` 时通过 CAS 将 state 加 1，并唤醒等待队列中的线程。Semaphore 支持公平模式（FairSync）和非公平模式（NonfairSync）——公平模式下，如果等待队列中有线程，新来的 acquire 也会排队；非公平模式下，新来的线程可以直接尝试 CAS 竞争许可证。

**3. 实践应用（怎么用）**

CountDownLatch 典型场景：
```java
// 场景：主线程等待所有子任务完成
CountDownLatch latch = new CountDownLatch(5);
ExecutorService executor = Executors.newFixedThreadPool(5);

for (int i = 0; i < 5; i++) {
    executor.submit(() -> {
        try {
            doWork();
        } finally {
            latch.countDown();
        }
    });
}
latch.await();
System.out.println("All tasks complete");
```

Semaphore 典型场景：
```java
// 场景：限制数据库连接池并发数
Semaphore semaphore = new Semaphore(10);
for (int i = 0; i < 100; i++) {
    executor.submit(() -> {
        try {
            semaphore.acquire();
            accessDatabase();
        } finally {
            semaphore.release();
        }
    });
}
```

配合使用——并发任务限流聚合：
```java
Semaphore semaphore = new Semaphore(10);
CountDownLatch latch = new CountDownLatch(100);
for (int i = 0; i < 100; i++) {
    executor.submit(() -> {
        try {
            semaphore.acquire();
            doWork();
        } finally {
            semaphore.release();
            latch.countDown();
        }
    });
}
latch.await();
```

**4. 深入思考（注意事项）**

- **重用性**：CountDownLatch 不可重用，如果需要重置计数器应该使用 CyclicBarrier。
- **超时控制**：`await(long timeout, TimeUnit unit)` 设置等待超时，避免永久阻塞。
- **许可证泄露**：Semaphore 使用时必须确保 `release()` 在 finally 块中调用。
- **二元信号量**：Semaphore(1) 可以用作互斥锁（Mutex），但 Java 中更推荐使用 ReentrantLock。
- **公平性选择**：非公平 Semaphore 吞吐量更高，但可能导致线程饥饿。

### 🗺️ 回答思路

**答题逻辑框架**：定义区别 → 原理对比 → 使用场景 → 配合案例

**重点得分点**：AQS 底层原理、CAS 操作、公平/非公平区别

**常见误区**：混淆 CountDownLatch 和 CyclicBarrier（后者可以重置计数器）

**时间分配建议**：30s 定义区别 → 1min 原理 → 1min 代码示例 → 30s 深入思考

**过渡话术**："虽然它们都属于 Java 并发包中的同步工具类，但解决的问题截然不同..."

## 第2题：乐观锁，sql 语句手写乐观锁

### 🧠 联想记忆法

**记忆口诀**: "悲观锁像写作业前锁门怕人打扰，乐观锁像不锁门写完检查有没有被改过"

**记忆原理**: 悲观锁（Pessimistic Locking）认为并发冲突一定会发生，所以操作前就加锁；乐观锁（Optimistic Locking）认为冲突很少发生，操作时不做限制，只在更新时检查冲突。

**关联知识**: Git 的 merge 机制——各自改，提交时发现冲突了再处理，这就是乐观锁思想。

### 📖 深度解答

**1. 核心概念**

乐观锁（Optimistic Locking）是一种并发控制机制，它假设多线程/多事务之间的冲突不会频繁发生，因此允许并发操作同时进行，只在数据提交更新时通过版本号或时间戳检查数据是否被其他事务修改过。

**2. 底层原理**

乐观锁的核心实现机制是 CAS 思想，在数据库层面通常通过版本号（Version）或时间戳来实现：

- **版本号机制**：在表中增加一个 version 字段。每次更新时执行 `SET version = version + 1 WHERE version = oldVersion`。
- **CAS 思想**：比较当前值与期望值，如果相等则更新，否则放弃或重试。

**3. 实践应用**

```sql
-- 创建带有 version 字段的表
CREATE TABLE account (
    id BIGINT PRIMARY KEY,
    balance DECIMAL(10,2),
    version INT DEFAULT 0
);

-- 读取数据
SELECT id, balance, version FROM account WHERE id = 1;
-- 结果: id=1, balance=100.00, version=3

-- 乐观锁更新
UPDATE account 
SET balance = balance - 50, version = version + 1 
WHERE id = 1 AND version = 3;
-- 影响行数 = 1 → 成功；0 → 冲突，需要重试
```

```java
// Java 重试逻辑
public boolean updateWithRetry(Account account) {
    int retryCount = 0;
    int maxRetries = 3;
    while (retryCount < maxRetries) {
        Account current = accountDao.selectById(account.getId());
        int updated = accountDao.updateBalanceWithVersion(
            account.getId(), current.getBalance() - 50, current.getVersion()
        );
        if (updated > 0) return true;
        retryCount++;
    }
    throw new OptimisticLockException("更新失败");
}
```

**4. 深入思考**

- **ABA 问题**：数据库版本号机制天然解决（version 只增不减）
- **重试性能**：高并发下大量重试影响性能，可用退避策略（Backoff Strategy）
- **适用场景**：读多写少、冲突概率低的场景

### 🗺️ 回答思路

**答题逻辑框架**：定义 → 原理（版本号/时间戳/CAS）→ SQL 示例 → 重试逻辑 → 优缺点

**重点得分点**：版本号机制的完整 SQL、重试逻辑、ABA 问题

**常见误区**：不要把乐观锁的 select 和 update 放在一个长事务中

## 第3题：责任链模式是怎样的？特征是什么？

### 🧠 联想记忆法

**记忆口诀**: "责任链像流水线，每个工位干自己的活，处理不了就传给下一个"

**记忆原理**: 想象快递分拣流水线——检查地址、称重、贴单，每个工位只做自己的事，做完传给下一个。

**关联知识**: Java Web 中的 Filter 链、Spring MVC 的 Interceptor 链、Netty 的 ChannelPipeline 都是责任链模式的应用。

### 📖 深度解答

**1. 核心概念**

责任链模式（Chain of Responsibility Pattern）是一种行为型设计模式（Behavioral Design Pattern），它将请求的发送者和接收者解耦，使多个对象都有机会处理请求。将这些对象连成一条链，请求沿着链传递，直到有一个对象处理它为止。

**2. 底层原理**

三个角色：
- **Handler（抽象处理器）**：定义处理接口，持有下一个处理器的引用
- **ConcreteHandler（具体处理器）**：实现处理接口，能处理则处理，不能则转发
- **Client（客户端）**：向链的第一个处理器提交请求

**3. 实践应用**

```java
public abstract class Handler {
    protected Handler next;
    public void setNext(Handler next) { this.next = next; }
    public abstract void handle(Request request);
}

public class ValidationHandler extends Handler {
    @Override
    public void handle(Request request) {
        if (request.getParam() == null) {
            throw new IllegalArgumentException("参数不能为空");
        }
        System.out.println("参数校验通过");
        if (next != null) next.handle(request);
    }
}

public class AuthHandler extends Handler {
    @Override
    public void handle(Request request) {
        if (!"admin".equals(request.getRole())) {
            throw new SecurityException("权限不足");
        }
        System.out.println("权限校验通过");
        if (next != null) next.handle(request);
    }
}

// 客户端组装链
Handler validation = new ValidationHandler();
Handler auth = new AuthHandler();
validation.setNext(auth);
validation.handle(request);
```

- **与装饰器模式区别**：责任链用于请求处理和传递，装饰器用于动态增加功能

## 第5题：讲一下单例模式，它的好处是什么？

### 🧠 联想记忆法

**记忆口诀**: "单例单例，全局唯一；节省资源，统一管理"

**关联知识**: Spring Bean 默认作用域就是 singleton，数据库连接池通常也是单例。

### 📖 深度解答

**1. 核心概念**

单例模式（Singleton Pattern）是一种创建型设计模式，确保一个类在 JVM 中只有一个实例，并提供一个全局访问点。

**2. 底层原理**

- 私有化构造方法
- 静态成员变量持有实例引用
- 静态方法提供全局访问
- 线程安全保证
- 防止序列化破坏（readResolve）
- 防止反射攻击（枚举方式）

**3. 实现方式**

饿汉式：
```java
public class EagerSingleton {
    private static final EagerSingleton INSTANCE = new EagerSingleton();
    private EagerSingleton() {}
    public static EagerSingleton getInstance() { return INSTANCE; }
}
```

懒汉式（双检锁 DCL）：
```java
public class LazySingleton {
    private static volatile LazySingleton instance;
    private LazySingleton() {}
    public static LazySingleton getInstance() {
        if (instance == null) {
            synchronized (LazySingleton.class) {
                if (instance == null) instance = new LazySingleton();
            }
        }
        return instance;
    }
}
```

静态内部类：
```java
public class HolderSingleton {
    private HolderSingleton() {}
    private static class Holder {
        private static final HolderSingleton INSTANCE = new HolderSingleton();
    }
    public static HolderSingleton getInstance() { return Holder.INSTANCE; }
}
```

枚举（最安全）：
```java
public enum EnumSingleton { INSTANCE; }
```

**单例模式的好处**：
1. **节省内存**：只有一个实例，避免频繁创建销毁
2. **统一控制**：集中管理共享资源（配置文件读取）
3. **延迟加载**：懒汉式可延迟创建，优化启动速度
4. **全局访问**：降低使用成本
5. **线程安全**：合理实现下天然线程安全
6. **性能优化**：避免重复创建对象的性能开销

**4. 深入思考**

- 反射可破坏除枚举外的所有单例
- 反序列化需要添加 readResolve()
- 不同 ClassLoader 会产生不同实例
- 单元测试困难（耦合），推荐依赖注入

## 第6题：单例模式有两种加载方式，讲一下

### 🧠 联想记忆法

**记忆口诀**: "饿汉着急先吃饭，懒汉饿了再做饭"

**记忆原理**: 饿汉式（Eager Loading）类加载时就创建实例；懒汉式（Lazy Loading）首次调用 getInstance() 时才创建。

### 📖 深度解答

**饿汉式（Eager Initialization）：**
- 类加载到 JVM 时就创建实例，由 ClassLoader 完成
- 天然线程安全、实现简单、访问速度快
- 缺点：类加载就初始化，可能浪费资源（如果从未使用过）

**懒汉式（Lazy Initialization）：**
- 首次调用 getInstance() 时才创建
- 实现方式：双检锁 DCL、静态内部类、枚举
- 延迟加载，资源利用率高
- 缺点：实现复杂（需处理线程安全）、首次访问有性能开销

**选择原则**：如果实例一定会被使用且创建开销小，选饿汉式；否则选懒汉式。

## 第9题：讲一下你了解的排序算法

### 🧠 联想记忆法

**记忆口诀**: "冒泡选择插入归并快排堆排希尔桶排基数"

**记忆原理**: 三类时间复杂度——O(n²) 基础三兄弟（冒泡、选择、插入），O(nlogn) 四大天王（归并、快排、堆排、希尔），O(n) 特殊部队（桶排、计数、基数）。

### 📖 深度解答

**O(n²) 排序算法（适合小数据量）：**

冒泡排序：
```java
public void bubbleSort(int[] arr) {
    for (int i = 0; i < arr.length - 1; i++) {
        boolean swapped = false;
        for (int j = 0; j < arr.length - 1 - i; j++) {
            if (arr[j] > arr[j + 1]) { swap(arr, j, j + 1); swapped = true; }
        }
        if (!swapped) break; // 优化
    }
}
```

选择排序：
```java
public void selectionSort(int[] arr) {
    for (int i = 0; i < arr.length - 1; i++) {
        int minIdx = i;
        for (int j = i + 1; j < arr.length; j++) {
            if (arr[j] < arr[minIdx]) minIdx = j;
        }
        swap(arr, i, minIdx);
    }
}
```

插入排序：
```java
public void insertionSort(int[] arr) {
    for (int i = 1; i < arr.length; i++) {
        int key = arr[i];
        int j = i - 1;
        while (j >= 0 && arr[j] > key) { arr[j + 1] = arr[j]; j--; }
        arr[j + 1] = key;
    }
}
```

**O(nlogn) 排序（适合大数据量）：**

快速排序：
```java
public void quickSort(int[] arr, int left, int right) {
    if (left >= right) return;
    int pivot = partition(arr, left, right);
    quickSort(arr, left, pivot - 1);
    quickSort(arr, pivot + 1, right);
}
```

归并排序：
```java
public void mergeSort(int[] arr, int left, int right) {
    if (left >= right) return;
    int mid = left + (right - left) / 2;
    mergeSort(arr, left, mid);
    mergeSort(arr, mid + 1, right);
    merge(arr, left, mid, right);
}
```

**对比总结**：

| 算法 | 平均时间 | 最坏时间 | 空间 | 稳定 |
|------|---------|---------|------|------|
| 冒泡 | O(n²) | O(n²) | O(1) | 稳定 |
| 选择 | O(n²) | O(n²) | O(1) | 不稳定 |
| 插入 | O(n²) | O(n²) | O(1) | 稳定 |
| 快排 | O(nlogn) | O(n²) | O(logn) | 不稳定 |
| 归并 | O(nlogn) | O(nlogn) | O(n) | 稳定 |
| 堆排 | O(nlogn) | O(nlogn) | O(1) | 不稳定 |

**选型建议**：小数据量（<100）选插入排序，大数据量选快排（默认），需要稳定选归并，Top K 选堆排/快速选择。

## 第10题：什么叫做稳定的排序算法

### 🧠 联想记忆法

**记忆口诀**: "稳定排序：相同元素，排序前后相对位置不变"

**关联知识**: 数据库 ORDER BY 多个字段时，第二次排序不稳定会打乱第一次排序结果。

### 📖 深度解答

**定义**：稳定排序算法（Stable Sort Algorithm）指当待排序序列中存在两个或以上相等的关键字时，排序前后这些相等元素的相对次序保持不变。

**为什么重要**：
1. **多关键字排序**：先按年龄排，再按姓名排，需要第二次排序稳定
2. **基数排序基础**：基数排序依赖于稳定排序作为子过程
3. **业务意义**：交易记录先按时间再按金额排序，稳定性保留时间顺序

**判断方法**：
- **稳定**：冒泡（相等不交换）、插入（相等不越前）、归并（相等先取左）
- **不稳定**：选择（交换破坏位置）、快排（partition 交换）、堆排（远距离交换）

## 第12题：100万数据中找前十位用什么排序

### 🧠 联想记忆法

**记忆口诀**: "Top K 海量数据，堆排是最优解"

### 📖 深度解答

**推荐方案：最小堆**

维护一个大小为 10 的最小堆：
```java
public int[] topK(int[] arr, int k) {
    PriorityQueue<Integer> minHeap = new PriorityQueue<>(k);
    for (int num : arr) {
        if (minHeap.size() < k) minHeap.offer(num);
        else if (num > minHeap.peek()) {
            minHeap.poll();
            minHeap.offer(num);
        }
    }
    return minHeap.stream().mapToInt(Integer::intValue).toArray();
}
```

时间复杂度 O(nlogK) ≈ O(n)，空间 O(K) = O(1)（K=10 极小）。

**方案对比**：
- 全排 O(nlogn)：不必要
- 快速选择 O(n)：无法处理流式数据
- 堆排序：支持流式数据、内存友好、性能稳定

## 第13题：快速选择的时间复杂度？

### 🧠 联想记忆法

**记忆口诀**: "快速选择 O(n) 平均，最坏 O(n²) 看枢轴"

### 📖 深度解答

**快速选择（Quickselect）**：
- **平均时间复杂度**：O(n)——每次 partition 只递归一边
- **最坏时间复杂度**：O(n²)——每次 pivot 都选到极端值
- **优化**：中位数的中位数（Median of Medians）策略，保证 O(n) 最坏情况
- **空间复杂度**：O(1) 原地操作

```java
public int quickSelect(int[] nums, int k) {
    return quickSelect(nums, 0, nums.length - 1, k - 1);
}
private int quickSelect(int[] nums, int left, int right, int k) {
    if (left == right) return nums[left];
    int pivotIndex = partition(nums, left, right);
    if (k == pivotIndex) return nums[k];
    else if (k < pivotIndex) return quickSelect(nums, left, pivotIndex - 1, k);
    else return quickSelect(nums, pivotIndex + 1, right, k);
}
```

## 第15题：Dijkstra算法解决什么问题？

### 🧠 联想记忆法

**记忆口诀**: "Dijkstra 找最短路径，加权图无负边，BFS 升级版"

### 📖 深度解答

**1. 核心概念**

Dijkstra（迪杰斯特拉）算法解决加权有向图/无向图中，从一个源点到所有其他节点的最短路径问题。要求边权值非负。

**2. 算法原理**
1. 维护距离数组 dist[] 和已确定最短路径的节点集合 S
2. 每次取距离最小且不在 S 中的节点 u
3. 松弛（Relaxation）：如果 dist[u] + w(u,v) < dist[v]，更新 dist[v]
4. 将 u 加入 S，重复直到所有节点处理完毕

**3. 代码实现**
```java
public int[] dijkstra(int n, List<int[]>[] graph, int start) {
    int[] dist = new int[n];
    Arrays.fill(dist, Integer.MAX_VALUE);
    dist[start] = 0;
    PriorityQueue<int[]> pq = new PriorityQueue<>((a,b) -> a[1] - b[1]);
    pq.offer(new int[]{start, 0});
    while (!pq.isEmpty()) {
        int[] cur = pq.poll();
        int node = cur[0], d = cur[1];
        if (d > dist[node]) continue;
        for (int[] edge : graph[node]) {
            int next = edge[0], weight = edge[1];
            if (dist[node] + weight < dist[next]) {
                dist[next] = dist[node] + weight;
                pq.offer(new int[]{next, dist[next]});
            }
        }
    }
    return dist;
}
```

**4. 应用场景**
- 地图导航、OSPF 网络路由、社交网络最短关系链、物流配送最优路线

**5. 局限性**
- 不能处理负权边（存在负权用 Bellman-Ford）
- 稠密图中时间复杂度较高

## 第16题：进程和线程有什么区别？

### 🧠 联想记忆法

**区别对比**：

| 维度 | 进程 | 线程 |
|------|------|------|

| 创建速度 | 极快（纳秒级） | 较慢（需系统调用） |
| 上下文切换 | 用户态完成，无系统调用 | 需陷入内核，开销大 |
| 多核利用 | 一般无法利用多核 | 天然支持 |
| 一个线程阻塞 | 同一进程所有用户态线程都阻塞 | 不影响其他 |

**协程（Coroutine）**：轻量化用户态线程，如 Go 的 Goroutine、Kotlin Coroutine。优势在于极高的创建数量和极低的切换成本。

**Java 线程**：采用 1:1 模型（通过 pthreads），每个 Java 线程对应一个内核线程。

## 第23题：进程调度算法

### 🧠 联想记忆法

**记忆口诀**: "FCFS SJF RR 优先级 多级反馈队列"

### 📖 深度解答

**五种主要算法**：

1. **先来先服务（FCFS）**：按到达顺序，非抢占。缺点：护航效应（大任务阻塞小任务）
2. **短作业优先（SJF）**：执行时间最短优先。优点：平均等待最小。缺点：长任务可能饿死
3. **时间片轮转（RR）**：固定时间片轮流执行。优点：响应快、公平。缺点：时间片大小影响性能
4. **优先级调度**：按优先级执行。缺点：低优先级可能饿死。解决方案：优先级老化
5. **多级反馈队列（MLFQ）**：综合上述优点，多级队列 + 动态优先级。Linux 核心调度算法

## 第24题：FCFS 的缺点

### 🧠 联想记忆法

**记忆口诀**: "FCFS 护航效应——一个大任务拦住后面所有小任务"

### 📖 深度解答

**主要缺点**：
1. **护航效应（Convoy Effect）**：CPU 密集型长任务阻塞后续所有短任务
2. **平均等待时间长**：短任务被长任务阻塞
3. **不适合分时系统**：交互式任务响应无法保证
4. **设备利用率低**：长任务占 CPU 时 I/O 设备空闲

举例：P1 100ms + P2 1ms，FCFS 下 P2 等待 100ms。换 SJF 则 P2 等待 0ms。

## 第25题：优先级调度算法

### 🧠 联想记忆法

**记忆口诀**: "优先级调度：VIP 先办，普通客户等"

### 📖 深度解答

**两种方式**：
1. **静态优先级**：进程创建时确定，运行中不变
2. **动态优先级**：随时间调整（如 nice 值老化）

## 第26题：TCP 四次挥手

### 🧠 联想记忆法

**记忆口诀**: "四次挥手：FIN→ACK→FIN→ACK，谁先关谁后关要记清"

### 📖 深度解答

**步骤**：
1. **A 发送 FIN**：FIN=1, seq=u。A 进入 FIN_WAIT_1
2. **B 回复 ACK**：ACK=1, seq=v, ack_seq=u+1。B 进入 CLOSE_WAIT。A 收到后 FIN_WAIT_2
3. **B 发送 FIN**：FIN=1, seq=w。B 进入 LAST_ACK
4. **A 回复 ACK**：ACK=1, seq=u+1, ack_seq=w+1。A 进入 TIME_WAIT（2MSL）

**关键点**：
- 为什么四次？TCP 全双工，每个方向单独关闭
- 第 2、3 步不能合并，因为 B 可能需要处理剩余数据
- TIME_WAIT 持续 2MSL（约 60s），保证最后一个 ACK 能到达

## 第27题：TIME_WAIT 可以调吗？

### 🧠 联想记忆法

**记忆口诀**: "TIME_WAIT 可以调但通常不要改"

### 📖 深度解答

**调整方式**：
1. `sysctl -w net.ipv4.tcp_fin_timeout=30`（默认 60s）
2. `tcp_tw_reuse`：允许 TIME_WAIT 套接字用于新连接
3. `tcp_tw_recycle`：快速回收（Linux 4.12 后移除，NAT 下有问题）
4. 调整 MSL：TIME_WAIT = 2 × MSL

**风险**：缩短 TIME_WAIT 可能导致最后一个 ACK 无法重传，造成数据丢失。

## 第39题：MVCC 是什么？

### 🧠 联想记忆法

**记忆口诀**: "MVCC 多版本并发控制，读写不冲突，靠快照实现"

### 📖 深度解答

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 InnoDB 的核心机制，每个事务看到数据的快照。

**解决问题**：
1. **读写冲突**：读不用加锁，写不用等读完成
2. **一致性读**：事务内多次读取一致（可重复读）
3. **并发性能**：大幅提升读多写少场景性能

**实现原理**（Undo Log + Read View）：
- 每行数据隐藏字段：DB_TRX_ID（最后修改事务 ID）、DB_ROLL_PTR（回滚指针）
- Undo Log 保存历史版本链条
- 事务开始时生成 Read View（活跃事务列表），根据 Read View 和 Undo Log 判断读取哪版本

## 第40题：MySQL 的锁

### 🧠 联想记忆法

**记忆口诀**: "按粒度分：表锁行锁间隙锁；按模式分：共享锁排他锁意向锁"

### 📖 深度解答

**按粒度**：
1. **表级锁**：整表加锁（MyISAM 默认、DDL 操作）
2. **行级锁**：锁特定行（InnoDB 默认）
3. **间隙锁（Gap Lock）**：锁索引记录间间隙，防幻读
4. **临键锁（Next-Key Lock）**：行锁 + 间隙锁

**按模式**：
1. **共享锁（S Lock / 读锁）**：`SELECT ... LOCK IN SHARE MODE`
2. **排他锁（X Lock / 写锁）**：`SELECT ... FOR UPDATE`
3. **意向锁（Intention Lock）**：表级锁，表示事务准备获取行锁

## 第41题：临键锁是什么？

### 🧠 联想记忆法

**记忆口诀**: "Next-Key Lock = 行锁 + 间隙锁，解决幻读"

### 📖 深度解答

临键锁（Next-Key Lock）是行锁（Record Lock）和间隙锁（Gap Lock）的组合。InnoDB 在 RR 级别下默认使用。

**示例**：索引列有值 1, 3, 5, 8
```sql
SELECT * FROM t WHERE id > 3 AND id < 8 FOR UPDATE
```
- 行锁：锁住 id=5
- 间隙锁：锁住 (3,5) 和 (5,8) 的间隙
- 其他事务不能插入 id=4 或 id=7，解决幻读

## 第42题：MySQL 索引结构

## 第43题：主键索引是聚簇还是非聚簇？

### 🧠 联想记忆法

**记忆口诀**: "InnoDB 主键 = 聚簇索引，数据即索引，索引即数据"

### 📖 深度解答

在 InnoDB 中，主键索引（Primary Key Index）就是聚簇索引（Clustered Index）。数据行直接存储在 B+ 树叶子节点上。

**特点**：
- 数据物理顺序与索引顺序一致
- 每张表只有一个聚簇索引
- 无主键时选择唯一非空索引；都没有则隐式生成 6 字节 ROWID

## 第44题：什么是非聚簇索引？

### 🧠 联想记忆法

**记忆口诀**: "非聚簇索引（二级索引），叶子节点存主键值，需要回表"

### 📖 深度解答

非聚簇索引（Non-Clustered Index / Secondary Index）的叶子节点存储主键值。通过二级索引查询时需要"回表"（Table Access by Index RowID）。

```sql
CREATE INDEX idx_age ON user(age);
SELECT * FROM user WHERE age = 25;
-- 1. 通过 idx_age 找到主键 id=10
-- 2. 通过 id=10 回表聚簇索引找到完整行
```

覆盖索引（Covering Index）可避免回表——索引包含查询所需的所有字段。

## 第47题：LeetCode 编辑距离

### 🧠 联想记忆法

**记忆口诀**: "DP 二维表，左上角替换，左边插入，上边删除"

### 📖 深度解答

```java
public int minDistance(String word1, String word2) {
    int m = word1.length(), n = word2.length();
    int[][] dp = new int[m + 1][n + 1];
    for (int i = 0; i <= m; i++) dp[i][0] = i;
    for (int j = 0; j <= n; j++) dp[0][j] = j;
    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            if (word1.charAt(i - 1) == word2.charAt(j - 1)) {
                dp[i][j] = dp[i - 1][j - 1];
            } else {
                dp[i][j] = 1 + Math.min(dp[i - 1][j], // 删除
                    Math.min(dp[i][j - 1], // 插入
                             dp[i - 1][j - 1])); // 替换
            }
        }
    }
    return dp[m][n];
}
```
时间复杂度 O(m*n)，空间可优化到 O(min(m,n))。

## 第48题：LeetCode 接雨水

### 🧠 联想记忆法

**记忆口诀**: "双指针左右夹逼，维护左右最大高度"

### 📖 深度解答

```java
public int trap(int[] height) {
    int left = 0, right = height.length - 1;
    int leftMax = 0, rightMax = 0;
    int water = 0;
    while (left < right) {
        if (height[left] < height[right]) {
            if (height[left] >= leftMax) leftMax = height[left];
            else water += leftMax - height[left];
            left++;
        } else {
            if (height[right] >= rightMax) rightMax = height[right];
            else water += rightMax - height[right];
            right--;
        }
    }
    return water;
}
```
时间复杂度 O(n)，空间 O(1)。

    db.update(data);              // 1. 更新数据库
    cache.delete(data.getId());   // 2. 删除缓存

## 第71题：LeetCode 岛屿数量

### 🧠 联想记忆法

**记忆口诀**: "DFS/BFS 扫描，遇到 '1' 就淹掉，计数加一"

### 📖 深度解答

```java
public int numIslands(char[][] grid) {
    if (grid == null || grid.length == 0) return 0;
    int m = grid.length, n = grid[0].length;
    int count = 0;
    for (int i = 0; i < m; i++) {
        for (int j = 0; j < n; j++) {
            if (grid[i][j] == '1') {
                count++;
                dfs(grid, i, j);
            }
        }
    }
    return count;
}
private void dfs(char[][] grid, int i, int j) {
    if (i < 0 || i >= grid.length || j < 0 || j >= grid[0].length
        || grid[i][j] == '0') return;
    grid[i][j] = '0';
    dfs(grid, i + 1, j);
    dfs(grid, i - 1, j);
    dfs(grid, i, j + 1);
    dfs(grid, i, j - 1);
}
```
时间复杂度 O(m*n)。变体：BFS（队列）、Union-Find（并查集）。

## 第74题：RAG 的分类和全链路能力

### 🧠 联想记忆法

**记忆口诀**: "RAG 三板斧：索引（Ingestion）→ 检索（Retrieval）→ 生成（Generation）"

### 📖 深度解答

**全链路**：

**1. 数据预处理（Indexing）**
- 文档解析：PDF/HTML/Word → 文本
- 文本分割（Chunking）：chunk_size=512, overlap=50
- 向量化（Embedding）：text-embedding-3-small
- 向量存储：Redis/Pinecone/Milvus

**2. 检索阶段（Retrieval）**
- 向量检索（ANN）：最相似 K 个
- 混合检索：BM25 + 向量 + 重排序
- 重排序（Re-ranking）：cross-encoder

**3. 生成阶段（Generation）**
- Prompt 构建：System + 检索结果 + 问题
- LLM 生成 + 引用溯源

**RAG 分类**：
- **Naive RAG**：简单向量检索 + LLM
- **Advanced RAG**：Query Rewriting、HyDE、多路召回
- **Modular RAG**：可插拔模块（路由、过滤、迭代）
- **Agentic RAG**：Agent 自主决定何时检索、检索什么

## 第75题：RAG 检索不准确的原因

### 🧠 联想记忆法

**记忆口诀**: "检索不准五大因：分块不合理、embedding 不匹配、chunk 不够细、重排序缺失、query 不准确"

### 📖 深度解答

**主要原因**：

1. **分块策略不当（Chunking）**
   - 分块过大：语义稀释
   - 分块过小：上下文不完整
   - 无重叠：关键信息在切分点丢失

2. **Embedding 质量不足**
   - 模型与领域不匹配
   - 多语言混合效果差
   - 专业术语不在预训练词汇中

3. **检索召回不足**
   - Top K 太小
   - 只有向量检索（无关键词补充）
   - 没有重排序阶段

4. **Query 问题**
   - Query 太短/模糊
   - Query 和文档表达方式差异大
   - Query 含噪音

5. **元数据过滤不足**
   - 未利用时间/分类等元数据
   - 多源数据低质量污染

**优化方案**：Query 改写、混合检索、重排序、语义分块、多轮检索
