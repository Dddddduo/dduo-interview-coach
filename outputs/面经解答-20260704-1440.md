## 第1题：Spring AI 结构化输出实现旅游报告生成，将 AI 回复转为结构化价格对象，展开讲解

### ✅ Spring AI 结构化输出实现旅游报告生成

#### 🧠 联想记忆法

**记忆口诀/联想**: "AI 说人话 → POJO 说结构话" — 把 LLM 的"散文"用 Spring AI 的 `BeanOutputConverter` "翻译"成 Java 对象，就像旅行社把散乱的价格信息填入标准化的 Excel 表格。

**记忆原理**:
核心意象是"翻译官"——LLM 输出的是自然语言（如"机票 2000 元，酒店 800 元/晚"），Spring AI 的 `StructuredOutputConverter` 充当翻译官，将这段"散文"按照预定义的 JSON Schema（对应 Java POJO）重新组织成结构化的 `PriceInfo` 对象。记忆锚点是：**非结构化输入 → 结构化输出 = LLM 自由文本 → POJO 约束**。

**关联知识**:
- Jackson 的 `@JsonProperty` 注解映射 JSON → Java 对象，Spring AI 在此基础上增加了 LLM-aware 的 schema 生成和 prompt 注入
- Java 的 `Record` 类型（Java 14+）天然适合做不可变的数据传输对象（DTO, Data Transfer Object）
- 与传统的正则提取 JSON block 的方案对比：正则提取是"事后补救"，Spring AI Structured Output 是"事前约定"

#### 📖 深度解答

##### 1. 核心概念（是什么）

**Spring AI Structured Output（结构化输出）** 是 Spring AI 框架提供的一种机制，用于将大语言模型（Large Language Model, LLM）返回的非结构化自然语言文本，自动映射为预先定义的 Java POJO（Plain Old Java Object）或 Record 类型。

传统上，调用 LLM API 获取的是字符串文本（如 OpenAI 的 `choices[0].message.content`），开发者需要自行解析这段文本——提取 JSON 片段、处理格式异常、手动反序列化。Spring AI Structured Output 从根本上解决了这一问题：**在调用 LLM 之前就约定好输出格式，让 LLM 直接生成符合 JSON Schema 的结构化数据**，框架自动完成反序列化。

**核心价值**：
- 类型安全（Type Safety）：编译期即可确定输出类型，而非运行时字符串
- 消除样板代码（Boilerplate Elimination）：无需手动编写 JSON 提取和解析逻辑
- 一致性保证（Consistency Guarantee）：通过 prompt 工程 + schema 约束，大幅提升输出格式的稳定性

##### 2. 底层原理（为什么）

Spring AI Structured Output 的底层实现遵循 **"Schema 注入 + 结构化解析 + 容错降级"** 的三层架构：

**第一层：Schema 注入（Prompt Engineering Layer）**
Spring AI 在发送给 LLM 的 prompt 中自动附加一段 **JSON Schema 描述**。例如，当目标类型是 `PriceInfo` 时，框架会生成类似以下的 schema 声明：

```
You MUST respond in JSON format following this schema:
{
  "type": "object",
  "properties": {
    "itemName": {"type": "string"},
    "amount": {"type": "number"},
    "currency": {"type": "string"},
    "date": {"type": "string"},
    "category": {"type": "string"}
  },
  "required": ["itemName", "amount", "currency"]
}
```

这一层利用了 LLM 的 **指令遵循能力（Instruction Following）**——现代 LLM（GPT-4、Claude 3 等）经过训练后能够高度可靠地按照指定格式输出 JSON。

**第二层：结构化解析（Parsing Layer）**
`BeanOutputConverter` 是核心类，其工作流程如下：

```
BeanOutputConverter.convert(text) 的调用链：
1. 接收 LLM 返回的原始字符串 text
2. 尝试从 text 中提取 JSON 块（查找第一个 { 和最后一个 })
3. 使用 Jackson ObjectMapper 反序列化为目标类型
4. 如果失败，尝试常见的 LLM 输出修复（如去除 markdown 代码块标记 ```json）
```

关键源码逻辑：

```java
public class BeanOutputConverter<T> implements StructuredOutputConverter<T> {
    private final Class<T> targetClass;
    private final ObjectMapper objectMapper;
    
    @Override
    public T convert(String text) {
        // 尝试提取并解析 JSON
        String json = extractJsonBlock(text);
        try {
            return objectMapper.readValue(json, targetClass);
        } catch (JsonProcessingException e) {
            // 降级：使用 AI 自身修复格式
            return attemptFixAndRetry(text, e);
        }
    }
}
```

**第三层：容错降级（Fallback Layer）**
当 LLM 输出的格式不符合预期时，Spring AI 提供了多层防护：
1. **格式检测**：检测 JSON 结构完整性
2. **重试机制**：重新调用 LLM，附加"请确保输出符合 JSON 格式"的指令
3. **默认值降级**：在 POJO 中定义 `@JsonProperty(defaultValue = ...)` 或字段默认值

**设计哲学**：Spring AI 选择"约束 LLM"而非"事后解析"——与其费力处理混乱的输出，不如从一开始就让 LLM 输出规范格式。这体现了 **"Shift Left"（左移）** 的质量理念，即在问题发生的更早阶段解决它。

**Service 层**：

```java
@Service
public class TravelReportService {
    
    @Autowired
    private ChatClient chatClient;
    
    @Autowired
    private StructuredOutputConverter<TravelReport> converter;
    
    public TravelReport analyzeTravelCost(ReportRequest request) {
        // 1. 构造带 Schema 约束的 Prompt
        String prompt = buildTravelPrompt(request);
        
        // 2. 调用 AI 并获取结构化输出
        String aiResponse = chatClient.prompt()
            .user(prompt)
            .call()
            .content();
        
        // 3. 解析为 POJO
        try {
            return converter.convert(aiResponse);
        } catch (Exception e) {
            // 4. 异常处理 —— 重试 + 降级
            return handleConversionFailure(request, aiResponse, e);
        }
    }
    
    private TravelReport handleConversionFailure(ReportRequest request, 
                                                   String failedResponse, 
                                                   Exception originalException) {
        // 策略1：重试（最多 2 次）
        for (int i = 0; i < 2; i++) {
            try {
                String retryResponse = chatClient.prompt()
                    .user(buildRetryPrompt(request, i))
                    .call()
                    .content();
                return converter.convert(retryResponse);
            } catch (Exception ignored) {
                // 继续重试
            }
        }
        
        // 策略2：返回默认值（降级）
        logger.warn("AI 结构化输出解析失败，返回默认值", originalException);
        return buildDefaultReport(request);
    }
}
```

private String buildRetryPrompt(ReportRequest request, int attempt) {
    return """
        上次输出格式不符合要求。请严格遵循 JSON Schema 输出，
        不要添加任何额外文字。

        ## 旅行信息
        - 目的地：%s
        - 行程：%s 至 %s

        ## JSON Schema
        %s

        重要：只输出 JSON，不要 markdown 标记。
        这是第 %d 次尝试。
        """.formatted(
            request.destination(),
            request.startDate(),
            request.endDate(),
            converter.getJsonSchema(),
            attempt + 1
        );
}

##### 4. 深入思考（注意事项）

###### 4.1 异常处理矩阵

| 异常类型 | 原因 | 处理策略 | 优先级 |
|---------|------|---------|-------|
| JsonParseException | LLM 输出非 JSON 文本 | 重试 + 加强 prompt 约束 | 高 |
| UnrecognizedPropertyException | 字段名不匹配 | 添加 `@JsonIgnoreProperties(ignoreUnknown = true)` | 中 |
| NumberFormatException | 金额格式错误 | 自定义反序列化器 + 默认值 | 中 |
| 空返回值 | AI 拒绝回答或 API 超时 | 重试 + Circuit Breaker（断路器） | 高 |
| Schema 不匹配 | LLM 遗漏必需字段 | 校验后自动补全默认值 | 中 |

###### 4.2 与传统的 JSON 解析方案对比

| 维度 | Spring AI Structured Output | 传统方案（提取 JSON + Jackson） |
|------|---------------------------|-------------------------------|
| **工作时机** | 调用前约束 LLM 输出格式 | 调用后手动解析和清洗 |
| **类型安全** | 编译期类型检查 | 运行时字符串操作 |
| **配置复杂度** | 自动注入 Schema 到 prompt | 需要手动编写 JSON 提取正则 |
| **格式稳定性** | 高（LLM 被明确告知格式） | 低（依赖 LLM 随机输出） |
| **错误处理** | 内置重试和降级策略 | 需要自行实现 |
| **维护成本** | 低（POJO 变更自动同步） | 高（解析逻辑需同步更新） |
| **支持的类型** | Java Record, POJO, List, Map | 任意（但解析自实现） |
| **Prompt 侵入性** | 自动注入 schema 声明 | 无侵入（完全手动控制 prompt） |

**传统方案的问题代码示例**：

```java
// 传统方案：手动提取 JSON 块 + Jackson 反序列化
private PriceInfo parsePriceInfo(String aiResponse) {
    // Step 1: 手动提取 JSON 块 —— 脆弱且容易出错
    int start = aiResponse.indexOf('{');
    int end = aiResponse.lastIndexOf('}');
    if (start == -1 || end == -1 || end <= start) {
        throw new IllegalArgumentException("无法找到 JSON 块");
    }
    String jsonBlock = aiResponse.substring(start, end + 1);
    
    // Step 2: 清洗 —— 需要处理 markdown 代码块、多余逗号等
    jsonBlock = jsonBlock.replace("```json", "").replace("```", "")
                         .replaceAll(",\\s*}", "}");  // 去除尾随逗号
    
    // Step 3: 手动反序列化
    try {
        return objectMapper.readValue(jsonBlock, PriceInfo.class);
    } catch (JsonProcessingException e) {
        // 解析失败，需要重试或返回默认值
        log.error("JSON 解析失败，原始响应: {}", aiResponse, e);
        return PriceInfo.defaults();
    }
}
```

###### 4.3 性能优化建议

1. **缓存 Schema 字符串**：`converter.getJsonSchema()` 的结果应缓存，避免重复生成
2. **异步调用**：AI 调用建议使用 `CompletableFuture` 或 Spring 的 `@Async` 异步执行
3. **连接池配置**：为 OpenAI/Anthropic 客户端配置合适的连接池参数
4. **超时设置**：设置合理的读超时（通常 30-60 秒，视模型而定）
5. **结果缓存**：相同输入的结果可以缓存（如 Redis），减少 AI 调用次数

###### 4.4 面试官可能追问的问题

- "如果 LLM 返回的是 XML 而非 JSON 怎么办？" → 使用 `OutputParser` 的 XML 变体或自定义 parser
- "如何保证金额计算的精确性？" → 使用 `BigDecimal` 而非 `Double`，并在 prompt 中指定精度
- "在多语言场景下如何处理？" → 在 prompt 中指定输出语言，`currency` 字段使用 ISO 4217 标准
- "与 Spring AI 的 Advisor 如何配合？" → 可结合 `RetryAdvisor` 实现自动重试

##### 常见误区

- **误区 1**："Structured Output 就是 JSON 解析"——实际上是 prompt 约束 + 结构化解析的组合，重点在"事前约束"而非"事后解析"
