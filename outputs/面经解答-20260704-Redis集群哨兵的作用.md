# Redis 集群、哨兵的作用

---

### 🧠 联想记忆法

**口诀："哨兵管活，集群管分；哨兵三职责，集群两核心"**

- **哨兵（Sentinel）→ "门卫"**：想象写字楼门口三个保安（哨兵集群）。他们不做前台登记（不存数据），只做三件事：盯着大门看老板在不在（监控）、老板倒了就找二把手顶上（故障转移）、打电话通知各部门（通知）。三个保安一起投票才可信（quorum 机制防止误判）。

### 📖 深度解答

#### 一、核心概念

**Redis Sentinel（哨兵）** 是 Redis 官方提供的高可用性（High Availability, HA）解决方案，本质上是一个独立运行的进程，负责监控 Redis 主节点（master）和从节点（replica）的运行状态，并在主节点发生故障时自动完成故障转移（failover），将从节点提升为新的主节点。

**Redis Cluster（集群）** 是 Redis 官方提供的分布式（distributed）解决方案，同时实现了**数据分片（data sharding）**和**高可用性**两个目标。它通过将数据自动分散到多个节点上，解决了单机 Redis 内存容量受限和读写瓶颈的问题。

两者的核心区别在于：Sentinel 解决的是"主节点挂了怎么办"的问题，Cluster 解决的是"数据太多一台装不下怎么办"的问题。

---

#### 二、底层原理

##### 2.1 Sentinel（哨兵）底层机制

**1）三大核心职责**

- **监控（Monitoring）**：哨兵进程定期向所有 Redis 实例发送 PING 命令，检查其是否存活。
- **自动故障转移（Automatic Failover）**：当主节点不可用时，哨兵自动选举一个从节点提升为主节点，并更新其他从节点复制指向。
- **通知（Notification）**：通知客户端新的主节点地址，客户端据此更新连接。

**2）主观下线与客观下线**

哨兵对节点的下线判定分两个阶段：

```
SDOWN（Subjective Down，主观下线）：
  → 哨兵向节点发送 PING 后在配置的 down-after-milliseconds 时间内未收到有效回复
  → 该哨兵单方面认为节点已下线

ODOWN（Objective Down，客观下线）：
  → 当 Sentinel 集群中达到 quorum 数量的哨兵都认为主节点 SDOWN
  → 触发真正的故障转移流程
```

配置示例（sentinel.conf）：

```conf
# 监控 mymaster，IP 为 127.0.0.1，端口 6379，quorum 为 2
sentinel monitor mymaster 127.0.0.1 6379 2

# 主观下线判定时间：30000ms 内无响应即认为 SDOWN
sentinel down-after-milliseconds mymaster 30000

# 故障转移超时：180s
sentinel failover-timeout mymaster 180000

# 同时可并行同步的新主从节点数
sentinel parallel-syncs mymaster 1
```

**3）领导者选举与故障转移流程**

```
步骤 1：哨兵集群发现主节点 ODOWN（至少 quorum=2 个哨兵确认）
步骤 2：哨兵间通过 Raft 类算法选举一个领导者（leader sentinel）
步骤 3：领导者从从节点中选出新主节点（优先级 repl-priority 最高 → 复制偏移量最大 → runid 最小）
步骤 4：向新主节点发送 SLAVEOF NO ONE 命令
步骤 5：向其他从节点发送 SLAVEOF <new-master-ip> <new-master-port> 更改复制目标
步骤 6：更新记录的 master 地址，通知客户端
```

**4）为什么哨兵也要集群**

- **避免单点故障（Single Point of Failure, SPOF）**：如果只有一个哨兵，它自己挂了就无法监控。
- **quorum 投票机制需要**：客观下线判定需要多个哨兵投票，至少 quorum 个哨兵达成一致，防止网络抖动导致的误判。推荐的哨兵数量为奇数（如 3 个或 5 个），quorum 设为 N/2 + 1。

##### 2.2 Cluster（集群）底层机制

**1）16384 个哈希槽（Hash Slot）**

Redis Cluster 采用**一致性哈希（Consistent Hashing）的变体——哈希槽（Hash Slot）分区**。整个键空间被划分为 **16384** 个槽位，每个键通过 CRC16 算法计算后对 16384 取模，确定所属槽：

```
HASH_SLOT = CRC16(key) % 16384
```

集群中的每个节点负责一部分哈希槽，例如：
- 节点 A：槽 0~5460
- 节点 B：槽 5461~10922
- 节点 C：槽 10923~16383

当新增或移除节点时，只需移动部分槽位，无需全量重哈希。

**2）客户端重定向机制**

```bash
# MOVED 重定向（永久性）—— 键所属的槽由另一个节点负责
# 客户端应更新缓存的路由表，后续请求直接发往目标节点
$ redis-cli -c -p 7000
127.0.0.1:7000> SET name "redis"
-> MOVED 15495 127.0.0.1:7002
# 客户端收到 MOVED 错误，连接到 127.0.0.1:7002 重新发送命令
127.0.0.1:7002> SET name "redis"
OK

# ASK 重定向（临时性）—— 槽正在迁移中，仅当前请求需要重定向
127.0.0.1:7000> GET migrating_key
-> ASK 12345 127.0.0.1:7002
# 客户端先发送 ASKING 命令，再重新发送 GET 命令
127.0.0.1:7000> ASKING
OK
127.0.0.1:7000> GET migrating_key
"value"
```

两者区别总结：

| 特性 | MOVED | ASK |
|------|-------|-----|
| 性质 | 永久重定向 | 临时重定向 |
| 触发场景 | 槽已迁移完成 | 槽正在迁移中 |
| 客户端行为 | 更新路由表，后续直连新节点 | 不更新路由表，每次先 ASKING |

**3）Gossip（八卦）协议**

集群节点之间不通过中心化的协调者（coordinator）通信，而是通过 **Gossip 协议**（一种去中心化的最终一致性协议）交换元数据信息：

```
PING/PONG 消息中包含：
- 自身节点状态
- 随机选取的若干其他节点状态（包含节点 ID、IP:Port、槽位信息、主从关系）
- 纪元（epoch）信息用于版本控制
```

Gossip 的优点是**去中心化、容错性好、节点可动态扩缩容**，缺点是**消息传播有延迟（最终一致性，eventual consistency）**。

**4）集群节点配置示例**

```conf
# redis-cluster-7000.conf
port 7000
cluster-enabled yes                    # 启用集群模式
cluster-config-file nodes-7000.conf    # 集群节点配置文件
cluster-node-timeout 5000              # 节点超时时间（ms）
appendonly yes

# 创建集群（3 主 3 从，6 个节点）
$ redis-cli --cluster create \
  127.0.0.1:7000 127.0.0.1:7001 \
  127.0.0.1:7002 127.0.0.1:7003 \
  127.0.0.1:7004 127.0.0.1:7005 \
  --cluster-replicas 1
```

**5）集群主从结构**

每个分片（shard）由一个主节点和多个从节点组成。当主节点宕机时，其从节点会自动提升为新的主节点，保证该分片的高可用性。这就是 Cluster 既实现数据分片又实现高可用的方式。

#### 三、实践应用

##### 3.1 Sentinel 适用场景

- 数据量在单机内存范围内（如 < 32GB），但需要高可用保障的业务
- 追求强一致性（strong consistency），所有写操作集中在单主节点
- 不需要数据分片的场景，如缓存层、会话存储等

##### 3.2 Cluster 适用场景

- 数据量超过单机内存容量，需要水平扩展（horizontal scaling）
- 高并发写入需求，可以通过多主节点分散写入压力
- 对最终一致性可接受的业务场景（因为集群间数据同步有延迟）

##### 3.3 两者对比

| 维度 | Sentinel | Cluster |
|------|----------|---------|
| 数据分布 | 全量数据在单一主节点 | 数据分片到多个主节点 |
| 高可用机制 | 故障转移后从节点升主 | 每个分片自带主从复制 |
| 写扩展性 | 单点写入，无水平扩展 | 多主节点并行写入 |
| 读扩展性 | 可从从节点读扩展 | 各分片从节点读取 |
| 一致性 | 强一致性（单主） | 最终一致性 |
| 复杂度 | 配置简单，运维友好 | 配置复杂，需管理分片 |
| 客户端要求 | 普通 Redis 客户端即可 | 需支持集群协议的客户端 |

#### 四、深入思考

##### 4.1 架构演进路径

```
单机模式
  │ 瓶颈：单点故障（single point of failure），宕机即服务不可用
  ▼
主从复制（Master-Replica）
  │ 解决：从节点做读备份，主节点故障后手动切换
  │ 瓶颈：切换需人工介入，存在业务中断时间
  ▼
哨兵模式（Sentinel）
  │ 解决：自动故障转移，无需人工介入
  │ 瓶颈：数据仍全量在单节点，内存和写入瓶颈受限
  ▼
集群模式（Cluster）
  │ 解决：数据分片实现水平扩展，每个分片自带高可用
  │ 瓶颈：运维复杂度提升，跨分片操作受限（如 multi-key 操作需在同一槽）
  ▼
Redis 7.0+ 增强
  │ Function、Sharded Pub/Sub、ACL 改进
```

##### 4.2 面试高频追问及其应对

**追问 1："为什么哈希槽是 16384 个而不是更多？"**

答：16384（2^14）是 Redis 官方在性能和内存之间权衡的结果。如果槽位太多，集群节点间通过 Gossip 协议交换的位图（bitmap）消息体过大（每 16384 bit = 2KB），心跳消息占用带宽过多；如果太少，数据分布不够均匀。16384 在大多数集群规模（通常不超过 1000 个节点）下能保证数据均匀分布，同时心跳包大小控制在合理范围内。

**追问 2："Sentinel 选举新主节点时如何选择？"**

答：按照以下优先级逐级筛选：
1. 剔除已下线的从节点
2. 选择 repl-priority（复制优先级）配置值最小的（值越小优先级越高）
3. 选择复制偏移量（replication offset）最大的，即数据最接近原主节点的
4. 选择 run ID 最小的（字典序）

**追问 3："Cluster 不支持多键操作吗？"**

答：Cluster 通过 **hash tags** 机制支持多键操作。将多个键放入同一个哈希标签（如 `{user:123}:name` 和 `{user:123}:age`），CRC16 只计算 `{}` 内的内容，确保这些键落在同一槽内，从而支持事务（MULTI/EXEC）和管道（pipeline）操作。

### 🗺️ 回答思路

**第一层：定义先行，厘清概念（30 秒）**

面试官问"Redis 集群、哨兵的作用"，首先要明确区分 Sentinel 和 Cluster 是两个不同的东西。建议用一句话点出本质差异：**Sentinel 是高可用方案，Cluster 是分布式分片方案**。让面试官知道你清楚地理解两者定位不同。

**第二层：分层展开，结构清晰（2 分钟）**

按"Sentinel → Cluster → 对比 → 演进"的结构展开，每个部分按"是什么→怎么工作→关键细节"的方式讲述：

- **Sentinel 部分**：强调 SDOWN→ODOWN 的两阶段判定，quorum 投票机制，为什么需要奇数个哨兵。顺便提一句"哨兵不做数据分片"避免混淆。
- **Cluster 部分**：讲清楚 16384 哈希槽 + CRC16 取模的映射方式。MOVED vs ASK 是高频考点，一定要对比讲。Gossip 协议可以说出"去中心化、最终一致性"等关键词。
- **对比部分**：建议用表格形式呈现，直观体现维度差异。

**第三层：实战经验，体现深度（1 分钟）**

- 提到 Sentinel 的 `down-after-milliseconds` 配置需要根据网络质量调整，过短易误判，过长影响恢复速度。
- 提到 Cluster 的 hash tags 可以解决多键操作问题，展示你对实际开发痛点的理解。
- 如果时间允许，可以提到 Redis 7.0 对集群功能的增强（如 Sharded Pub/Sub），说明你关注版本演进。

**第四层：避坑提示**

- ❌ 避免：把 Sentinel 和 Cluster 混为一谈，说"Redis 集群就是主从加哨兵"
- ❌ 避免：说 Cluster 的哈希槽是 16384 个节点，这是常见误读
- ✅ 正确：Cluster 有 16384 个槽，不是 16384 个节点
- ✅ 补充：可以用位图（bitmap）的思想解释每个节点如何知道自己负责哪些槽

**如果面试官追问，准备以下延伸方向**：

- Redis Cluster 的**脑裂（split-brain）问题**如何处理（cluster-node-timeout + 多数派机制）

- 对比一致性哈希（Consistent Hashing）vs 哈希槽（Hash Slot）的设计差异
