## 📖 深度解答

### 一、核心概念（Core Concepts）

本项目是一个基于 **RAG 架构（Retrieval-Augmented Generation，检索增强生成）** 的企业级智能知识问答系统，采用 **Spring AI**（Java 生态的 AI 框架）和 **LangChain4j** 作为 AI 编排层，后端以 Spring Boot 3.x 构建微服务架构，前端基于 React + TypeScript 实现交互界面。

项目定位为**企业私有知识库的 AI 问答助手**，核心解决场景是：企业内部存在大量分散的文档（技术手册、产品文档、运维日志），员工需要从海量文档中快速获取准确答案。传统搜索引擎依赖关键词匹配，无法理解语义；直接使用通用 LLM 又因缺乏领域知识和存在幻觉而不可靠。RAG 架构恰好弥补这一缺口——先检索相关文档片段，再将其注入 LLM 的上下文窗口（Context Window），由 LLM 基于检索结果生成回答。

### 二、底层原理（Underlying Principles）

#### 2.1 RAG 全链路原理

RAG 架构的核心公式可以表达为：

```
Answer = LLM.generate(Query ⊕ Retrieve(Top-K, Embed(Query, DocumentStore)))
```

其中：

- **Embed(·)**：将文本转换为稠密向量（Dense Vector），使用 text-embedding-ada-002 / bge-large-zh-v1.5 等嵌入模型（Embedding Model）
- **Retrieve(·)**：在 Vector Store（向量数据库）中进行近似最近邻搜索（ANN, Approximate Nearest Neighbor），常用算法包括 HNSW（Hierarchical Navigable Small World）和 IVFFlat（Inverted File with Flat Indexing）
- **⊕**：将检索到的上下文片段拼接至 Prompt 模板，形成增强后的输入
- **LLM.generate(·)**：调用大语言模型生成最终答案

#### 2.2 向量检索原理

文本向量化本质上是将离散的语义信息映射到高维连续空间（通常为 768 维或 1536 维）。语义相近的文本在向量空间中的余弦距离（Cosine Distance）更小。HNSW 算法通过构建多层跳表结构的图索引，将检索时间复杂度从 O(n) 降低到 O(log n)。

```
// 伪代码：向量检索流程
List<Document> retrieveRelevantDocs(String query, int topK) {
    float[] queryVector = embeddingModel.embed(query);    // 文本 -> 向量
    return vectorStore.similaritySearch(                  // ANN 搜索
        SearchRequest.query(queryVector)
            .withTopK(topK)                              // 返回 top-K
            .withSimilarityThreshold(0.75)               // 相似度阈值
    );
}
```

#### 2.3 Chat Completion 原理

LLM 的 Chat Completion 本质上是自回归（Autoregressive）的文本生成过程：给定 Token 序列，模型逐个预测下一个 Token 的概率分布。通过 Temperature（温度系数，控制输出的随机性）、Top-P（核采样，控制候选词累积概率）等参数调节生成风格。

#### 3.2 核心功能模块

**模块一：智能问答（Intelligent Q&A）**

```java
// Spring AI 实现 Chat Completion
@RestController
@RequestMapping("/api/chat")
public class ChatController {
    
    private final ChatClient chatClient;
    private final VectorStore vectorStore;
    
    @PostMapping("/ask")
    public Mono<String> ask(@RequestBody QuestionRequest request) {
        // 1. 向量检索增强
        List<Document> docs = vectorStore.similaritySearch(
            SearchRequest.query(request.question())
                .withTopK(5)
                .withSimilarityThreshold(0.7)
        );
        
        // 2. 构建增强 Prompt
        String context = docs.stream()
            .map(Document::getContent)
            .collect(Collectors.joining("\n---\n"));
        
        // 3. 调用 LLM 生成答案
        return chatClient.prompt()
            .system(s -> s.text("你是一个企业知识库助手。请基于以下文档片段回答问题。"
                + "如果文档中没有相关信息，请明确说明。"))
            .user(u -> u.text("文档片段：\n" + context + "\n\n问题：" + request.question()))
            .call()
            .content();
    }
}
```

**模块二：文档智能分析（Document Analysis）**

支持 PDF / Word / Markdown 文档的批量导入：

- 文档解析（Document Parsing）：使用 Apache Tika / Unstructured.io 提取文本

- 向量入库（Indexing）：逐片 Embedding 后写入 pgvector

**架构要点**：

- **前后端分离**（Front-End / Back-End Separation）：前端 SPA 通过 REST API 与后端通信，支持 SSE（Server-Sent Events）实现流式输出（Streaming Output）
- **AI 网关层**：统一管理模型路由，主模型（GPT-4o）故障时自动降级至备模型（qwen-max），集成限流（Rate Limiting）和熔断（Circuit Breaker）
- **模型服务层**：Embedding 模型独立部署，避免与 LLM 争抢 GPU 资源；Vector Store 采用 pgvector，与业务数据同库，降低运维复杂度

### 四、深入思考（Deep Reflections）

#### 4.1 挑战与解决方案

| 挑战 | 具体表现 | 解决方案 |
|------|---------|---------|
| 模型幻觉（Hallucination） | LLM 生成看似合理但事实错误的内容 | RAG 检索注入 + Prompt 约束 "无相关信息则明确声明" + 答案引用溯源 |
| Token 限制（Token Limit） | 长文档超出上下文窗口（如 128K tokens） | 智能分块 + 滑动窗口 + 关键片段优先召回 |
| 响应速度（Latency） | 端到端响应 >5s，用户等待感强 | 流式输出 SSE + 向量索引优化（HNSW） + 缓存热点问题 |
| 成本控制（Cost Control） | API 调用费用随用量线性增长 | 多级缓存（Redis 热点缓存 + Embedding 结果缓存）+ 备用小模型处理简单查询 |

#### 4.2 模型幻觉的深层治理

模型幻觉的根因在于 LLM 本质是**概率生成模型（Probabilistic Generative Model）**，其训练目标是最大化下一个 Token 的对数似然（Log-Likelihood），而非事实正确性。解决幻觉需要从三个维度入手：

1. **检索质量（Retrieval Quality）**：优化 Chunking 策略——基于 Markdown 标题层级进行分层分块（Hierarchical Chunking），确保每个 Chunk 语义完整
2. **注入策略（Injection Strategy）**：采用 "先检索后回答"（Retrieve-then-Read）而非 "边生成边检索"（Generate-then-Retrieve），降低对模型内部知识的依赖
3. **验证机制（Verification Mechanism）**：对生成的答案逐句检索来源文档，计算语义相似度，低于阈值的句子标注为"低可信度"
