# Redis 主从切换、异常导致锁丢失，有什么补救措施

## 📖 深度解答

### 一、核心概念

#### 1.1 问题场景还原

在 Redis 分布式锁（Distributed Lock）的典型实现中，客户端向 Redis 主节点（Master）写入一个带有过期时间（TTL）的 Key 来获取锁。Redis 的主从复制（Master-Slave Replication）默认是异步的（Asynchronous Replication）——Master 将命令写入复制积压缓冲区（Replication Backlog），Slave 通过 `replicaof` 命令异步拉取。

当出现以下时序时锁会丢失：

```
T0: Client A 向 Master SET lock:key random_value NX PX 30000  → 加锁成功
T1: Master 尚未将命令同步到 Slave（异步延迟）
T2: Master 宕机
T3: Sentinel 执行故障转移（Failover），Slave 晋升为 Master
T4: Client B 向新 Master SET lock:key random_value2 NX PX 30000  → **加锁成功！**
→ Client A 和 Client B 同时认为自己持有锁，锁的互斥性（Mutual Exclusion）被打破
```

这就是 Redis 分布式锁在主从切换场景下的**锁丢失（Lock Loss）**问题。

#### 1.2 根因分析

| 因素 | 说明 |
|------|------|
| **异步复制** | `SET` 命令先返回客户端成功，再异步同步到 Slave，无法保证主从强一致（Strong Consistency） |
| **AP 倾向** | Redis 设计优先保证可用性（Availability）和分区容错性（Partition Tolerance），牺牲了一致性（Consistency），属于 CAP 理论的 AP 系统 |
| **故障切换时机** | Failover 可能在复制完成前的任意时刻发生 |

---

### 二、底层原理

#### 2.1 Redis 主从复制的异步机制

Redis 主从复制围绕 `replicationId` 和 `replicationOffset` 两个核心概念工作：

- Master 维护一个**复制积压缓冲区（Replication Backlog）**，默认 1MB 环形缓冲区
- Slave 发送 `PSYNC {replicationId} {offset}` 命令请求增量同步
- 如果 slave 的 offset 落后太多或 replicationId 不匹配，触发全量重同步（Full Resynchronization），生成 RDB 快照传输

**关键缺陷**：Master 执行完 `SET lock:key value NX PX 30000` 后立即返回 `+OK` 给客户端，然后将命令写入 backlog。如果 Master 在 backlog 被 Slave 拉取之前宕机，这条锁记录就永远丢失了。

Redis 官方在 `redis.conf` 中提供了 `min-replicas-to-write` 和 `min-replicas-max-lag` 参数，可以要求 Master 在写入时至少有一定数量的 Slave 确认（Weakly Acknowledged），但这只能降低概率，**无法从根本解决**——因为这是异步模型的天生局限。

#### 2.2 分布式锁的正确性要求

一个可靠的分布式锁必须满足三个性质：

| 性质 | 英文 | 说明 |
|------|------|------|
| **互斥性** | Mutual Exclusion | 任何时刻只有一个客户端持有锁 |
| **死锁逃生** | Deadlock Freedom | 持有锁的客户端崩溃后，锁能自动释放（通过 TTL 实现） |
| **容错性** | Fault Tolerance | 部分节点宕机不影响锁的整体正确性 |

当主从切换发生时，互斥性被破坏——这就是需要补救措施的根源。

### 三、实践应用：五大补救方案

#### 方案一：RedLock（红锁算法）

**提出者**：Redis 作者 Antirez（Salvatore Sanfilippo）

**核心思想**：不再依赖单个 Redis 主从结构，而是在 N（通常 N=5）个**完全独立**的 Redis 节点上同时申请锁，只要在 `N/2 + 1` 个节点上成功获取到锁，就认为加锁成功。

```
N = 5, 多数阀 = 3

Client 向 5 个独立 Redis 节点分别执行 SET key value NX PX 30000
成功数 = 4 ≥ 3 → 加锁成功
失败数 = 1
```

**实现步骤**：
1. 获取当前毫秒时间戳 `start_time`
2. 依次向 N 个 Redis 节点设置锁，每个 SET 的超时时间远小于锁 TTL（比如锁 TTL=10s，单节点超时=50ms）
3. 计算获取锁的耗时 `elapsed = now - start_time`
4. 如果成功节点数 >= N/2+1 **且** elapsed < lock_ttl，则锁有效
5. 释放锁时，删除所有 N 个节点上的 Key

**Java 伪代码示例**：
```java
public class RedLock {
    private static final int N = 5;
    private static final long LOCK_TTL_MS = 30000;
    private static final long PER_NODE_TIMEOUT_MS = 100;
    private List<RedisClient> nodes = initNodes(N);

    public boolean tryLock(String key, String value) {
        long start = System.currentTimeMillis();
        int successCount = 0;

        for (RedisClient node : nodes) {
            try {
                // 每个节点设置极短超时，防止某个节点卡死整个流程
                String result = node.set(key, value, 
                    SetArgs.Builder.nx().px(LOCK_TTL_MS), 
                    PER_NODE_TIMEOUT_MS);
                if ("OK".equals(result)) successCount++;
            } catch (TimeoutException e) {
                // 单个节点超时不影响整体
            }
        }

        long elapsed = System.currentTimeMillis() - start;
        int quorum = N / 2 + 1;

        // 多数节点成功 且 获取锁耗时小于锁TTL
        if (successCount >= quorum && elapsed < LOCK_TTL_MS) {
            return true; // 加锁成功
        }

        // 加锁失败，立即释放所有节点的锁
        for (RedisClient node : nodes) {
            node.delete(key);
        }
        return false;
    }
}
```

**优缺点**：

| 维度 | 评价 |
|------|------|
| 优点 | 不依赖主从关系，独立节点间故障隔离；理论证明保证互斥性 |
| 缺点 | 需要 5 个独立节点，运维成本高；依赖客户端时钟同步；N/2+1 机制在极端网络分区下仍有争议（Martin Kleppmann 曾撰文批评） |
| 适用场景 | 对一致性有较高要求、可接受额外硬件成本的场景 |

#### 方案二：延迟重启（Delay Restart / Delayed Replica Promotion）

**核心思想**：在主从切换后，延迟从节点晋升为主节点的时间，等待旧 Master 上尚未同步的锁自然过期（超过 TTL）。

```
旧 Master 宕机时间：T0
锁的 TTL：30 秒
新 Master 启动时间：T0 + 30s（等待至少一个锁 TTL 时长）
→ 所有旧锁已自动过期，新 Master 上不存在未同步的锁数据
```

**实现方式**：
- 在 Redis Sentinel 或 Redis Cluster 的故障转移逻辑中，增加一个 `wait_before_failover` 配置项，值设为锁 TTL 的上界
- 或者在应用的业务层面处理：连接旧 Master 的客户端先等待锁过期时间再重试

**代码示例（Sentinel 配置思路）**：
```bash
# sentinel.conf
sentinel wait-before-failover mymaster 30000  # 延迟 30 秒
```

**优缺点**：

| 维度 | 评价 |
|------|------|
| 优点 | 实现简单，无需额外组件；对现有系统侵入最小 |
| 缺点 | 牺牲了可用性（等待期间新 Master 不可用）；锁 TTL 越大，等待时间越长；无法处理 TTL 未设置或永久锁的情况 |
| 适用场景 | 锁 TTL 较短（< 10s）、可用性要求不是极致的情况 |

#### 方案三：使用 ZooKeeper / etcd 实现锁

**核心思想**：ZooKeeper 和 etcd 是 CP 系统（Consistency + Partition Tolerance），其底层一致性协议（ZAB / Raft）保证了写入的强一致性——写入必须过半节点确认后才算成功。

**ZooKeeper 实现原理**：
- 利用 ZooKeeper 的**临时顺序节点（Ephemeral Sequential Node）**
- 客户端在锁路径下创建临时顺序节点，节点序号最小的持有锁
- 其他客户端监听前一个节点的删除事件（Watcher 机制）
- 临时节点特性保证客户端断开后自动删除，天然解决死锁

```
锁路径：/locks/my_lock
客户端 1 创建 /locks/my_lock/lock-0000000001  → 序号最小，持有锁
客户端 2 创建 /locks/my_lock/lock-0000000002  → 监听前一个节点
客户端 3 创建 /locks/my_lock/lock-0000000003  → 监听前一个节点
```

**etcd 实现原理**：
- 利用 etcd 的 `lease`（租约）和 `revision`（版本号）机制
- 通过 `TXN` 事务保证锁写入的原子性
- `lease` 自动续期，客户端崩溃后租约过期自动释放

**优缺点**：

| 维度 | ZooKeeper | etcd |
|------|-----------|------|
| 一致性协议 | ZAB (ZooKeeper Atomic Broadcast) | Raft |
| 锁方式 | 临时顺序节点 | Lease + Revision |
| 优点 | 强一致性保证，不会锁丢失；Watcher 机制高效通知 | Kubernetes 生态标配；gRPC 协议性能好 |
| 缺点 | 吞吐量低于 Redis（约 10K/s vs Redis 100K+）；羊群效应（Herd Effect） | 同样吞吐量受限；相对 Redis 更重 |
| 适用场景 | 金融、交易等对一致性要求极高的场景 | 云原生环境、K8s 集成场景 |

#### 方案四：Fencing Token（栅栏令牌）

**提出者**：Martin Kleppmann（《Designing Data-Intensive Applications》作者），他在对 RedLock 的批评文章中提出了 Fencing Token 思想。

**核心思想**：每次锁服务分配锁时，返回一个严格递增的全局序号（Token）。客户端在操作共享资源时，必须附带这个 Token。共享资源端（如数据库）检查 Token，拒绝过期 Token 的写入。

```
Fencing Token 服务（可基于 Redis INCR 或 ZooKeeper 的事务ID）：
  Token 1 → Client A 获取锁
  Token 2 → Client B 获取锁（即使 B 的锁获得晚，但 Token 值更大）

数据库端：
  UPDATE resource SET data = 'new_value', token = 2 WHERE id = 1 AND token < 2;
  
如果 Client A（Token=1）的网络延迟导致后来才到达数据库，该 UPDATE 会因为 
"token < 2" 条件不满足而失败。
```

**实现示例**：
```java
// 锁服务端（使用 Redis INCR 生成递增 Token）
long token = redisTemplate.opsForValue().increment("lock:token");

// 客户端写入共享资源时携带 Token
public boolean writeWithToken(String resourceId, String data, long token) {
    String sql = "UPDATE resource SET data = ?, token = ? " +
                 "WHERE id = ? AND token < ?";
    int rows = jdbcTemplate.update(sql, data, token, resourceId, token);
    return rows > 0; // 0 行更新表示 token 过期
}
```

**优缺点**：

| 维度 | 评价 |
|------|------|
| 优点 | 与具体锁实现解耦，作为**第二道防线**；数学上严格保证安全 |
| 缺点 | 需要改共享资源端代码，引入额外校验逻辑；Token 生成服务本身需保证严格递增（可能成为单点） |
| 适用场景 | 对数据一致性有严格要求的核心业务（如扣款、下单） |

#### 方案五：业务层幂等设计（Idempotent Design）

**核心思想**：承认分布式锁在最坏情况下可能失效，转而在业务层通过**幂等性（Idempotency）**兜底——即使锁失效导致同一个操作被执行多次，最终的业务状态仍然正确。

**常见实现手段**：

| 手段 | 说明 | 示例 |
|------|------|------|
| **唯一索引** | 数据库表对业务标识字段建唯一约束 | `UNIQUE KEY idx_order_no (order_no)` |
| **乐观锁** | 用版本号（Version）控制更新 | `UPDATE stock SET count = count - 1, version = version + 1 WHERE id = ? AND version = ?` |
| **状态机** | 业务状态严格流转，不允许回滚 | 状态 `PENDING → PROCESSING → SUCCESS`，不允许 `SUCCESS → PROCESSING` |
| **去重表** | 单独建一张业务去重表，利用唯一索引 | `insert into idempotent(id, biz_key) values(?, ?)` 返回 DuplicateEntry 则跳过 |

**代码示例**（乐观锁 + 幂等）：
```java
@Service
public class OrderService {
    
    @Transactional
    public boolean deductStock(Long productId, Integer quantity) {
        // 即使没有锁保护，乐观锁保证不会超卖
        String sql = "UPDATE product SET stock = stock - ?, version = version + 1 " +
                     "WHERE id = ? AND stock >= ? AND version = ?";
        int rows = jdbcTemplate.update(sql, quantity, productId, quantity, oldVersion);
        return rows > 0;
    }
}
```

**优缺点**：

| 维度 | 评价 |
|------|------|
| 优点 | 最朴素的"最后防线"，不依赖任何额外组件；唯一能 100% 兜底的方案 |
| 缺点 | 不是"锁"的替代方案，而是业务层面的兜底；对业务代码侵入性强 |
| 适用场景 | **所有分布式锁场景都应配合幂等设计**，这是最佳实践 |

### 四、深入思考

#### 4.1 方案对比总结

| 方案 | 一致性保证 | 运维复杂度 | 性能影响 | 额外依赖 | 推荐场景 |
|------|-----------|-----------|---------|---------|---------|
| RedLock | 较强（多数同意） | 高（5节点） | 中（N次网络IO） | 无 | 中等一致性要求 |
| 延迟重启 | 弱（等待过期） | 低 | 低 | 无 | 低一致性要求 |
| ZooKeeper/etcd | **强**（CP系统） | 中-高 | 较低（ZK约10K/s） | ZK/etcd集群 | **高一致性要求** |
| Fencing Token | 最强（严格递增） | 中 | 低 | 递增ID服务 | **核心金融场景** |
| 业务幂等 | 最强（业务级保证） | 中 | 低 | 无 | **所有场景兜底** |

#### 4.2 黄金组合建议

在实际生产系统中，建议**组合使用**多个方案，形成纵深防御：

```
第一层（预防）：RedLock / ZooKeeper → 尽量在锁层保证互斥
第二层（兜底）：Fencing Token → 防止过期锁的写入
第三层（最后防线）：业务幂等 → 即使前两层失效，业务状态仍然正确
```

#### 4.3 关于 RedLock 的学术争议

Martin Kleppmann 在 2016 年发表文章《How to do distributed locking》，指出 RedLock 存在根本性问题：
1. 依赖于客户端本地时钟（Local Clock），如果客户端 GC（Stop-The-World）导致时钟跳跃，锁可能提前释放
2. 没有 Fencing Token 机制，无法防止旧锁持有者写入

Antirez 在后续文章中回应：RedLock 的设计前提是时钟误差可控，且 Fencing Token 可以补充实现。

**实际工程共识**：
- 如果要求**绝对正确性**（如金融转账），使用 ZooKeeper/etcd + Fencing Token + 幂等
- 如果追求**高吞吐 + 可接受的概率性错误**（如缓存重建、非关键业务），RedLock 足够

#### 4.4 CAP 理论的权衡启示

这个问题的本质是 **AP vs CP 的权衡**：

| 选择 | 代表系统 | 特点 |
|------|---------|------|
| AP（可用性优先） | Redis + RedLock | 高吞吐，一致性弱，需业务兜底 |
| CP（一致性优先） | ZooKeeper / etcd | 吞吐受限，强一致，实现复杂 |

没有银弹（No Silver Bullet），选择取决于业务对一致性的敏感度。
