```
结构：我是谁（10s）→ 技术栈（20s）→ 核心项目+量化成果（90s）→ AI经验（30s）→ 求职动机（10s）

- **技术栈**：Java/Spring Cloud/MySQL/Redis/Kafka，一句话覆盖

- **钩子**：每个项目留一个"悬念"——让面试官忍不住追问

### Q2：你是怎么做慢SQL分析以及优化的？ {#q2}

#### 🧠 联想记忆法

**记忆口诀**: "**抓-剖-优-防**"四步法：抓日志 → 剖执行计划 → 优索引/SQL → 防复发。

**关联知识**: EXPLAIN 字段口诀——"**类键行，额外排序**"（type、key、rows、Extra），这四项是 EXPLAIN 中最重要的列。

#### 📖 深度解答

**1. 抓——定位慢SQL**
- 开启 `slow_query_log`，设置 `long_query_time`（如 1s）
- 或用 `performance_schema` 监控
- 生产环境用 `pt-query-digest` 分析慢查询日志

**2. 剖——EXPLAIN 分析**
重点关注 4 个字段：

| 字段 | 关注值 | 含义 |
|------|--------|------|
| type | ALL(全表扫) < index < range < ref < const | 访问类型 |
| key | NULL = 未用索引 | 实际使用的索引 |
| rows | 越大越慢 | 预估扫描行数 |
| Extra | Using filesort/Using temporary | 需要额外排序/临时表 |

**3. 优——优化手段**
- **加索引**: 最有效。WHERE/JOIN/ORDER BY 字段建联合索引
- **覆盖索引**: SELECT 列都在索引中，避免回表
- **改写SQL**: 避免 `SELECT *`、用 JOIN 代替子查询、LIMIT 分页优化
- **分库分表**: 单表 > 500 万行考虑

**4. 防——预防复发**
- 上线前 SQL Review
- 监控慢查询数量趋势
- 索引使用率监控

#### 🗺️ 回答思路

- **答题逻辑框架**: 先讲四步方法论 → 每步展开关键操作 → 给一个真实案例
- **重点得分点**: EXPLAIN 字段解读必须准确；区分"索引优化"和"SQL改写"两个维度
- **常见误区**: 只提加索引不提 SQL 改写；不说 type 的具体判断标准
- **过渡话术**: "我一般按四个步骤来排查..."

### Q3：Excel批量导入相关问题 {#q3}

#### 🧠 联想记忆法

**记忆口诀**: "**限-事-分**"——单次限流、事务回滚、分批上传。把"一万条"拆成 10 个"一千条"。

#### 📖 深度解答

**1. Excel批量导入上限是否有控制？**

必须控制，三层限流：
- **前端校验**: 文件上传前用 `FileReader` 读行数，超限禁用提交按钮
- **后端校验**: Controller 层解析后立即 `if (list.size() > 1000) throw BizException`
- **数据库层**: 使用批量插入 `rewriteBatchedStatements=true`，一次 batch 1000 条

**2. 一批插入失败如何处理？**

核心原则：**同一批次要么全成功，要么全失败。**

```java
@Transactional(rollbackFor = Exception.class)
public ImportResult batchImport(List<ExcelRow> rows) {
    try {
        jdbcTemplate.batchUpdate(sql, rows, 1000);
        return ImportResult.success(rows.size());
    } catch (Exception e) {
        // 记录失败日志，返回失败详情
        log.error("批量导入失败，行数={}", rows.size(), e);
        return ImportResult.fail("导入失败: " + e.getMessage());
    }
}
```

注意：用 `batchUpdate` 而非逐条 INSERT，性能差距可达 **10-50 倍**。

**3. 用户需要上传 1 万条，限定 500/1000 条如何处理？**

给用户三条路径：
- **客户端分批**: Excel 拆分功能——"检测到 10000 条数据，已自动拆为 10 个批次，逐批上传"
- **服务端分批**: 前端一次上传，后端异步拆分处理
- **异步任务模式**: 提交文件 → 返回任务 ID → 后台处理 → 轮询/推送结果

```java
// 推荐：异步分批处理
@Async
public void asyncBatchImport(MultipartFile file, String taskId) {
    List<List<ExcelRow>> batches = Lists.partition(parseExcel(file), 1000);
    for (int i = 0; i < batches.size(); i++) {
        batchImport(batches.get(i));
        updateProgress(taskId, (i + 1) * 100 / batches.size());
    }
}
```

#### 🗺️ 回答思路

- **答题逻辑框架**: 先讲控制的必要性 → 讲失败处理策略 → 讲大批量方案
- **重点得分点**: `@Transactional` + `batchUpdate`；异步处理方案；前端友好提示
- **常见误区**: 只在前端限制（可被绕过）；不处理批次边界（最后一批不足 1000 条的 case）

### Q5：Spring AI结构化输出实现旅游报告生成 {#q5}

#### 🧠 联想记忆法

**记忆口诀**: "**约-转-校**"——约定 Schema → 模型输出 JSON → 反序列化校验。

**关联知识**: 类比 API 开发中的"请求体校验"——只不过现在输出方是不可控的 LLM，所以校验必须严格。

#### 📖 深度解答

**核心原理**：

Spring AI 的结构化输出本质是：**用 JSON Schema 约束 LLM 的输出格式，让不可控的自然语言变为可控的结构化数据。**

```
用户输入旅行需求
    │
    ▼
构造 System Prompt（含 JSON Schema）
    │
    ▼
调用 LLM（要求按 Schema 输出 JSON）
    │
    ▼
获取 JSON 字符串
    │
    ▼
Jackson/ObjectMapper 反序列化 → PriceInfo 对象
    │
    ▼
业务校验（价格范围、字段完整性）
```

**代码示例**：

```java
// 1. 定义结构化输出 Schema
public record TourPrice(
    @JsonProperty(required = true) String destination,
    @JsonProperty(required = true) BigDecimal totalPrice,
    List<PriceItem> priceBreakdown,
    String currency
) {}

public record PriceItem(String item, BigDecimal amount, String note) {}

// 2. Spring AI 调用
@Service
public class TourReportService {
    
    private final ChatClient chatClient;
    
    public TourPrice generateTourPrice(String userRequirement) {
        // 构造带 Schema 约束的 Prompt
        String prompt = """
            根据用户需求生成旅游报价，按以下 JSON 格式输出：
            {
              "destination": "目的地",
              "totalPrice": 总价,
              "priceBreakdown": [
                {"item": "项目名", "amount": 金额, "note": "说明"}
              ],
              "currency": "CNY"
            }
            
            用户需求：%s
            """.formatted(userRequirement);
        
        // 调用 LLM
        String llmResponse = chatClient.call(prompt);
        
        // 结构化解析 + 校验
        try {
            TourPrice price = objectMapper.readValue(
                extractJson(llmResponse), TourPrice.class);
            validatePrice(price); // 业务校验
            return price;
        } catch (Exception e) {
            // 重试或降级
            return retryOrFallback(userRequirement, e);
        }
    }
    
    // 从 LLM 输出中提取 JSON（处理 LLM 可能包裹的 markdown 代码块）
    private String extractJson(String response) {
        return response
            .replaceAll("```json", "")
            .replaceAll("```", "")
            .trim();
    }
}
```

**为什么不用 Function Calling？**
- Function Calling 适合简单参数提取，不适合复杂嵌套对象
- 结构化输出 Schema 更灵活，可以定义任意复杂度

#### 🗺️ 回答思路

- **答题逻辑框架**: 为什么需要结构化 → Schema 设计 → 调用链路 → 异常处理
- **重点得分点**: `extractJson()` 方法体现了对 LLM 输出不可控的认知；`retryOrFallback` 体现了工程健壮性
- **常见误区**: 直接 `JSON.parse(llmOutput)` 不处理 markdown 包裹；不做业务校验

### Q6：Chat Memory是否做了持久化 {#q6}

#### 🧠 联想记忆法

**记忆口诀**: "**会话级在内存，历史级落盘**"——当前对话用 Redis/Memory 快存，长期历史写到 DB。

#### 📖 深度解答

**Chat Memory 持久化方案**：

**短期记忆（当前会话）**：
- 存在 Redis 中，用 sessionId 做 key
- 格式：`List<Message>`，保留最近 N 轮（如 20 轮）
- TTL 设为 30 分钟（会话超时）

**长期记忆（历史记录）**：
- 落 MySQL，表结构设计：
```sql
CREATE TABLE chat_history (
    id BIGINT PRIMARY KEY,
    session_id VARCHAR(64) NOT NULL,
    user_id BIGINT NOT NULL,
    role ENUM('user', 'assistant'),
    content TEXT,
    created_at DATETIME,
    INDEX idx_session (session_id, created_at)
);
```

**记忆管理策略**：
1. **滑动窗口**: 只保留最近 20 轮对话，超出的自动截断
2. **摘要压缩**: 旧对话用 LLM 生成摘要，替代原文
3. **分层存储**: 热数据 Redis → 温数据 MySQL → 冷数据归档

**Spring AI 实现**：
```java
@Bean
public ChatMemory chatMemory() {
    // 生产环境用 JdbcChatMemory
    return new JdbcChatMemory(dataSource);
    // 开发环境用 InMemoryChatMemory
    // return new InMemoryChatMemory();
}
```

#### 🗺️ 回答思路

- **答题逻辑框架**: 分层存储（热/温/冷）→ 表结构设计 → 淘汰策略
- **重点得分点**: 区分"会话内记忆"和"跨会话记忆"；滑动窗口 + 摘要压缩
- **常见误区**: 只说"存在数据库"没有细节；不考虑上下文长度限制

### Q7：使用AI工具的情况及对Trae的评价 {#q7}

#### 🧠 联想记忆法

**记忆口诀**: "**Codex 写代码，Claude Code 做工程，Trae 做探索**"——三工具各有主战场。

#### 📖 深度解答

**AI 工具使用矩阵**：

| 工具 | 核心场景 | 优势 | 不足 |
|------|---------|------|------|
| **Claude Code** | 主力工具，日常开发、重构、CR、Agent 编排 | 长上下文、Agent 能力、自定义 Skills | 速度有时慢 |

**对 Trae 的评价**：
- 👍 **适合**: 初学者、快速原型、小规模脚本。交互设计比终端更直观

**核心观点**：工具选择取决于任务复杂度，而不是工具本身的好坏。

#### 🗺️ 回答思路

- **答题逻辑框架**: 先说用了什么 → 用表格对比差异 → 评价 Trae 时保持客观

### Q8：是否了解Harness Engineering {#q8}

#### 🧠 联想记忆法

**记忆口诀**: "**驾驭而非依赖**"——AI 是马，人是骑手。工程化的本质是：定义路径、设定边界、做质量检查。

#### 📖 深度解答

**Harness Engineering（驾驭工程）核心思想**：

**不是"让 AI 自由发挥"，而是"给 AI 设定轨道"**。

**五大原则**：

1. **关注点分离**: 答题、审查、归档三个职责独立——每个 Agent 只做一件事
2. **质量内建**: 审查（quality-reviewer）是强制阶段，FAIL 必须重做——质量不是事后补救
3. **自动沉淀**: 每次问答自动归档到题库（question_manager.py），知识不流失
4. **数据驱动**: index.json 驱动网页端展示，结构即产品
5. **确定性编排**: 10 阶段流程硬编码在 SKILL.md 中，MUST/MUST NOT 规则明确——不依赖模型自主决策

**实践案例（Interview Coach）**：
```
输入面试题 → 题目解析 → 并行答题(3 Agent) → 质量审查(15项清单)
→ 重答循环(最多2次) → 文档组装 → 落盘 → 题库归档 → 站点生成 → Git推送
```

**核心价值**：把 AI 的不确定性关进工程的笼子里。让 AI 可以犯错，但错误会被发现、被纠正、被记录。

#### 🗺️ 回答思路

- **答题逻辑框架**: 定义 → 五大原则 → 实践案例 → 核心价值
- **重点得分点**: 重点是"确定性编排"和"质量内建"；给出具体案例而非空谈
- **常见误区**: 把它描述成"用 AI 工具"而不是"驾驭 AI 的方法论"

### Q9：简单介绍Java的垃圾回收机制 {#q9}

#### 🧠 联想记忆法

**记忆口诀**: "**标记-清除-整理-复制，分代回收**"——年轻代用 Copying(快)，老年代用 Mark-Compact(省)，记住"朝生夕死"四个字概括年轻代。

#### 📖 深度解答

**核心概念**：JVM 自动管理堆内存——不需要手动 `free()`，GC 线程在后台回收不可达对象。

**分代回收模型**：

```
新对象 → Eden区（满了触发 Minor GC）
           │
    ┌──────┴──────┐
    │ 存活对象     │ 垃圾对象
    ▼              ▼
Survivor S0/S1   回收
    │ (反复Copying, 熬过15次)
    ▼
老年代（满了触发 Major/Full GC）
```

**三种 GC 算法**：
| 算法 | 适用 | 做法 | 优缺点 |
|------|------|------|--------|
| **标记-清除** | 老年代(基础) | 标记垃圾 → 清除 | 产生内存碎片 |
| **标记-整理** | 老年代(优化) | 标记 → 存活对象向一端移动 | 无碎片，但耗时长 |
| **复制** | 年轻代 | 存活对象复制到另一块Survivor | 快且无碎片，但浪费一半空间 |

**主流垃圾收集器**：
- **Serial/Parallel**: 单线程/多线程，适合小堆
- **CMS**（已废弃）: 并发标记清除，低停顿
- **G1**（JDK 9+ 默认）: 按 Region 分区，优先回收垃圾最多的 Region
- **ZGC/Shenandoah**: 超低延迟（<10ms），大堆（TB级）

**GC 触发时机**：
- Minor GC: Eden 区满
- Full GC: 老年代满、System.gc()、Metaspace 满

#### 🗺️ 回答思路

- **答题逻辑框架**: 分代模型 → 三种算法 → 收集器演进 → 触发时机
- **重点得分点**: "朝生夕死"概括年轻代设计哲学；G1 是现在的主流选择
- **常见误区**: 只说"自动回收"不讲原理；混淆 Minor GC 和 Full GC 的触发条件

### Q10：手撕代码——母牛生小牛问题 {#q10}

#### 🧠 联想记忆法

**记忆口诀**: "**今年 = 去年 + 三年前**"——`f(n) = f(n-1) + f(n-3)`。每头母牛从第 4 年开始每年生一头小牛，所以今年的牛 = 去年的牛 + 三年前出生的牛（它们今年开始生）。

#### 📖 深度解答

**题目**：一头母牛，从第 4 年开始每年生一头小母牛。小母牛也从出生后第 4 年开始每年生一头。N 年后有多少头母牛？

**解法一：DP 递推**
```java
public class CowProblem {
    /**
     * 第 n 年的母牛数量
     * f(n) = f(n-1) + f(n-3)
     * 解释：今年 = 去年活着的 + 三年前出生的（今年开始生育）
     */
    public static long countCows(int n) {
        if (n <= 0) return 0;
        if (n <= 4) return n; // 前4年只有最初的母牛，第4年有1+1=2头
        
        long[] dp = new long[n + 1];
        dp[1] = 1; dp[2] = 2; dp[3] = 3; dp[4] = 4;
        // 注意：第5年开始有小牛出生
        // f(5) = f(4) + f(2) = 4 + 2 = 6
        // 或者更准确的递推：
        // f(n) = f(n-1) + f(n-3)  （n > 4）
        
        for (int i = 5; i <= n; i++) {
            dp[i] = dp[i-1] + dp[i-3];
        }
        return dp[n];
    }
    
    public static void main(String[] args) {
        for (int i = 1; i <= 10; i++) {
            System.out.printf("第%d年: %d头母牛%n", i, countCows(i));
        }
    }
}
```

**解法二：空间优化**
```java
public static long countCowsOptimized(int n) {
    if (n <= 4) return n;
    long a = 1, b = 2, c = 3, d = 4; // 第1,2,3,4年
    for (int i = 5; i <= n; i++) {
        long next = d + b; // f(n) = f(n-1) + f(n-3)
        a = b; b = c; c = d; d = next;
    }
    return d;
}
```

**递推公式推导**：
- 第 n 年的牛 = 第 n-1 年的牛 + 第 n 年新出生的牛
- 第 n 年能生育的牛 = 第 n-3 年及之前出生的牛
- 新出生的牛 = 第 n-3 年的牛（那年的每头牛在今年都生一头）
- 所以 `f(n) = f(n-1) + f(n-3)`

#### 🗺️ 回答思路

- **答题逻辑框架**: 理解题意 → 推导递推公式 → 写 DP 解 → 空间优化
- **重点得分点**: 递推公式推导过程；注意边界条件（n≤4）；空间优化 O(1)
- **常见误区**: 递推公式写成 `f(n-1) + f(n-2)`；不提大数溢出（n 大时用 BigInteger）
- **考点分析**: DP 思维、递推推导、边界处理

### Q11：HashMap底层实现，插入元素时会发生什么 {#q11}

#### 🧠 联想记忆法

**记忆口诀**: "**hash → 桶 → 链表 → 红黑树**"——hash 值定位桶，桶里链表追尾，链表太长变红黑树（>8），扩容时翻倍转移。

#### 📖 深度解答

**核心组成（JDK 8+）**：
- **数组 `Node<K,V>[] table`**: 主干，每个位置叫"桶(bucket)"
- **链表**: 哈希冲突时，新节点追加到链表尾部
- **红黑树**: 链表长度 > 8 且数组长度 ≥ 64 时，链表转为红黑树，查询从 O(n) → O(log n)

**put(key, value) 完整流程**：

```
1. hash = key.hashCode() ^ (key.hashCode() >>> 16)  // 高16位异或低16位，减少碰撞
2. index = (table.length - 1) & hash                // 位运算取模，定位桶
3. 如果 table[index] == null → 直接存入
4. 如果 table[index] 有值：
   a. 先判断 key 是否相等（== 或 equals）→ 相等则替换 value
   b. 如果是 TreeNode → 走红黑树插入
   c. 否则遍历链表：
      - 找到相等 key → 替换 value
      - 遍历到尾节点 → 尾插法插入新节点
      - 判断链表长度 >= 8 → treeifyBin() 转为红黑树
5. size++ 后判断是否 > threshold（capacity * loadFactor）
   → 超过则 resize() 扩容为 2 倍
   → 扩容时重新计算每个节点的桶位置：要么在原位置，要么在原位置 + oldCap
```

**关键设计点**：
- **容量必须是 2 的幂**: 这样 `(length-1) & hash` 等价于 `hash % length`，但位运算快得多
- **负载因子 0.75**: 空间和时间的折中——Poisson 分布下链表 > 8 的概率 < 千万分之一
- **尾插法**: JDK 8 改为尾插法，避免 JDK 7 头插法在并发 resize 时的死循环

#### 🗺️ 回答思路

- **答题逻辑框架**: 数据结构 → put 流程(5步) → 扩容机制 → 为什么容量是2的幂
- **重点得分点**: hash 扰动函数的目的；红黑树转化条件（8 和 64 两个阈值）；扩容时节点位置计算
- **常见误区**: 说"链表>8就转红黑树"而不提数组长度≥64的条件；不知道1.7和1.8的区别

### Q14：事务四大特性、隔离级别、默认级别 {#q14}

#### 🧠 联想记忆法

**ACID 记忆口诀**: "**原一隔久**"——原子性(Atomicity)、一致性(Consistency)、隔离性(Isolation)、持久性(Durability)。

**隔离级别**: "**读未→读已→可重→串行**"，"RU → RC → RR → Serializable"。

#### 📖 深度解答

**事务四大特性(ACID)**：

| 特性 | 含义 | 实现机制 |
|------|------|----------|
| 原子性 A | 事务内的操作要么全做要么全不做 | Undo Log |
| 一致性 C | 事务前后数据满足所有约束条件 | 其他三个特性共同保证 |
| 隔离性 I | 并发事务之间互不干扰 | 锁 + MVCC |
| 持久性 D | 已提交事务的数据永久保存 | Redo Log |

**四种隔离级别**：

| 级别 | 解决的问题 | 存在的问题 | 实现 |
|------|-----------|-----------|------|
| **READ UNCOMMITTED** | — | 脏读、不可重复读、幻读 | 不加锁 |
| **READ COMMITTED** | 脏读 | 不可重复读、幻读 | 每次读生成新 ReadView |
| **REPEATABLE READ (MySQL默认)** | 脏读、不可重复读 | 幻读（部分解决） | 事务开始生成 ReadView + Gap Lock |
| **SERIALIZABLE** | 全部 | 并发性能差 | 读加共享锁 |

**为什么 MySQL 默认 RR 而不是 RC？**
- 历史原因：MySQL 5.0 前 binlog 只有 statement 格式，RC 下可能导致主从不一致
- RR + Gap Lock 可以避免 statement binlog 的幻读导致的主从差异
- Oracle 默认 RC 是因为它用 Redo Log 做物理复制，主从不依赖语句一致性

#### 🗺️ 回答思路

- **答题逻辑框架**: ACID → 四种级别 → 问题对比 → 为什么RR是默认
- **重点得分点**: 脏读/不可重复读/幻读的定义不能混；"MySQL 默认 RR"要解释原因
- **常见误区**: 认为 RR 完全解决了幻读（MVCC 解决了快照读的幻读，当前读靠 Gap Lock）

### Q15：MySQL事务过程中有哪些日志 {#q15}

#### 🧠 联想记忆法

**记忆口诀**: "**undo 回滚，redo 恢复，binlog 同步**"——undo 管回滚、redo 管崩溃恢复、binlog 管主从同步。

#### 📖 深度解答

| 日志 | 作用 | 记录内容 | 存储位置 |
|------|------|----------|----------|
| **Undo Log** | 事务回滚 + MVCC | 修改前的旧数据（逻辑日志） | Undo 表空间 |
| **Redo Log** | 崩溃恢复 | 物理页的修改记录（物理日志） | ib_logfile |
| **Binlog** | 主从复制 + 数据恢复 | SQL 语句或行变更（逻辑日志） | binlog 文件 |
| **Relay Log** | 从库中继 | 从主库接收的 binlog | 从库 relay log |
| **Error Log** | 错误诊断 | 启动/关闭/错误信息 | error.log |
| **Slow Query Log** | 性能诊断 | 超过阈值的 SQL | slow.log |

**关键区别——Redo Log vs Binlog**：

| 维度 | Redo Log | Binlog |
|------|----------|--------|
| 产生者 | InnoDB 引擎层 | MySQL Server 层 |
| 记录方式 | 循环写（固定大小） | 追加写（无限增长） |
| 用途 | 崩溃恢复（crash-safe） | 主从复制、数据恢复 |
| 记录内容 | 物理日志：页的变化 | 逻辑日志：SQL 或 row |

**两阶段提交（2PC）**：
```
1. Prepare: 写 Redo Log (prepare状态)
2. 写 Binlog
3. Commit:  Redo Log (commit状态)
```
任一阶段崩溃都能保证 Redo Log 和 Binlog 一致——这就是"崩溃恢复"不会丢数据的原因。

#### 🗺️ 回答思路

- **答题逻辑框架**: 先讲有哪些日志 → 重点对比 Redo/Binlog → 两阶段提交
- **重点得分点**: Redo Log 是物理日志、Binlog 是逻辑日志；两阶段提交的作用
- **常见误区**: 混淆 Redo 和 Binlog 的用途；不知道两阶段提交

### Q16：索引设计需要满足哪些条件；MVCC执行流程 {#q16}

#### 🧠 联想记忆法

**索引口诀**: "**最左前缀是铁律，高选择先建，覆盖不用回表，联合优于多个单列**"。

**MVCC口诀**: "**ReadView 快照 + Undo Log 回滚链**"——事务开始时拍一张"快照"，通过比对事务 ID 决定能看到哪些版本。

#### 📖 深度解答

**索引设计原则**：

1. **最左前缀原则**: 联合索引 `(a,b,c)`，查询条件必须从最左列开始
2. **高选择性列优先**: 区分度高的列放联合索引最前面
3. **覆盖索引**: SELECT 的列都在索引中时，不需要回表
4. **避免过多的索引**: 索引占用空间且拖慢写入（每个索引都要同步更新）
5. **避免冗余索引**: `(a,b)` 和 `(a)` 中，后者可被前者覆盖（查询只查 a 时也能用前者）
6. **使用 EXPLAIN 验证**: 确保 key 不为 NULL，type 至少 range 以上

**MVCC 执行流程**：

```
1. 事务开始时分配唯一的事务 ID（trx_id）
2. 每行数据有两个隐藏列：
   - DB_TRX_ID: 最近修改此行的事务ID
   - DB_ROLL_PTR: 指向 undo log 的回滚指针

3. 快照读(SELECT)时生成 ReadView：
   - m_ids: 当前活跃事务ID列表
   - min_trx_id: 最小活跃事务ID
   - max_trx_id: 下一个待分配的事务ID
   - creator_trx_id: 当前事务ID

4. 可见性判断：
   - trx_id == creator_trx_id → 可见（自己改的）
   - trx_id < min_trx_id → 可见（已提交）
   - trx_id >= max_trx_id → 不可见（未来事务）
   - min_trx_id <= trx_id < max_trx_id：
     → 在 m_ids 中 → 不可见
     → 不在 m_ids 中 → 可见（已提交）
   - 不可见时，沿 DB_ROLL_PTR 回滚链找上一个版本
```

**RR vs RC 的 MVCC 区别**：
- **RC**: 每次 SELECT 生成新的 ReadView
- **RR**: 事务开始时生成一次 ReadView，整个事务期间复用

#### 🗺️ 回答思路

- **答题逻辑框架**: 索引原则(6条) → MVCC 流程(可见性判断5步) → RR/RC区别
- **重点得分点**: ReadView 的四个关键字段；可见性判断的完整逻辑
- **常见误区**: 混淆快照读和当前读(SELECT ... FOR UPDATE 不走 MVCC)

### Q17：为什么要先写日志，再写入数据库 {#q17}

#### 🧠 联想记忆法

**记忆口诀**: "**日志是顺序写，数据是随机写**"——顺序写比随机写快 100 倍。WAL（Write-Ahead Logging）的核心就是用顺序写换随机写。

#### 📖 深度解答

**为什么采用 WAL（先写日志再写磁盘）？**

**1. 性能——顺序写 vs 随机写**

```
Redo Log：追加写（顺序IO）   → 速度 ~100MB/s
数据页：  随机写（随机IO）    → 速度 ~1MB/s（HDD场景）
```
即使 SSD 时代，顺序写仍比随机写快 5-10 倍。

**2. 崩溃恢复（Crash-Safe）**

```
场景：UPDATE user SET name='b' WHERE id=1
                         │
                         ▼
              写 Redo Log（持久化）
                         │
                         ▼
          ┌──── 系统崩溃！────┐
          │                   │
          ▼                   ▼
     Redo Log 完好      数据页未写入
          │
          ▼
    重启 → 重放 Redo Log → 数据恢复
```

**3. 事务提交的优化——组提交(Group Commit)**

多个事务的 Redo Log 合并为一次 fsync，大幅减少磁盘刷写次数：
```
T1写入redo → ┐
T2写入redo → ├→ 一次fsync → 三个事务同时提交
T3写入redo → ┘
```

#### 🗺️ 回答思路

- **答题逻辑框架**: WAL 定义 → 性能对比（顺序/随机）→ 崩溃恢复场景 → 组提交
- **重点得分点**: "顺序写 vs 随机写"是核心；宕机后通过 Redo Log 重放恢复
- **常见误区**: 不知道组提交的优化价值

### Q18：联合索引(a,b,c)，where c=2 and a=1，是否走索引 {#q18}

#### 🧠 联想记忆法

**记忆口诀**: "**有大哥才能用索引**"——联合索引好比楼梯，要上楼必须先踩第一级台阶。a 是大楼，b 是二楼，c 是三楼——你跳不过一楼直接上三楼。

#### 📖 深度解答

**答案：会走索引，但只用到了 a 列。**

原因——**最左前缀原则**：
- MySQL 优化器会自动调整 WHERE 条件顺序
- `WHERE c=2 AND a=1` → 优化器重排为 `WHERE a=1 AND c=2`
- 索引 `(a,b,c)` 中，先按 a 定位，然后用 c 过滤——但 b 断开后，c 列不走索引

```
EXPLAIN 结果：
key: idx_abc
key_len: a的长度         ← 只用了 a 列！
Extra: Using index condition  ← ICP，先用 a 过滤，再用 c 条件筛选
```

**联合索引匹配规则**：

| WHERE 条件 | 使用索引列 | 说明 |
|-----------|-----------|------|
| a=1 AND b=2 AND c=3 | a,b,c | 全匹配 |
| a=1 AND b=2 | a,b | 匹配前两列 |
| a=1 AND c=3 | a | b断开，c不走索引 |
| b=2 AND c=3 | 无 | 缺 a，全表扫描 |
| a=1 AND b>2 AND c=3 | a,b | 范围查询后 c 不走索引 |
| c=3 AND a=1 | a | 优化器重排后同上 |

#### 🗺️ 回答思路

- **答题逻辑框架**: 结论先行（走索引但只用a）→ 解释最左前缀 → checklist 表格
- **重点得分点**: 优化器会重排条件顺序；ICP(Index Condition Pushdown) 的作用
- **常见误区**: 认为"c 在中间断了就完全不走"（实际是走了 a 的索引 + ICP 过滤）

### Q19：MySQL索引实现；B+树与其他树的区别 {#q19}

#### 🧠 联想记忆法

**记忆口诀**: "**B+ 树——矮胖、叶子串、数据只在叶子上**"。"矮胖"减少 IO 次数，"叶子串"支持范围查询。

#### 📖 深度解答

**B+树结构特点**：

```
                  [10 | 20 | 30]          ← 非叶子节点（只存key，不存data）
                 /    |     |    \
          [5|8]   [12|15]  [22|25] [32|35]  ← 非叶子节点
          / | \    / | \    / | \    / | \
        [1][5][8] [12][15] [22][25] [32][35] ← 叶子节点（存key+data）
          ↕    ↕     ↕    ↕     ↕    ↕
        双向链表连接所有叶子节点 ← 范围查询利器
```

**B+树关键特性**：
1. **数据全在叶子节点**: 非叶子只做索引（省空间 → 更多索引 → 树更矮）
2. **叶子节点用双向链表连接**: 范围查询只需找到起点，顺着链表读即可
3. **高扇出（fan-out）**: 一个节点能存 ~1000 个 key，3 层 B+树能存 ~10 亿条记录

**B+树 vs 其他树**：

| 对比 | B+树 | B树 | 二叉树/AVL | 红黑树 |
|------|------|-----|-----------|--------|
| 数据存储 | 只在叶子节点 | 所有节点 | 所有节点 | 所有节点 |
| 树的高度 | 最矮（高扇出） | 矮 | 高 | 较高 |
| 范围查询 | O(logN + 范围大小) | O(N×logN) | O(N×logN) | O(N×logN) |
| 磁盘IO | 最少 | 少 | 多 | 多 |
| 为什么用B+ | 磁盘IO最优 + 范围查询友好 | — | 每个节点只存一个key,树太高 | 红黑树是内存结构 |

**核心问题：为什么 B+树而非 B 树？**
1. B+树非叶子不存数据 → 一个节点存更多 Key → 树更矮 → 磁盘 IO 更少
2. B+树叶子链表 → 范围查询 O(logN+M)，B 树需要中序遍历 O(N)

#### 🗺️ 回答思路

- **答题逻辑框架**: B+树结构图 → 三个核心特性 → 对比表格 → 磁盘IO解释
- **重点得分点**: "叶子节点链表"是 B+树独有优势；磁盘 IO 次数 = 树的高度
- **常见误区**: 不知道 B+树和 B 树的本质区别（数据在哪）

### Q21：Redis分布式锁如何实现 {#q21}

#### 🧠 联想记忆法

**记忆口诀**: "**setnx 加锁，lua 解锁，看门狗续期**"——`SET key value NX EX 30` 一行命令搞定。

#### 📖 深度解答

**基础版——SETNX + EXPIRE**：
```java
// ❌ 错误：两步非原子，setnx 后宕机 → 死锁
redis.setnx("lock:order:1001", "uuid");
redis.expire("lock:order:1001", 30);

// ✅ 正确：原子命令
redis.set("lock:order:1001", "uuid", "NX", "EX", 30);
```

**解锁——Lua脚本保证原子性**：
```lua
-- 必须校验 value（UUID），防止删了别人的锁
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

**关键要素**：
1. **互斥性**: `NX` 保证只有一个客户端能获取
2. **防死锁**: `EX` 设置过期时间
3. **解锁安全**: value 用 UUID 标识持有者，Lua 脚本原子判断+删除
4. **锁续期**: 业务没执行完 → 后台线程自动续期（看门狗）

#### 🗺️ 回答思路

- **答题逻辑框架**: 为什么不用 setnx+expire 两步 → 原子命令 → Lua解锁 → 续期
- **重点得分点**: "NX EX 一行命令"体现对原子性理解；UUID 防误删
- **常见误区**: 分两步执行 setnx 和 expire；不解锁就直接等过期

### Q22：setnx和setx区别；setnx如何释放锁 {#q22}

#### 🧠 联想记忆法

**记忆口诀**: "**SETNX 只加不设期，SET...NX 一步到位**"。

#### 📖 深度解答

**SETNX vs SETEX vs SET ... NX EX**：

| 命令 | 作用 | 原子性 |
|------|------|--------|
| `SETNX key value` | key 不存在时设值，返回 1/0 | 单步原子 |
| `SETEX key seconds value` | 设值并设置过期时间 | 单步原子 |
| `SETNX + EXPIRE` | 先设值，再设过期 | ❌ 两步非原子 |
| `SET key value NX EX 30` | 不存在则设值，同时设 30s 过期 | ✅ 一步原子 |

**SETNX 释放锁的正确方式**：
```lua
-- Lua 脚本（原子执行）
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

**为什么不能直接 DEL？**
```
线程A: 获取锁 → 业务执行超时 → 锁自动过期
线程B: 获取锁 → 正在执行业务
线程A: 业务完成 → DEL 锁 ← ❌ 删了 B 的锁！
```

**为什么 Lua 脚本？**
Redis 是单线程执行命令，Lua 脚本在执行期间不会被其他命令打断 → GET + DEL 是原子的。

#### 🗺️ 回答思路

- **答题逻辑框架**: 三个命令对比 → 锁释放的坑 → Lua 脚本方案
- **重点得分点**: "不能直接 DEL"是核心考点；Lua 的原子性原理
- **常见误区**: 不知道 SETNX 已废弃（Redis 官方推荐 SET ... NX）

### Q23：Redisson锁实现；看门狗机制原理 {#q23}

#### 🧠 联想记忆法

**记忆口诀**: "**看门狗——锁没完，狗叫唤，自动续期 30 秒**"。每隔 10 秒将锁续期到 30 秒，直到业务完成显式解锁。

#### 📖 深度解答

**Redisson 分布式锁核心流程**：

```
RLock lock = redisson.getLock("order:1001");
lock.lock();  // 默认 30s 过期
try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```

**看门狗（Watchdog）机制**：

```
获取锁（默认 leaseTime = 30s）
    │
    ▼
启动定时任务（每 10s 执行一次）  ← 看门狗
    │
    ▼
检查：锁还是我持有的吗？
    ├─ 是 → resetExpire(30s)  ← 续期
    └─ 否 → 停止续期
    │
    ▼
unlock() 被调用 → 取消定时任务 + 删除锁
```

**关键参数**：
- `lockWatchdogTimeout`: 30s（默认），internalLockLeaseTime
- 续期间隔：`internalLockLeaseTime / 3 = 10s`

**如果指定了 leaseTime**：
```java
lock.lock(10, TimeUnit.SECONDS); // 手动指定过期时间 → 看门狗不启动！
```
只有不指定 leaseTime 时才启动看门狗——这很容易踩坑。

**Redisson vs 手写方案**：
| | Redisson | 手写 SET NX |
|------|----------|-------------|
| 自动续期 | ✅ 看门狗 | ❌ 需自己实现 |
| 可重入 | ✅ hash + 计数器 | ❌ 需自己实现 |
| 红锁(RedLock) | ✅ 内置 | ❌ |
| 代码量 | 3行 | 20+行 |

#### 🗺️ 回答思路

- **答题逻辑框架**: 基本用法 → 看门狗流程图 → leaseTime 区别 → 对比手写
- **重点得分点**: "不设 leaseTime 才开启看门狗"是最大考点；可重入用 hash + counter 实现
- **常见误区**: 以为指定了 leaseTime 后看门狗还会工作

### Q24：使用Lua脚本实现锁和不使用的区别 {#q24}

#### 🧠 联想记忆法

**记忆口诀**: "**不用 Lua 是两步，用了 Lua 是一步**"。两步之间就是并发窗口——任何插入两步之间的操作都可能破坏原子性。

#### 📖 深度解答

**不用 Lua——解锁的并发问题**：

```java
// 线程A执行解锁
String value = redis.get("lock");  // 步骤1: 读取
if (myValue.equals(value)) {
    redis.del("lock");             // 步骤2: 删除
}

T1: A GET lock → "A-uuid"（判断：是我的锁）
T2: 锁过期，自动删除

T4: A DEL lock → ❌ 删除了 B 的锁！

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

Redis 单线程执行 Lua 脚本 → GET + 判断 + DEL 一气呵成，中间不会被任何命令打断。

**其他需要 Lua 的场景**：
- 限流（Token Bucket）
- 库存扣减（判断库存 > 0 再减）
- 原子性更新多个 key

#### 🗺️ 回答思路

- **答题逻辑框架**: 不用 Lua 的问题（时序图）→ Lua 方案 → Redis 单线程原理
- **重点得分点**: 画时间线展示竞态窗口；"单线程执行 Lua"是核心
- **常见误区**: 只说"更安全"但没有说明"单线程"的机制

### Q25：Redis主从切换、异常导致锁丢失，补救措施 {#q25}

#### 🧠 联想记忆法

**记忆口诀**: "**红锁多节点，过半才算赢**"。RedLock——向 N 个独立 Redis 节点（N 为奇数）同时申请锁，获得 > N/2 个节点的锁才算成功。

#### 📖 深度解答

**问题场景**：
```
客户端A → Master → 获取锁成功
Master 宕机（锁还未同步到 Slave）
Sentinel 选举 Slave 为新 Master
客户端B → 新Master → 获取锁成功  ← ❌ A和B同时持有锁！

```
客户端同时向 5 个独立 Redis 节点申请锁
    │
    ▼
获得 ≥ 3 个节点(N/2+1)的锁 → 锁获取成功
    │
    ▼
只要大多数节点没丢锁 → 整体锁安全

缺点：部署成本高（至少 3 个独立节点）；性能开销大

- 主从使用半同步复制，锁操作必须同步到至少一个从库才返回成功
- 业务层加乐观锁版本号兜底

**方案三：ZooKeeper（CP 系统）**
- ZK 用 ZAB 协议保证一致性，主节点挂后新 Leader 一定包含所有已提交数据
- 代价：性能不如 Redis，更适合对一致性要求极高的场景

**方案四：业务层兜底——乐观锁**
```sql
UPDATE order SET status='paid', version=version+1
WHERE id=1001 AND version=5;  -- 版本号不匹配 → 更新失败 → 业务感知
```

#### 🗺️ 回答思路

- **答题逻辑框架**: 场景描述 → RedLock → 其他方案 → 各方案适用场景
- **重点得分点**: "主从异步复制"是锁丢失的根因；RedLock 不是银弹
- **常见误区**: 认为 RedLock 是唯一的解决方式

### Q26：缓存击穿、穿透、雪崩了解吗 {#q26}

#### 🧠 联想记忆法

**记忆口诀**: "**穿——不存在，击——热点死，崩——批量挂**"。

- **穿透**: 查不存在的数据 → 穿过了缓存，打在 DB 上
- **击穿**: 一条热点 key 过期 → 大量请求"击穿"打 DB
- **雪崩**: 大量 key 同时过期 → DB "雪崩"

#### 📖 深度解答

| 问题 | 场景 | 方案 |
|------|------|------|
| **穿透** | 查 `userId=-1`，缓存和 DB 都没有 | ① 布隆过滤器（不存在的直接拦截）② 缓存空值（NULL，TTL 设短） |
| **击穿** | 秒杀商品缓存过期，一瞬间百万请求打 DB | ① 互斥锁（一个线程查 DB 回写，其他等待）② 逻辑过期（永不过期 + 异步刷新）③ 热点永不过期 |
| **雪崩** | 同一批 key 同时过期 + 同时大量请求 | ① TTL + 随机偏移值（`TTL=300+random(0,60)`）② 多级缓存③ 限流熔断 |

**代码示例——互斥锁防击穿**：
```java
public Order getOrder(Long id) {
    Order order = redis.get("order:" + id);
    if (order != null) return order;
    
    // 缓存未命中，加锁查DB
    String lockKey = "lock:order:" + id;
    if (redis.set(lockKey, "1", "NX", "EX", 10)) {
        try {
            // 双重检查
            order = redis.get("order:" + id);
            if (order != null) return order;
            
            order = db.findById(id);
            redis.set("order:" + id, order, 300 + ThreadLocalRandom.current().nextInt(60));
        } finally {
            redis.del(lockKey);
        }
    } else {
        Thread.sleep(50);  // 等待持锁线程回写缓存
        return getOrder(id); // 递归重试
    }
    return order;
}
```

#### 🗺️ 回答思路

- **答题逻辑框架**: 三个概念对比表格 → 每个的解决方案 → 给代码示例
- **重点得分点**: 三者的"根因"不同（不存在 vs 过期 vs 批量过期）；互斥锁要双重检查
- **常见误区**: 混淆击穿和雪崩（一个 key vs 多个 key）

### Q27：Redis持久化机制了解吗 {#q27}

#### 🧠 联想记忆法

**记忆口诀**: "**RDB 是快照，AOF 是日志，混血儿最实用**"。

#### 📖 深度解答

| 机制 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **RDB** | 定时全量快照（`save 900 1`） | 文件小，恢复快 | 可能丢最后几分钟数据 |
| **AOF** | 追加写命令日志 | 数据安全（最多丢1s） | 文件大，恢复慢 |
| **混合持久化** | RDB + AOF 增量 | 恢复快 + 数据安全 | Redis 4.0+ 支持 |

**RDB 流程**：
```
主进程 fork() 子进程
    │
    ▼
子进程：遍历所有数据 → 写入临时RDB文件 → 原子rename替换旧文件
主进程：继续处理请求
```
关键：fork 时 COW(Copy-On-Write)，父子进程共享内存页，只复制被修改的页。

**AOF 刷盘策略**：

| appendfsync | 行为 | 安全性 |
|-------------|------|--------|
| **always** | 每条命令 fsync | 最高，性能最差 |
| **everysec**（默认） | 每秒 fsync 一次 | 丢最多 1s 数据 |
| **no** | 交给 OS | 不可控 |

**AOF 重写（Rewrite）**：
AOF 文件无限增长 → 触发 BGREWRITEAOF → 将多条命令合并为最小集（如 6 次 incr → 1 次 set）。

**混合持久化（Redis 4.0+，推荐）**：
```
RDB快照(某个时间点) + AOF增量(快照之后的命令)
```
兼具 RDB 的快速恢复和 AOF 的数据安全。

#### 🗺️ 回答思路

- **答题逻辑框架**: RDB vs AOF 对比 → 混合持久化 → 生产建议
- **重点得分点**: fork + COW 是 RDB 核心机制；AOF everysec 是默认；混合持久化是推荐
- **常见误区**: 生产环境只开 RDB 不开 AOF（丢数据风险大）

### Q28：Redis集群、哨兵的作用 {#q28}

#### 🧠 联想记忆法

**记忆口诀**: "**哨兵——看门的，集群——分家的**"。哨兵管主从切换（高可用），集群管数据分片（水平扩展）。

#### 📖 深度解答

| 维度 | 哨兵(Sentinel) | 集群(Cluster) |
|------|---------------|---------------|
| **解决问题** | 高可用（HA） | 水平扩展 + 高可用 |
| **核心功能** | 主节点故障自动切换 | 数据分片（16384槽） + 故障转移 |
| **数据分布** | 所有节点存全量数据 | 每个节点存部分数据（1/N） |
| **节点数** | 至少 3 个哨兵（奇数） | 至少 3 主 3 从 |
| **客户端** | 连哨兵获取主节点地址 | 连任意节点，自动重定向 |

**哨兵工作原理**：
```
Sentinel节点（≥3个）
    │
    ├─ 监控：每1秒向Master/Slave发PING
    ├─ 通知：Master主观下线(SDOWN) → 多个Sentinel确认 → 客观下线(ODOWN)
    ├─ 选举：Raft算法选出一个Sentinel Leader
    └─ 切换：Leader将Slave提升为新Master，通知客户端
```

**集群——数据分布**：
```
16384 个哈希槽
    │
    ├─ 节点1：槽 0-5460
    ├─ 节点2：槽 5461-10922
    └─ 节点3：槽 10923-16383

key → CRC16(key) % 16384 → 定位到槽 → 定位到节点
```

**选型建议**：
- 数据量 < 单机内存（< 32GB）→ 哨兵就够了
- 数据量 > 单机内存 → 必须用集群

#### 🗺️ 回答思路

- **答题逻辑框架**: 哨兵 vs 集群对比表 → 各自原理 → 选型建议
- **重点得分点**: 哨兵是"高可用"不是"扩展"；集群用 16384 个槽分配数据
- **常见误区**: 认为哨兵能水平扩展（它不能，数据还是全量的）

### Q29：zset底层跳表原理 {#q29}

#### 🧠 联想记忆法

**记忆口诀**: "**跳表——给链表修高速公路**"。每层跳跃一半节点，查询从最高层开始，每层"跳过一半"最终 O(log N)。

#### 📖 深度解答

**跳表结构**：

```
Level 2:  [1] ────────────── [20] ────────────── [45]
Level 1:  [1] ──── [10] ──── [20] ──── [35] ──── [45]
Level 0:  [1]→[5]→[10]→[15]→[20]→[25]→[35]→[40]→[45]
          （底层完整有序链表）
```

**查询流程（查 35）**：
```
从最高层(Level 2)开始：
  Level 2: 1 → 20 → 45(>35,退回20)
  Level 1: 20 → 35(找到了！)
```
不需要遍历所有节点就能定位——这就是 O(log N) 的来源。

**zset 为什么用跳表而不是红黑树？**
1. **范围查询更简单**: 跳表找到起点后顺着最底层链表走即可，红黑树需要中序遍历
2. **实现更简单**: zset 同时需要按 score 排序和按 member 查找，跳表+哈希表组合更自然
3. **并发友好**: 跳表插入只需改局部指针，红黑树需要全局 rebalance

**zset 内部结构**：
- `ziplist`（小数据时）：元素 < 128 且成员 < 64 字节
- `skiplist + dict`（大数据时）：跳表按 score 排序 + 哈希表 O(1) 按 member 查找

#### 🗺️ 回答思路

- **答题逻辑框架**: 结构图 → 查询流程 → 与红黑树对比 → Redis 内部实现
- **重点得分点**: "多层索引"+"概率性层高"；范围查询优势是选跳表的核心理由
- **常见误区**: 不知道 zset 底层是跳表+哈希表双重结构

### Q30：使用Kafka时如何保证数据不丢失 {#q30}

#### 🧠 联想记忆法

**记忆口诀**: "**生产等确认，Broker多副本，消费手动交**"——producer acks=all → broker min.insync.replicas ≥ 2 → consumer 手动提交 offset。

#### 📖 深度解答

**三条防线——从生产到消费全链路**：

**1. Producer 端——等确认再发下一条**
```java
// 关键配置
acks=all                   // 所有 ISR 副本都写入才返回成功
retries=3                  // 失败重试
enable.idempotence=true    // 幂等性，配合 acks=all 保证 exactly once
max.in.flight.requests.per.connection=1  // 保证消息顺序
```

**2. Broker 端——多副本 + 最少同步副本**
```
replication.factor=3        // 3个副本
min.insync.replicas=2       // 至少2个 ISR 副本写入成功
unclean.leader.election.enable=false  // 不允许非ISR副本当选Leader
```

// ❌ 错误：自动提交 —— 消息可能还没处理就提交了
enable.auto.commit=true

// ✅ 正确：手动提交 —— 处理完再提交
consumer.poll();
processRecords(records);
consumer.commitSync();  // 同步提交，确保不丢
```

#### 🗺️ 回答思路

- **答题逻辑框架**: 三层防线（生产→Broker→消费）→ 每层关键配置 → 为什么这样配
- **重点得分点**: `acks=all` + `min.insync.replicas` 缺一不可；手动提交 offset
- **常见误区**: 只说 acks=all 不提 min.insync.replicas（Leader 挂了新 Leader 可能没这条消息）

### Q31：如何保证信息不重复消费 {#q31}

#### 🧠 联想记忆法

**记忆口诀**: "**消息ID加去重表，消费前查、消费后记**"。`SELECT + INSERT` 两步合一——要么用唯一索引，要么用 Redis set。

#### 📖 深度解答

**核心思路：消费端幂等设计**。

**方案一：数据库唯一索引（最可靠）**
```sql
CREATE TABLE consumed_messages (
    msg_id VARCHAR(64) PRIMARY KEY,  -- 消息唯一ID
    consumer_group VARCHAR(64),
    consumed_at DATETIME
);

-- 消费流程
INSERT INTO consumed_messages VALUES (msgId, groupId, NOW());
-- 如果 msgId 重复 → 唯一索引冲突 → 跳过
```

**方案二：Redis 去重（高性能）**
```java
// 消息ID作为 key，setnx 天然幂等
Boolean isNew = redis.set("msg:consumed:" + msgId, "1", "NX", "EX", 3600);
if (!isNew) return; // 已消费，跳过
processMessage(msg);
```

**方案三：业务幂等（最自然）**
```sql
-- 支付回调：通过订单号保证幂等
UPDATE order SET status='paid', pay_time=NOW()
WHERE order_id='ORD001' AND status='unpaid';
-- 第二次执行：status已经不是 unpaid → affected rows = 0 → 跳过
```

**Kafka 幂等 + 事务（Exactly Once）**：
- producer: `enable.idempotence=true` + `transactional.id`
- consumer: `isolation.level=read_committed`
- 一次事务内，生产者写入 + 消费者提交偏移量是原子的

#### 🗺️ 回答思路

- **答题逻辑框架**: 三种去重方案 → 适用场景对比 → Kafka Exactly Once
- **重点得分点**: 唯一索引是最可靠的方案；业务幂等比通用去重更优雅
- **常见误区**: 去重表不做过期清理（数据无限增长）

### Q33：订单更新场景，同一笔订单变更分布在多个分区中，如何解决并发冲突 {#q33}

#### 🧠 联想记忆法

**记忆口诀**: "**按订单ID分区——同一订单，同一分区，顺序消费**"。

#### 📖 深度解答

**问题根因**：
Kafka 只保证单个 Partition 内有序。如果同一笔订单的消息被散列到不同分区，可能出现"先取消后支付"的乱序。

**方案一：按订单 ID 分区（根本解决）**
```java
// 生产者指定分区键
ProducerRecord<String, String> record = new ProducerRecord<>(
    "order-topic",
    orderId,   // ← key：同一订单ID进入同一分区
    message
);
// Kafka 默认用 key 的 hash 选择分区 → 同一 orderId → 同一 partition → 顺序消费
```

**方案二：消费端做版本号 + 乐观锁（兜底）**
```sql
-- 消息带有版本号
UPDATE order SET status='paid', version=version+1, amount=100
WHERE order_id='ORD001' AND version=5;
-- 乱序消息到达时 version 不匹配 → affected rows = 0 → 丢弃/死信队列
```

**方案三：消费端做状态机校验**
```java
// 定义合法的状态转移
Map<OrderStatus, Set<OrderStatus>> allowedTransitions = Map.of(
    UNPAID, Set.of(PAID, CANCELLED),
    PAID,   Set.of(SHIPPED, REFUNDING),
    SHIPPED, Set.of(COMPLETED)
);

if (!allowedTransitions.get(currentStatus).contains(newStatus)) {
    // 非法状态转移 → 丢弃或死信
    log.warn("Invalid transition: {} -> {}", currentStatus, newStatus);
    return;
}
```

#### 🗺️ 回答思路

- **答题逻辑框架**: 根因分析 → 分区键方案（治本）→ 消费端方案（兜底）
- **重点得分点**: "同一 key → 同一 partition"是 Kafka 的基础保证；消费端版本号是最后防线
- **常见误区**: 只依赖消费端兜底而不从源头解决

### Q36：HTTP、TCP、UDP相关知识点 {#q36}

#### 🧠 联想记忆法

**记忆口诀**: "**TCP——打电话（先接通，不漏话）；UDP——发短信（直接发，可能丢）**"。

#### 📖 深度解答

| 特性 | TCP | UDP |
|------|-----|-----|
| **连接** | 面向连接（三次握手） | 无连接 |
| **可靠性** | 可靠（确认重传） | 不可靠（尽力而为） |
| **顺序** | 有序 | 无序 |
| **速度** | 慢 | 快 |
| **头部开销** | 20 字节 | 8 字节 |
| **适用场景** | HTTP、文件传输、邮件 | 视频直播、DNS、游戏 |

**TCP 三次握手**：
```
Client                              Server
  │─────── SYN, seq=x ──────────────▶│
  │◀── SYN+ACK, seq=y, ack=x+1 ─────│
  │─────── ACK, ack=y+1 ────────────▶│   连接建立
```

**TCP 四次挥手**：
```
Client                              Server
  │─────── FIN ──────────────────────▶│
  │◀────── ACK ──────────────────────│  (半关闭: Client不再发, Server可能还在发)
  │◀────── FIN ──────────────────────│
  │─────── ACK ──────────────────────▶│   连接关闭
```

**HTTP 各版本**：
- **1.0**: 每次请求新建 TCP 连接
- **1.1**: 持久连接(Keep-Alive) + 管道化
- **2.0**: 多路复用（一个连接并发多个请求）、头部压缩、服务器推送
- **3.0**: 基于 QUIC(UDP)，0-RTT 建连，解决 TCP 队头阻塞

#### 🗺️ 回答思路

- **答题逻辑框架**: TCP vs UDP 对比 → 握手挥手 → HTTP 版本演进
- **重点得分点**: 三次握手为什么是三次（不是两次也不是四次）——双方都确认自己收发正常
- **常见误区**: 混淆 HTTP 2.0 多路复用和 HTTP 1.1 管道化

### Q37：HTTP和HTTPS的区别 {#q37}

#### 🧠 联想记忆法

**记忆口诀**: "**HTTPS = HTTP + TLS**"。多了 TLS 加密层：先握手交换密钥，再加密传输数据。

#### 📖 深度解答

| 维度 | HTTP | HTTPS |
|------|------|-------|
| **安全性** | 明文传输 ❌ | TLS 加密 ✅ |
| **端口** | 80 | 443 |
| **证书** | 不需要 | 需要 CA 证书 |
| **速度** | 快 | 慢（多 TLS 握手） |
| **SEO** | 无优势 | Google 优先收录 |

**HTTPS = HTTP over TLS**：

```
应用层:    HTTP
           │
安全层:    TLS (加密 + 身份认证 + 完整性校验)  ← 新增
           │
传输层:    TCP
```

**TLS 握手（简化版）**：
```
Client                                    Server
  │── ClientHello(支持的加密套件) ──────────▶│
  │◀─ ServerHello(选定套件) + 证书 ──────────│
  │── 生成pre-master key, 用公钥加密 ───────▶│
  │                                         │
  │ 双方用 pre-master key + 随机数           │
  │ → 生成对称密钥(session key)              │
  │                                         │
  │◀══════ 后续通信用对称密钥加密 ═══════════▶│
```

**为什么用混合加密？**
- 非对称加密（RSA/ECDHE）用于密钥交换（安全但不快）
- 对称加密（AES）用于数据传输（快速）
- 各取所长

#### 🗺️ 回答思路

- **答题逻辑框架**: 对比表 → TLS 层的作用 → 握手流程
- **重点得分点**: "混合加密"——非对称交换密钥，对称传输数据；TLS 提供三大保证（加密/认证/完整性）
- **常见误区**: 认为 HTTPS 是独立的协议（它只是 HTTP + TLS）

### Q38：浏览器输入URL后完整执行过程 {#q38}

9. 加载子资源

```

### Q40：线程、进程、协程相关知识点 {#q40}

#### 🧠 联想记忆法

**记忆口诀**: "**进程是房子，线程是房间里一起干活的人，协程是一个人同时做多件事**"。

#### 📖 深度解答

| 维度 | 进程 | 线程 | 协程 |
|------|------|------|------|
| **资源** | 独立内存空间 | 共享进程内存 | 共享线程内存 |
| **切换开销** | 大（切换页表） | 中（内核态切换） | 小（用户态切换） |
| **通信** | IPC（管道/共享内存） | 共享变量（需加锁） | 共享变量（无需锁） |
| **创建开销** | 大 | 中 | 极小 |
| **调度** | 操作系统 | 操作系统 | 用户程序（协作式） |
| **适用场景** | 独立服务 | CPU密集型 | IO密集型 |

**协程的核心优势——为什么 IO 密集型用协程？**
```
线程模型（处理 10000 个并发请求）：
  需要 10000 个线程 → 每个线程 1MB 栈 → 10GB 内存 + 大量上下文切换

协程模型：
  1 个线程 + 10000 个协程 → 协程在用户态切换，内存开销几KB/协程
  遇到 IO 时让出执行权，IO 完成后回来继续
```

**Java 中的协程——Virtual Thread（Java 21+）**：
```java
// 平台线程（传统）
Thread.ofPlatform().start(() -> doSomething());

// 虚拟线程（协程）
Thread.ofVirtual().start(() -> doSomething());  
// 遇到阻塞IO自动"挂起"，不占用OS线程
```

#### 🗺️ 回答思路

- **答题逻辑框架**: 三者对比 → 协程核心优势 → Java Virtual Thread
- **重点得分点**: 进程隔离、线程共享、协程轻量；协程是用户态调度
- **常见误区**: 认为协程一定比线程快（CPU 密集型场景优势不明显）

### Q41：Linux转换权限的指令 {#q41}

#### 🧠 联想记忆法

**记忆口诀**: "**r=4, w=2, x=1**"——读 4、写 2、执行 1，`chmod 755` 就是 `rwxr-xr-x`。

#### 📖 深度解答

**chmod——修改文件权限**：

```
权限数字：     r(read)=4   w(write)=2   x(execute)=1

chmod 755 file.sh  →  rwxr-xr-x
                    │  ├─┬─┘ └─┬─┘ └─┬─┘
                    │  │  │     │      └─ others: r-x (5)
                    │  │  │     └──────── group:  r-x (5)
                    │  │  └────────────── owner:  rwx (7)
                    │  └────────────────  suid/sgid/sticky
                    └─────────────────── 文件类型(- 普通, d 目录, l 链接)
```

**符号模式**：
```bash
chmod u+x script.sh        # 给 owner 加执行权限
chmod g-w file.txt          # 去掉 group 的写权限
chmod o=r file.txt          # 设 others 为只读
chmod a+x script.sh         # 所有用户加执行权限
```

**chown/chgrp——修改所有者/组**：
```bash
chown user:group file.txt          # 同时改所有者和组
chown -R user:group /opt/app/      # 递归修改目录下所有文件
chgrp devops file.txt              # 只改组
```

**常见权限组合**：
| 数字 | 权限 | 适用场景 |
|------|------|----------|
| 755 | rwxr-xr-x | 目录、可执行脚本 |
| 644 | rw-r--r-- | 普通文件 |
| 600 | rw------- | 私钥、配置文件含密码 |
| 777 | rwxrwxrwx | ❌ 不推荐！安全隐患 |

#### 🗺️ 回答思路

- **答题逻辑框架**: 数字权限(r4/w2/x1) → chmod 用法 → chown/chgrp → 常见权限组合
- **重点得分点**: 4/2/1 的数字计算；755 和 644 是最常用的两个权限
- **常见误区**: 生产环境用 777（正确的权限原则是最小权限）
