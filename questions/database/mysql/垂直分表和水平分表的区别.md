# 垂直分表和水平分表的区别

> Difference between Vertical Partitioning and Horizontal Partitioning

---

## 🧠 联想记忆法

### 记忆口诀/联想

**"切蛋糕：竖切分配料，横切分人数"**

一套完整的记忆框架包含三个层次：

| 层次 | 口诀 | 对应关系 |
|------|------|---------|
| 核心口诀 | 竖切分列，横切分行 | 垂直=按列拆分，水平=按行拆分 |
| 场景口诀 | 宽表竖切，大表横切 | 垂直解决"表太宽"，水平解决"表太大" |
| 问题口诀 | 竖切怕JOIN，横切怕扩容 | 垂直分表的核心代价是JOIN查询，水平分表的核心代价是扩容重分布 |

### 记忆原理

这个口诀利用**空间类比法**建立强记忆锚点：

- **物理空间映射**："竖切"对应数据库的"列"（Column）方向，将一张宽表的多个列分散到不同表中；"横切"对应数据库的"行"（Row）方向，将大量行均匀分布到多个结构相同的表中。人的空间直觉对"竖/横"有天然的方向辨别力，无需额外记忆。
- **分层递进编码**：第一层解决"是什么"（按列 vs 按行），第二层解决"为什么"（宽表vs大表的两类问题），第三层解决"有什么代价"（JOIN vs 扩容）。三层递进形成完整的知识链：**定义→动机→代价**，面试中自然展开不会遗漏。
- **反义对偶结构**："竖切分列，横切分行"采用四字对偶结构，类似"左顾右盼"、"上窜下跳"的成语节奏，朗朗上口且形成对比强化。

### 关联知识

- **文件系统类比**：垂直分表类似将一个大文件按字段拆分为多个小文件（如JSON中拆出独立的`user_base`和`user_ext`）；水平分表类似将一个大文件按行拆分为多个分片文件（如日志文件按日期拆分）。前者解决单行数据过宽问题，后者解决单文件行数过多问题。
- **操作系统类比**：垂直分表 ≈ 页面分段（Segmentation）——将进程地址空间按逻辑功能拆分为代码段、数据段、堆栈段；水平分表 ≈ 分页（Paging）——将内存等分为固定大小的页框。这一类比在学习数据库时帮助理解两类拆分的思想来源。
- **Java类设计类比**：垂直分表 ≈ 类的拆分原则（单一职责：一个类不应包含过多字段，将大字段拆到独立类）；水平分表 ≈ 线程池任务分片（将大量任务平均分配给多个工作线程并行处理）。

## 📖 深度解答

### 1. 核心概念

**垂直分表（Vertical Partitioning）与水平分表（Horizontal Partitioning/Sharding）** 是数据库拆分（Database Splitting）的两种基本策略，分别从列维度和行维度解决单表性能瓶颈问题。

#### 1.1 垂直分表：按列拆分

垂直分表（Vertical Partitioning）是指将一张列数过多的表，按列拆分为多张结构不同的子表。每张子表包含原表的部分列，通过主键（Primary Key）保持关联。

**形式化定义**：给定原表 `R(A1, A2, ..., An)`，垂直分表将其拆分为 `R1(A1, A2, ..., Ak)` 和 `R2(A1, Ak+1, ..., An)`，其中 `A1` 为主键，两张子表共享主键。

**典型示例**：电商用户表

```sql
-- 拆分前：一张宽表，包含所有用户信息
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    username VARCHAR(32),
    password VARCHAR(64),
    email VARCHAR(64),
    phone VARCHAR(16),
    avatar_url VARCHAR(256),
    bio TEXT,                    -- 个人简介，访问频率低
    last_login_ip VARCHAR(16),
    register_time DATETIME,
    register_ip VARCHAR(16),
    vip_level TINYINT,
    points INT,
    preferences JSON             -- 偏好设置，大字段
);

-- 拆分后：用户基本信息表（高频访问字段）
CREATE TABLE user_base (
    id BIGINT PRIMARY KEY,
    username VARCHAR(32),
    password VARCHAR(64),
    email VARCHAR(64),
    phone VARCHAR(16),
    last_login_ip VARCHAR(16),
    register_time DATETIME,
    register_ip VARCHAR(16)
);

-- 拆分后：用户扩展信息表（低频访问字段 + 大字段）
CREATE TABLE user_ext (
    id BIGINT PRIMARY KEY,
    avatar_url VARCHAR(256),
    bio TEXT,
    vip_level TINYINT,
    points INT,
    preferences JSON
);
```

**常用拆分规则**：

| 规则 | 说明 | 拆分对象 |
|------|------|---------|
| 大字段分离 | 将TEXT/BLOB/JSON等大字段拆到独立表 | `bio`、`preferences`、`avatar_data` |
| 冷热数据分离 | 将高频访问字段（热数据）与低频字段（冷数据）拆分 | 登录信息 vs 历史设置 |
| 更新频率分离 | 将高频更新字段与低频更新字段拆分 | `last_login_ip`（高频写）vs `vip_level`（低频写） |
| 访问模式分离 | 将列表页需要的字段与详情页需要的字段拆分 | 列表需`username`+`avatar`，详情需`bio`+`preferences` |

#### 1.2 水平分表：按行拆分

水平分表（Horizontal Partitioning / Sharding）是指将一张行数过大的表，按行拆分为多张结构完全相同的子表（也叫分片，Shard）。每张子表包含原表的一个行子集。

**形式化定义**：给定原表 `R` 的行集合 `{r1, r2, ..., rm}`，水平分表将其拆分为 `R1, R2, ..., Rn`，满足 `R = R1 ∪ R2 ∪ ... ∪ Rn` 且 `Ri ∩ Rj = ∅ (i≠j)`，所有子表结构相同。

**典型示例**：订单表按用户ID哈希分16张表

```sql
-- 拆分后：16张结构完全相同的子表
CREATE TABLE order_0 (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    order_no VARCHAR(32) NOT NULL,
    total_amount DECIMAL(10,2),
    status TINYINT,
    create_time DATETIME
);
-- 同理 order_1, order_2, ..., order_15，结构完全一致

-- 路由规则（Sharding Strategy）：user_id % 16
-- 当 user_id = 1001 时，路由到 order_1001%16 = order_9
-- 当 user_id = 2050 时，路由到 order_2050%16 = order_2
```

### 2. 底层原理

#### 2.1 垂直分表的底层原理：页存储与行溢出

**InnoDB Page（数据库页）存储结构**

MySQL InnoDB 存储引擎以 **Page（页）** 为最小I/O单位，默认每个 Page 大小为 **16KB**。数据行（Row）以 B+ 树（B+ Tree）的叶子节点形式存储在 Page 中：

```
InnoDB Page (16KB)
┌──────────────────────────────────┐
│  File Header (38B)               │
├──────────────────────────────────┤
│  Page Header (56B)               │
├──────────────────────────────────┤
│  Infimum + Supremum Records (26B)│
├──────────────────────────────────┤
│  ┌── Row Data ──────────────┐   │
│  │  row_1 (500B)            │   │
│  │  row_2 (500B)            │   │
│  │  ...                     │   │
│  │  row_n (500B)            │   │  ← 一行越宽，Page能容纳的行数越少
│  └──────────────────────────┘   │
├──────────────────────────────────┤
│  Free Space                      │
├──────────────────────────────────┤
│  Page Directory                  │
├──────────────────────────────────┤
│  File Trailer (8B)               │
└──────────────────────────────────┘
```

**关键计算**：一个 16KB 的 Page，除去 File Header（38B）、Page Header（56B）、File Trailer（8B）等固定开销（约120B）后，可用空间约 16256B。如果表中包含 `TEXT`、`BLOB`、`JSON` 等大字段，或者列数过多导致单行数据很大（如 4KB/行），一个 Page 只能存储约 4 行，导致 B+ 树层级加深、查询需要读取更多 Page、I/O 次数剧增。

**行溢出（Row Overflow）**

InnoDB 对行大小有硬限制：**一行的所有列（不包括TEXT/BLOB）不得超过 Page 大小的一半（约 8126B）**。当行数据超过此限制或包含 TEXT/BLOB 类型时，会发生行溢出：

- **行内存储（Inline Storage）**：对于 VARCHAR 或较短的 TEXT，InnoDB 会优先尝试在行内存储所有数据。
- **页外存储（Off-page Storage）**：如果 TEXT/BLOB 字段超过一定长度（TEXT 最大 65535B，BLOB 最大 65535B），InnoDB 会在行内只保留一个 20 字节的溢出指针（Overflow Pointer），实际数据存储在单独的溢出页（Overflow Page）中，查询时需要额外的一次 I/O 获取数据。

```
行内（未拆分）：
┌─────────────────────────────────────────────┐
│ id | name | email | bio (TEXT, 5000B)        │
│ 1  | Tom  | t@m   │ bio content... (溢出指针) │  ← 需要额外I/O读取溢出页
└─────────────────────────────────────────────┘

垂直分表后：
┌────────────────────┐   ┌──────────────────────┐
│ id | name | email  │   │ id | bio (TEXT)       │
│ 1  | Tom  | t@m    │   │ 1  | bio content...   │  ← 查基本信息时不触发生物字段
└────────────────────┘   └──────────────────────┘
```

垂直分表的本质优化就是**将行溢出的大字段分离到独立表**，使主表每行变窄，一个 Page 容纳更多行，大幅降低查询时的 I/O 次数。

**聚簇索引（Clustered Index）与非聚簇索引（Secondary Index）**

InnoDB 使用 **聚簇索引（Clustered Index）** 作为主键索引，叶子节点直接存储完整行数据。非聚簇索引（Secondary Index / Non-clustered Index）的叶子节点存储主键值，回表查询（Index Lookup）时通过主键到聚簇索引查找完整行。

```
聚簇索引（按主键组织）        非聚簇索引（按username组织）
┌──────────────┐            ┌──────────────┐
│  Page 1      │            │  Page 1      │
│  id=1 行数据  │            │  "Alice"→id=3│
│  id=2 行数据  │            │  "Bob"  →id=1│
│  id=3 行数据  │            │  "Tom"  →id=2│
└──────────────┘            └──────┬───────┘
                                    │ 回表查询
                                    ▼
                             ┌──────────────┐
                             │  聚簇索引     │
                             │  id=3 行数据  │
                             └──────────────┘
```

垂直分表后，每张子表各有一套聚簇索引，行宽变窄，索引层级减少（通常 B+ 树从 3 层降为 2 层），查询效率显著提升。

#### 2.2 水平分表的底层原理：分片路由与数据分布

**分片策略（Sharding Strategy）**

水平分表的核心是**分片键（Sharding Key）**的选择和**路由策略（Routing Strategy）**的确定。常用的分片策略包括四种：

**（1）Range 分片（范围分片，Range Sharding）**

按分片键的范围区间将数据分配到不同表：

```sql
-- 订单按时间范围分片
-- order_202401: 2024年1月的订单
-- order_202402: 2024年2月的订单
-- order_202403: 2024年3月的订单

-- 路由规则：根据 create_time 确定目标表
-- 2024-01-15 → order_202401
-- 2024-02-20 → order_202402
```

优点：实现简单，范围查询友好（直接访问连续的分片）；缺点：**数据倾斜（Data Skew）**问题严重——热点月份数据集中写入某一片，该片成为性能瓶颈。

**（2）Hash 分片（哈希分片，Hash Sharding）**

对分片键计算哈希值后取模：

```sql
-- 用户表按 user_id 哈希分 8 张表
-- 路由规则：user_id % 8
-- user_id=1001 → 1001 % 8 = 1 → user_1
-- user_id=1002 → 1002 % 8 = 2 → user_2
```

优点：数据分布均匀，避免热点问题；缺点：**扩容困难**（改变分片数后取模结果全变，需要大规模数据迁移），范围查询需遍历所有分片。

**（3）List 分片（列表分片，List Sharding）**

按分片键的枚举值列表确定目标表：

```sql
-- 按地区分片
-- user_east: 华东地区用户（上海、江苏、浙江）
-- user_south: 华南地区用户（广东、福建、海南）
-- user_west: 西部地区用户（四川、重庆、陕西）

CASE region
    WHEN '上海' THEN 'user_east'
    WHEN '江苏' THEN 'user_east'
    WHEN '广东' THEN 'user_south'
    ...
END
```

**（4）Composite 分片（复合分片，Composite Sharding）**

组合多种策略分片：

```sql
-- 先按 user_id % 16 确定分库（16个库）
-- 再按 create_time 月做 Range 分表
-- 最终表名：order_{dbIndex}_{month}
```

### 3. 实践应用

#### 3.1 垂直分表的应用场景

**场景一：电商用户系统**

```sql
-- 用户基础表：高频访问 + 少量列
CREATE TABLE user_base (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(32) NOT NULL UNIQUE,
    password_hash VARCHAR(128) NOT NULL,
    email VARCHAR(64),
    phone VARCHAR(16),
    status TINYINT DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_email (email),
    INDEX idx_phone (phone)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 用户统计表：高频更新，单独拆分避免行锁竞争
CREATE TABLE user_stats (
    id BIGINT PRIMARY KEY,
    coins INT DEFAULT 0,
    points INT DEFAULT 0,
    experience INT DEFAULT 0,
    level INT DEFAULT 1,
    total_orders INT DEFAULT 0,
    total_spent DECIMAL(12,2) DEFAULT 0.00,
    FOREIGN KEY (id) REFERENCES user_base(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 用户设置表：大字段 + 低频访问
CREATE TABLE user_settings (
    id BIGINT PRIMARY KEY,
    preferences JSON,
    notification_config JSON,
    privacy_settings JSON,
    theme VARCHAR(16) DEFAULT 'light',
    FOREIGN KEY (id) REFERENCES user_base(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**场景二：内容管理系统（CMS，Content Management System）**

文章表拆分策略：
- `article_summary`：`id, title, author_id, status, create_time, view_count`（列表页需要，窄表）
- `article_body`：`id, content TEXT, html_content MEDIUMTEXT`（详情页需要，大字段）

#### 3.2 水平分表的应用场景

**场景一：订单系统（按用户ID哈希分片）**

```yaml
# ShardingSphere 水平分表配置示例
spring:
  shardingsphere:
    datasource:
      names: ds
      ds:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        url: jdbc:mysql://localhost:3306/order_db
        username: root
        password: root
    sharding:
      tables:
        t_order:
          actual-data-nodes: ds.t_order_$->{0..15}  # 16张分表
          table-strategy:
            standard:
              sharding-column: user_id                # 分片键
              precise-algorithm-class-name: com.example.OrderPreciseAlgorithm
          key-generator:
            column: id
            type: SNOWFLAKE                           # 雪花算法生成全局唯一ID
```

```java
// 自定义分片算法（PreciseShardingAlgorithm）
public class OrderPreciseAlgorithm implements PreciseShardingAlgorithm<Long> {
    @Override
    public String doSharding(Collection<String> tableNames, 
                             PreciseShardingValue<Long> shardingValue) {
        Long userId = shardingValue.getValue();
        int shardIndex = (int) (userId % 16);           // Hash分片
        for (String tableName : tableNames) {
            if (tableName.endsWith(String.valueOf(shardIndex))) {
                return tableName;
            }
        }
        throw new IllegalArgumentException("No table found for userId: " + userId);
    }
}
```

**场景二：日志系统（按时间Range分片）**

```sql
-- 每月一张日志表
CREATE TABLE access_log_202406 (
    id BIGINT,
    ip VARCHAR(16),
    url VARCHAR(512),
    status_code INT,
    duration_ms INT,

-- 查询时指定表名
SELECT COUNT(*) FROM access_log_202406 
WHERE request_time >= '2024-06-01' AND request_time < '2024-07-01';
```

#### 3.3 分片键（Sharding Key）的选择原则

分片键选择是水平分表设计中最关键的决策，遵循以下四个原则：

| 原则 | 说明 | 反例 |
|------|------|------|
| **数据均匀**（Data Uniformity） | 各分片数据量尽量一致 | 按地区分片导致某些地区订单量远大于其他地区 |
| **查询友好**（Query Affinity） | 大部分查询携带分片键 | 按创建时间分片但查询按用户ID |
| **不可变性**（Immutability） | 分片键值不应变更 | 按手机号分片但用户可以换号 |
| **高基数**（High Cardinality） | 分片键的取值分布足够广 | 按性别（仅2个值）分片 |

**推荐分片键**：用户ID、订单ID、设备ID等天然均匀分布且查询频率高的字段。

### 4. 深入思考

#### 4.1 垂直分表的JOIN问题与反范式化设计权衡

垂直分表将原本在一张表的字段分散到多张表，查询时需通过 JOIN 或多次查询拼装数据：

**问题**：JOIN操作对性能影响显著——需要读取多张表的聚簇索引，在内存中进行行匹配，表数越多性能下降越明显。

**权衡方案**：反范式化设计（Denormalization）——在基础表中冗余存储少量扩展表字段，减少JOIN次数。

**决策原则**：
- **读多写少**的字段：适合冗余（如用户等级），写入时同步更新即可
- **读少写多**的字段：不适合冗余（如登录IP），反范式化带来的写同步成本高于收益
- **一致性要求高**的字段：避免冗余，宁可JOIN也要保证数据一致性

#### 4.2 水平分表的扩容难题与一致性哈希（Consistent Hashing）

**扩容痛点**：Hash分片下，从 N 片扩容到 M 片时，`hash(key) % N` 变为 `hash(key) % M`，几乎所有数据的路由目标都会变化，需要大规模数据迁移（Re-sharding）。

**传统取模迁移代价示例**：
```
扩容前：4 张表，key=1 → 1%4=1 → 表1
扩容后：8 张表，key=1 → 1%8=1 → 表1（命中）
        key=5 → 5%4=1 → 表1（旧），5%8=5 → 表5（新）→ 需要迁移！
        平均约 75% 的数据需要迁移
```

**一致性哈希（Consistent Hashing）** 的解决思路：

一致性哈希将哈希值空间组织成一个虚拟环（Hash Ring，范围 0~2^32-1），数据节点映射到环上，数据按顺时针方向找到最近的节点。

**扩容时的影响**：新增一个节点时，仅影响该节点顺时针方向的相邻节点的数据，平均只需迁移 `1/(N+1)` 的数据（N为原节点数）。相比传统取模的 `N/(N+1)`，迁移量大幅降低。

**虚拟节点（Virtual Node）优化**：一致性哈希在节点较少时可能出现数据倾斜——部分节点负载远大于其他节点。引入虚拟节点（每个物理节点对应 K 个虚拟节点）使数据分布更均匀。K 的典型取值为 100~200。

#### 4.3 分布式全局ID生成：雪花算法（Snowflake Algorithm）

水平分表后，数据库自增ID不再适用（不同分片会生成重复ID）。需要全局唯一的分布式ID生成器。**雪花算法（Snowflake Algorithm）** 是 Twitter 开源的最广泛使用的解决方案。

**ID 结构（64位 Long）**：
- 1 bit：符号位，始终为 0
- 41 bits：毫秒级时间戳，可用 69 年
- 10 bits：工作机器 ID（5 bit 数据中心 + 5 bit 机器 ID）
- 12 bits：序列号，每毫秒可生成 4096 个 ID

**算法特点**：
- **全局唯一**：分布式环境下不依赖中心节点，每台机器独立生成不重复ID
- **趋势递增**：时间戳高位在前，整体呈递增趋势
- **高性能**：纯位运算生成，单机可达百万级/秒
- **容错**：时钟回拨（Clock Drift）问题——如果服务器时间回拨，可能生成重复ID。解决方案：记录上次生成时间戳，回拨时等待或使用备用序列号段

#### 4.4 跨分片查询（Cross-shard Query）与分布式事务

**跨分片查询（Cross-shard Query）**

当查询条件不包含分片键时，系统需要将查询广播到所有分片，然后合并结果——这是水平分表最严重的性能问题之一。

```sql
-- 订单表按 user_id 分片
-- 当按 order_no 查询时，无法路由到特定分片：
SELECT * FROM t_order WHERE order_no = 'ORD202406150001';
-- → 需要遍历所有 16 张分表

-- 解决方案一：建立全局索引表
CREATE TABLE order_no_index (
    order_no VARCHAR(32) PRIMARY KEY,
    user_id BIGINT NOT NULL,
    shard_index INT NOT NULL
);

-- 解决方案二：Elasticsearch 构建二级索引
```

其他跨分片查询场景：

| 场景 | 问题 | 解决方案 |
|------|------|---------|
| 跨分片排序 | 各分片TOP N合并后不是全局TOP N | 各分片取N行，应用层归并排序 |
| 跨分片聚合 | COUNT/SUM/AVG 需要汇总各分片结果 | 分片分别计算后汇总；AVG需SUM+COUNT二次计算 |
| 跨分片分页 | 深度分页页面越大性能越差 | 禁止深度分页，改用游标分页（Cursor-based Pagination） |

**分布式事务（Distributed Transaction）**

| 方案 | 一致性模型 | 适用场景 | 核心算法 |
|------|-----------|---------|---------|
| 两阶段提交（2PC） | 强一致性 | 金融、支付 | Prepare → Commit/Rollback |
| TCC（Try-Confirm-Cancel） | 最终一致性 | 高并发场景 | 预留资源 → 确认/回滚 |
| 可靠消息最终一致性 | 最终一致性 | 异步解耦场景 | 本地消息表 + MQ补偿 |
| Seata AT | 最终一致性 | 通用场景 | 全局事务代理 + 回滚日志 |

### 重点得分点

1. **说清楚本质区别（按列 vs 按行）**——这是最基础的区分，也是面试官最在意的第一印象。能够直接说出"垂直分表对列进行拆分（Column-level），水平分表对行进行拆分（Row-level）"即获得基准分。

2. **讲透底层原理而非停留在表面**——从InnoDB Page 16KB、行溢出（Off-page Storage）角度解释垂直分表为什么能提升I/O性能；从分片路由算法角度解释水平分表为什么能扩展写能力。这一层是区分"背答案"和"真理解"的关键。

3. **展示代码和配置实战能力**——能写出ShardingSphere的`actual-data-nodes`配置和自定义`PreciseShardingAlgorithm`实现，证明不是纸上谈兵。

4. **点出扩容和跨分片查询等核心难点**——面试官特别关注候选人是否对分表方案的"代价"有清醒认知。主动提出一致性哈希（Consistent Hashing）、分布式全局ID（Snowflake Algorithm）和跨分片查询（Cross-shard Query）解决方案，展现系统性思考能力。

5. **区分分表与分区（Partitioning vs. Partition）**——这是高频易混淆点，主动点出差异可加分。

### 常见误区

1. **"垂直分表和水平分表只能选一种"**——实际上两者可以同时使用（混合分片）。先垂直拆分解决宽表问题，再水平拆分解决大表问题。
2. **"分表一定能提升所有查询性能"**——片面理解。垂直分表减少了不必要的列扫描，但需要JOIN查询；水平分表若查询条件不含分片键，需遍历所有分片，性能比单表更差。
3. **"分表和分区（Partition）是同一回事"**——重大误区。MySQL Partition是单个实例内的行级拆分，对应用透明；分表（Sharding）是跨实例/跨表的拆分，通常需要中间件支持。
4. **"分表只解决读性能问题"**——不准确。水平分表的核心动机之一是提升写性能，垂直分表主要提升读性能。
5. **"Hash分片一劳永逸"**——Hash分片数据分布均匀，但扩容时需要全量数据重分布，成本极高。

### 过渡话术

- **从概念到原理**："以上是两者的基本定义。接下来我从底层存储引擎的角度解释，为什么列数多会降低性能，以及为什么行数多需要做分片。"
