# 线程、进程、协程相关知识点

---

## 🧠 联想记忆法

**记忆口诀："进线协，重中轻"**

将三者想象为三个不同规模的"工作单位"：

- **进程（Process）** → 独立公司
  - 每家公司有自己的办公楼（独立地址空间）、自己的资源（内存/文件句柄）
  - 公司之间来往需要打电话/发邮件（IPC：管道、消息队列、共享内存）
  - 搬一整个公司到新地址（上下文切换）成本极高，需要重新装修办公室（切换页表、刷新 TLB）
  - "进" = "进"入一个全新的独立世界

- **线程（Thread）** → 公司里的员工
  - 所有员工在同一栋楼办公（共享进程地址空间），使用公共资源
  - 员工之间的沟通很方便（共享内存通信），但需要协调避免冲突（synchronized、Lock）
  - 换一个员工上岗（上下文切换）只需换工牌和工具（切换寄存器、栈指针），成本中等
  - "线" = "线"程是执行的基本单元，像一条执行流水线

- **协程（Coroutine）** → 自由职业者在家办公
  - 不需要公司办公楼（无内核资源消耗），自己安排时间（用户态调度）
  - 切换任务只需保存几页笔记（切换栈帧），成本极低，几 KB 内存即可
  - 一个人可以同时接多个项目（海量并发），但重活（CPU 密集型）还是得去公司
  - "协" = "协"作、协同，主动让出执行权

**一句话记忆**：进程是独立公司（重），线程是公司员工（中），协程是自由职业者（轻）。

## 📖 深度解答

### 一、核心概念

#### 1. 进程（Process）

**进程**是操作系统进行**资源分配**的最小单位。每个进程拥有独立的地址空间（Address Space），包括代码段、数据段、堆、栈，以及独立的文件描述符表、信号处理表等系统资源。进程之间彼此隔离，一个进程的崩溃通常不会影响其他进程。

**进程间通信（Inter-Process Communication, IPC）** 的主要方式：

| 通信方式 | 说明 | 特点 |
|---------|------|------|
| 管道（Pipe） | 半双工通信，父子进程间使用 | 简单但仅限有亲缘关系进程 |
| 消息队列（Message Queue） | 内核维护的消息链表 | 异步通信，消息有结构 |
| 共享内存（Shared Memory） | 多个进程映射同一物理内存 | 速度最快，需配合信号量同步 |
| 信号量（Semaphore） | 计数器，用于同步和互斥 | 解决竞争条件（Race Condition） |
| Socket | 跨网络通信 | 可用于不同主机间的进程通信 |

当 CPU 从一个进程切换到另一个进程时，需要**上下文切换（Context Switch）**：保存当前进程的寄存器状态、程序计数器（Program Counter），切换页表（Page Table）基地址寄存器，刷新 TLB（Translation Lookaside Buffer，页表缓存）。这一过程通常需要几十到几百微秒，且 TLB 刷新会导致后续内存访问的缓存缺失。

#### 2. 线程（Thread）

**线程**是 CPU **调度**的最小单位，隶属于进程。同一进程内的线程共享该进程的地址空间、文件描述符、信号处理器等资源，但每个线程拥有独立的栈（Stack）和寄存器上下文。

**Java 中的线程实现**：

```java
// 方式一：继承 Thread 类
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread executing: " + Thread.currentThread().getName());
    }
}
new MyThread().start();

// 方式二：实现 Runnable 接口
Thread thread = new Thread(() -> {
    System.out.println("Runnable executing");
});
thread.start();

// 方式三：实现 Callable 接口（带返回值）
FutureTask<Integer> futureTask = new FutureTask<>(() -> {
    return 42;
});
new Thread(futureTask).start();
Integer result = futureTask.get(); // 获取返回值

// 方式四：线程池 ThreadPoolExecutor
ExecutorService executor = Executors.newFixedThreadPool(10);
executor.submit(() -> System.out.println("Task executed by thread pool"));
executor.shutdown();
```

**线程同步机制**：

- **synchronized**：JVM 内置锁（Monitor），基于对象头的 Mark Word 实现，通过 monitorenter/monitorexit 指令保证互斥
- **ReentrantLock**：java.util.concurrent.locks 包提供的可重入锁，支持公平锁、可中断、条件等待（Condition）
- **volatile**：保证可见性（Visibility），禁止指令重排序（Instruction Reordering），但不保证原子性
- **CAS（Compare And Swap）**：硬件级别的原子操作，Unsafe 类提供，是 AtomicInteger 等原子类的底层实现，通过自旋避免阻塞

线程上下文切换仅需保存/恢复寄存器和栈指针（Stack Pointer），不涉及页表切换和 TLB 刷新，成本通常在几微秒量级。

#### 3. 协程（Coroutine）

**协程**是用户态（User Mode）的轻量级线程，由程序员或运行时库自行调度，而非操作系统内核。协程的创建和切换完全在用户态完成，不涉及系统调用（System Call），因此开销极低。

**Java 虚拟线程（Virtual Thread）**：Project Loom 的产物，Java 21（LTS）正式发布。

```java
// 传统线程 —— 创建 10000 个线程压力很大
try (ExecutorService executor = Executors.newFixedThreadPool(10000)) {
    for (int i = 0; i < 10000; i++) {
        executor.submit(() -> {
            try { Thread.sleep(1000); } catch (InterruptedException e) { }
            System.out.println("Platform thread");
        });
    }
}
// 实际运行可能导致 OOM 或系统崩溃

// 虚拟线程 —— 创建 100000 个轻量级任务轻松自如
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100000; i++) {
        executor.submit(() -> {
            try { Thread.sleep(1000); } catch (InterruptedException e) { }
            System.out.println("Virtual thread");
        });
    }
}
```

虚拟线程的核心优势：
- **创建成本极低**：约几 KB 内存（vs 传统线程约几 MB 栈空间）
- **无 OS 上下文切换**：由 JVM 在用户态调度，切换成本约纳秒级
- **海量并发**：一台服务器可轻松支撑百万级虚拟线程
- **兼容性**：不改变 Java `Thread` 的 API 语义，现有代码几乎无需修改

**其他语言的协程实现**：
- **Python**：`async/await` + `asyncio` 事件循环
- **Go**：`goroutine` + `channel`，Go 运行时调度器（GMP 模型）
- **Kotlin**：`suspend` 函数 + `CoroutineScope`
- **C++20**：无栈协程（Stackless Coroutine）

### 二、底层原理

#### 进程与线程的 Linux 内核视角

在 Linux 内核中，进程和线程都通过 `task_struct`（任务结构体）表示。`clone()` 系统调用的标志位决定了资源的共享程度：

```c
// 创建进程：完全独立的地址空间
clone(CLONE_VM, ...); // 不设置 CLONE_VM → 独立地址空间

// 创建线程：共享地址空间、文件描述符、信号处理器
clone(CLONE_VM | CLONE_FILES | CLONE_SIGHAND, ...);
```

**内核态 vs 用户态切换**：
- 进程/线程的创建、销毁、调度都需要陷入内核态（Trap/System Call）
- 协程的创建和切换完全在用户态完成，不需要系统调用

#### 协程的调度模型

**有栈协程（Stackful Coroutine）** vs **无栈协程（Stackless Coroutine）**：

| 类型 | 代表 | 特点 |
|------|------|------|
| 有栈协程 | Java 虚拟线程、Go goroutine | 拥有独立的栈，可在任意嵌套函数中挂起 |
| 无栈协程 | Python async/await、C++20 | 利用状态机实现，只能在函数顶层挂起 |

Java 虚拟线程的实现依赖 **continuation**（延续）机制：当虚拟线程执行阻塞 I/O 操作时，JVM 会将其挂起（yield），将底层平台线程（Carrier Thread）释放给其他虚拟线程使用，实现 **M:N 调度**（M 个虚拟线程映射到 N 个平台线程）。

### 三、实践应用

#### 三者对比表格

| 维度 | 进程（Process） | 线程（Thread） | 协程（Coroutine） |
|------|:----------:|:----------:|:--------------:|
| **调度者** | 操作系统内核 | 操作系统内核 | 程序/运行时（用户态） |
| **地址空间** | 独立 | 共享进程空间 | 共享/独立（取决于实现） |
| **内存占用** | 几十~几百 MB | 每线程约 1~8 MB 栈 | 每协程约几 KB |
| **创建开销** | 高（需分配独立地址空间） | 中等（需分配内核栈） | 极低（用户态分配） |
| **切换开销** | 高（切页表、刷新 TLB） | 中等（切换寄存器、栈） | 极低（仅保存栈帧） |
| **通信方式** | IPC（管道、消息队列、共享内存） | 共享内存 + 同步机制 | 共享变量 / Channel |
| **隔离性** | 强（进程间隔离） | 弱（同进程线程共享） | 弱 |
| **适用场景** | 多进程架构、隔离性要求高 | CPU 密集型任务 | I/O 密集型、高并发任务 |

#### 实际场景选择

**I/O 密集型**（网络请求、文件读写、数据库查询）：
- **最佳选择**：协程 / 虚拟线程
- 原因：I/O 操作时协程自动 yield，释放执行资源给其他协程
- Java 示例：`ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();`

**CPU 密集型**（计算、加密、图像处理）：
- **最佳选择**：线程池（固定大小）
- 原因：线程数通常设为 `CPU核心数 + 1`，避免过多上下文切换
- Java 示例：`Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors())`

**高隔离性需求**（不同租户、不同应用）：
- **最佳选择**：进程（容器 / Docker）

### 四、深入思考

#### 1. 为什么虚拟线程不适用于 CPU 密集型任务？

虚拟线程的切换依赖于阻塞操作（如 I/O）来触发 yield。对于纯计算任务，虚拟线程没有机会自动让出底层平台线程，一个虚拟线程会长期占用一个平台线程，此时与普通线程无异。同时，由于虚拟线程的调度器（ForkJoinPool）需要额外管理任务队列，反而带来微小的性能损耗。

#### 2. 协程能否完全替代线程？

不能。协程需要语言或运行时环境的原生支持。操作系统底层调度、硬件中断处理、信号处理等仍然必须依赖内核线程。协程是"协作式"（Cooperative）调度，依赖于协程主动让出执行权（yield/await），如果一个协程陷入死循环且不 yield，就会阻塞整个线程。而线程是"抢占式"（Preemptive）调度，由操作系统强制切换。

#### 3. 虚拟线程的性能真相

虚拟线程并非"免费"。当虚拟线程数量达到百万级时，JVM 需要管理大量的 continuation 对象，GC（垃圾回收，Garbage Collection）压力会显著增加。此外，`synchronized` 块中的虚拟线程可能会被固定（Pinning）到底层平台线程，导致无法按预期切换。Java 22 中的 `-XX:+UnlockDiagnosticVMOptions -XX:+DebugNonSafepoints` 可以帮助诊断此类问题。

#### 4. 三者的设计哲学

- **进程**：隔离与安全 —— 操作系统设计的核心是"保护"（Protection）
- **线程**：资源共享与并行 —— 解决的是"如何利用多核"的问题
- **协程**：高效协作 —— 解决的是"如何让等待不阻塞"的问题

### 避坑提醒
- **不要只说概念**：一定要有代码示例和切换开销的数字对比
- **不要忽略虚拟线程的局限**：提一下 Pinning 问题，展现辩证思考
- **不要说"协程比线程快"**：协程在 I/O 场景下整体吞吐更高，但单次操作不一定更快
- **不要忘记中英对照**：Context Switch、TLB、IPC、System Call 等术语第一次出现时给英文

> **面试官可能的追问**：
> 1. "一个进程中有 10000 个虚拟线程，实际底层有几个平台线程？"（答：通常取决于 I/O 等待数量，ForkJoinPool 默认 parallelism = CPU 核心数）
> 2. "虚拟线程和 Go 的 goroutine 有什么区别？"（答：虚拟线程是 JVM 实现的有栈协程，goroutine 是 Go 运行时实现的有栈协程，Go 的 GMP 模型更为成熟，但 Java 的优势在于兼容现有生态）
> 3. "synchronized 和 ReentrantLock 在虚拟线程中哪个更好？"（答：ReentrantLock 更好，因为 synchronized 会导致虚拟线程被 Pinning）
