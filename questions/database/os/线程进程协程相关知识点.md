# 线程、进程、协程相关知识点

## 🧠 联想记忆法

**记忆口诀："进独享、线共享、协轻量"**

将三者想象成不同层级的"工作单元"：

- **进程（Process）** = 一栋独立的别墅
  - 每栋别墅有独立的围墙（地址空间）、水电表（资源）
  - 别墅之间"老死不相往来"，通信必须通过电话（IPC：管道、Socket）
  - 搬家（上下文切换）代价巨大——要搬一整套家具、重新布线

- **线程（Thread）** = 别墅里的各个房间
  - 共享客厅、厨房、卫生间（进程的地址空间和资源）
  - 但每个房间有自己的床铺和书桌（栈空间和寄存器）
  - 换房间（线程切换）只需带个人物品走，不用动整栋别墅

- **协程（Coroutine）** = 房间里书桌上的笔记本
  - 想换本书看（切换协程）——手一伸就行，不用离开椅子
  - 完全由你自己决定什么时候换书（用户态调度）
  - 成本几乎为零，笔记本里可以夹几百张书签（几 GB 内存轻松创建百万协程）

**一句话串记**：进程是出租房独门独户（独立资源），线程是合租房共用客厅（共享内存），协程是你书桌上的笔记本（轻量自调度）。

## 📖 深度解答

### 第一层：核心概念

#### 1. 进程（Process）

**进程**是操作系统进行**资源分配**的最小单位。每个进程拥有**独立的虚拟地址空间**，包括代码段、数据段、堆和栈，进程A无法直接访问进程B的内存。

**进程间通信（Inter-Process Communication, IPC）** 是操作系统的核心机制，主要方式包括：

| IPC 方式 | 原理 | 特点 |
|---------|------|------|
| 管道（Pipe） | 内核缓冲区，一端写一端读 | 单向、仅限父子进程，半双工 |
| 消息队列（Message Queue） | 内核维护的消息链表 | 双向、消息有类型，适合小数据 |
| 共享内存（Shared Memory） | 映射同一块物理内存 | 速度最快，需搭配信号量同步 |
| 信号量（Semaphore） | 计数器 + PV 操作 | 主要用于同步，而非数据传输 |
| Socket | 网络套接字 | 跨机器通信，支持 TCP/UDP |

进程**上下文切换（Context Switch）** 开销大，因为需要：
1. 保存/恢复 CPU 寄存器状态
2. 切换**页表（Page Table）**——每个进程的虚拟地址到物理地址的映射表不同
3. **刷新 TLB（Translation Lookaside Buffer）**——页表缓存，切换后全部失效
4. 清空 CPU 流水线（Pipeline）

这些操作通常需要数百到数千个 CPU 周期，属于**重量级**操作。

#### 2. 线程（Thread）

**线程**是 CPU **调度**的最小单位，是进程内的一个执行流。同一进程的所有线程共享：
- 代码段（Code Segment）
- 数据段（Data Segment，含全局变量）
- 堆（Heap）
- 打开的文件描述符（File Descriptor）

每个线程独有：
- **线程栈（Thread Stack）**
- **程序计数器（Program Counter, PC）**
- **寄存器状态（Register Set）**

**Java 中的线程实现：**

```java
// 方式1：继承 Thread 类
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread running: " + Thread.currentThread().getName());
    }
}
new MyThread().start();

// 方式2：实现 Runnable 接口
Thread thread = new Thread(() -> System.out.println("Runnable running"));
thread.start();

// 方式3：实现 Callable 接口（有返回值、可抛异常）
FutureTask<String> futureTask = new FutureTask<>(() -> {
    Thread.sleep(1000);
    return "Callable result";
});
new Thread(futureTask).start();
String result = futureTask.get(); // 获取返回值

// 方式4：线程池 ThreadPoolExecutor
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2,                      // corePoolSize
    5,                      // maximumPoolSize
    60L, TimeUnit.SECONDS,  // keepAliveTime
    new LinkedBlockingQueue<>(10), // workQueue
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.AbortPolicy() // 拒绝策略
);
executor.execute(() -> System.out.println("Thread pool task"));
executor.shutdown();
```

**线程同步机制：**

| 机制 | 原理 | 适用场景 |
|------|------|----------|
| `synchronized` | JVM 层面的 Monitor 锁，基于 Object Header 的 Mark Word | 简单的互斥同步，自动释放锁 |
| `ReentrantLock` | AQS（AbstractQueuedSynchronizer）实现，支持公平/非公平锁 | 需要超时、可中断、尝试获取等高级功能 |
| `volatile` | 内存屏障（Memory Barrier），保证可见性，禁止指令重排序 | 单个变量的读写可见性保证 |
| CAS（Compare-And-Swap） | 硬件级原子指令（如 x86 的 `CMPXCHG`），非阻塞 | 乐观锁场景，如 AtomicInteger |

**特点对比：**
- `synchronized` 锁会膨胀：无锁 → 偏向锁 → 轻量级锁 → 重量级锁（JDK 6+ 优化）
- `volatile` 不保证原子性，仅保证可见性和有序性
- CAS 存在 ABA 问题（可通过 `AtomicStampedReference` 解决）

#### 3. 协程（Coroutine）与 Java 虚拟线程（Virtual Thread）

**协程（Coroutine）** 是**用户态（User Space）的轻量级线程**，由程序自身调度，而非操作系统内核。协程的调度器运行在用户态，切换时**不需要陷入内核态（Kernel Mode）**，因此开销极低。

**Java 虚拟线程（Virtual Thread）** 是 JDK 21 正式发布的协程实现，属于 **Project Loom** 的成果。虚拟线程由 JVM 管理，挂载在若干平台线程（Platform Thread，即传统的 OS 线程）上执行，采用 **M:N 调度模型**（M 个虚拟线程映射到 N 个平台线程）。

**代码示例：Java 虚拟线程 vs 传统线程**

```java
import java.util.concurrent.*;
import java.util.stream.IntStream;

public class ThreadVsVirtualThread {

    // 传统平台线程 —— 创建 10,000 个线程
    public static void platformThreads() throws InterruptedException {
        CountDownLatch latch = new CountDownLatch(10000);
        long start = System.currentTimeMillis();

        IntStream.range(0, 10000).forEach(i -> {
            Thread t = new Thread(() -> {
                try {
                    Thread.sleep(100); // 模拟 I/O 等待
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
                latch.countDown();
            });
            t.start();
        });

        latch.await();
        long end = System.currentTimeMillis();
        System.out.println("平台线程 10000 个耗时: " + (end - start) + "ms");
        // 结果：运行缓慢，有可能 OOM，因为每个 OS 线程默认栈 1MB
    }

    // 虚拟线程 —— 创建 100,000 个虚拟线程
    public static void virtualThreads() throws InterruptedException {
        CountDownLatch latch = new CountDownLatch(100000);
        long start = System.currentTimeMillis();

        IntStream.range(0, 100000).forEach(i -> {
            Thread.startVirtualThread(() -> {
                try {
                    Thread.sleep(100); // 虚拟线程在此处会 yield，不阻塞平台线程
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
                latch.countDown();
            });
        });

        latch.await();
        long end = System.currentTimeMillis();
        System.out.println("虚拟线程 100000 个耗时: " + (end - start) + "ms");
        // 结果：轻松完成，因为每个虚拟线程仅需几 KB
    }

    public static void main(String[] args) throws Exception {
        // 注意：不要同时运行，平台线程 10k 可能让系统变慢
        virtualThreads();
    }
}
```

**虚拟线程的关键优势：**
- 创建成本极低：几 KB（仅需栈帧对象）vs 传统线程的几 MB（需要内核栈分配）
- 切换成本趋近于零：纯用户态方法调用级别，无需系统调用
- 百万级并发：单台 JVM 可创建数百万虚拟线程
- I/O 阻塞时自动 yield：虚拟线程执行 I/O 操作时自动让出平台线程，不浪费 CPU

### 第二层：底层原理

#### CPU 的工作模式与上下文切换

CPU 运行在两种模式下：
- **内核态（Kernel Mode）**：可以执行特权指令，访问所有内存空间
- **用户态（User Mode）**：只能执行普通指令，访问受限内存

**进程/线程切换 → 需要系统调用（System Call）→ 从用户态切换到内核态 → 执行调度器代码 → 切回用户态**

涉及的关键步骤：
```
用户态 → int 0x80/syscall 指令 → 内核态 → 
  保存用户态寄存器 + 栈指针 → 
  调度器选择下一任务 → 
  切换页表 / 修改 CR3 寄存器（进程切换时才需要）→ 
  恢复下一任务寄存器 + 栈指针 → 
  返回用户态
```

**协程切换**只发生在用户态：
```
当前协程 → 主动 yield() → 协程调度器 → 选择下一协程 → 恢复其栈帧 → 继续执行
```
全程在用户态内完成，无需 **syscall**、无需修改 CR3（页表寄存器），因此开销极低（纳秒级）。

#### Java 虚拟线程的调度机制

虚拟线程采用 **Work-Stealing（工作窃取）调度**：
- JVM 维护一个 **FIFO 任务队列**，平台线程（称为 Carrier Thread）从中"窃取"虚拟线程执行
- 当虚拟线程执行到阻塞操作（I/O、`sleep()`、`lock()`）时，JVM 将其从平台线程上**卸载（Unmount）**，挂回队列
- 平台线程转而执行队列中下一个就绪的虚拟线程
- 阻塞操作完成后，虚拟线程重新加入就绪队列

```
[虚拟线程池]                    [平台线程（Carrier Threads）]
  VT1 (running)     ───────→    PT1 (执行 VT1)
  VT2 (ready)       ───────→    PT2 (执行 VT2)
  VT3 (blocked/I/O)              PT3 (VT3 yield → 执行 VT4)
  VT4 (ready)       ───────→    ...
```

### 第三层：实践应用

#### 三者对比表格

| 维度 | 进程（Process） | 线程（Thread） | 协程/虚拟线程（Coroutine/Virtual Thread） |
|------|---------------|---------------|----------------------------------------|
| **调度者** | 操作系统内核 | 操作系统内核 | 程序自身（用户态） |
| **内存占用** | 大（独立 4GB 虚拟地址空间） | 中等（约 1MB 栈/线程） | 极小（几 KB / 协程） |
| **创建开销** | 大（需复制页表、PCB等） | 中等（需系统调用分配内核栈） | 极小（纯用户态对象分配） |
| **切换开销** | 大（切换页表 + 刷新 TLB） | 中等（寄存器 + 栈切换） | 极小（用户态栈帧切换） |
| **通信方式** | IPC（Pipe、Socket、共享内存等） | 共享内存 + 同步机制 | 共享内存 + 通道（Channel） |
| **隔离性** | 完全隔离，互不影响 | 共享地址空间，一个崩溃影响全部 | 共享地址空间，但协作式调度更可控 |
| **适用场景** | 多进程架构、高隔离性需求 | CPU 密集型、并行计算 | I/O 密集型、高并发 Web 服务 |
| **Java 实现** | `ProcessBuilder` / `Runtime.exec()` | `Thread`、`ThreadPoolExecutor` | `Thread.startVirtualThread()` (JDK 21+) |

#### 实际场景选择

**I/O 密集型（I/O-Bound）—— 首选协程/虚拟线程**

场景：Web 服务器、API Gateway、数据库中间件

每个请求涉及大量的 I/O 等待（网络 I/O、磁盘 I/O），CPU 大部分时间在等待。协程的"阻塞即切换"特性完美解决线程资源浪费。

```java
// 虚拟线程版 HTTP 服务器（JDK 21+）
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 10000).forEach(i -> {
        executor.submit(() -> {
            // 每个请求是一个虚拟线程，阻塞时自动让出平台线程
            HttpClient client = HttpClient.newHttpClient();
            HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create("https://api.example.com/data"))
                .build();
            HttpResponse<String> response = client.send(request, 
                HttpResponse.BodyHandlers.ofString());
            return response.body();
        });
    });
}
```

**CPU 密集型（CPU-Bound）—— 首选线程池**

场景：视频编解码、加密解密、科学计算

线程数不宜超过 CPU 核心数，避免上下文切换带来的额外开销。线程池的 **CachedThreadPool** 或 **FixedThreadPool** 配合核心数配置最为高效。

```java
// CPU 密集型任务 —— 线程池最佳实践
int cores = Runtime.getRuntime().availableProcessors();
ThreadPoolExecutor cpuExecutor = new ThreadPoolExecutor(
    cores,       // corePoolSize = CPU 核心数
    cores * 2,   // maximumPoolSize = 可稍大，处理突发
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000)
);

// 执行 CPU 密集型计算任务
for (int i = 0; i < 100; i++) {
    cpuExecutor.submit(() -> {
        // 纯 CPU 计算，不需要 I/O 等待
        double result = 0;
        for (int j = 0; j < 1_000_000; j++) {
            result += Math.sin(j) * Math.cos(j);
        }
        return result;
    });
}
```

#### Java 虚拟线程学习要点

1. **不适用于 CPU 密集型**：虚拟线程的 yield 发生在阻塞时，纯计算不会 yield 也无优势
2. **不要池化虚拟线程**：每个虚拟线程成本极低，无需池化，`Executors.newVirtualThreadPerTaskExecutor()` 直接创建即可
3. **谨慎使用 ThreadLocal**：虚拟线程数量庞大，ThreadLocal 会导致大量对象残留，增大 GC 压力
4. **synchronized 优化**：JDK 21 对虚拟线程中的 `synchronized` 做了优化，不会固定（pin）平台线程

```java
// 虚拟线程的最佳实践：不要池化，需要即创建
// 推荐
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    //...
}

// 不推荐 —— 虚拟线程池化没有意义
ExecutorService bad = Executors.newFixedThreadPool(100, 
    Thread.ofVirtual().factory()); // 虚拟线程池化，浪费
```

### 第四层：深入思考

#### 1. 为什么进程切换比线程切换慢？

根本原因在于**地址空间切换**。线程只切换执行上下文（寄存器、栈指针），但进程切换需要：
- 切换页表目录基址寄存器（x86 的 CR3 → 刷新全部 TLB > 需要从内存重新加载页表项 > 多次缺页中断
- 进程的调度数据结构（PCB）比线程的 TCB 大得多
- 如果是写时拷贝（Copy-On-Write）的进程 fork，还可能触发缺页异常（Page Fault）

#### 2. 虚拟线程与传统协程（如 Kotlin Coroutine、Go Goroutine）有何不同？

| 特性 | Java 虚拟线程 | Kotlin Coroutine | Go Goroutine |
|------|-------------|-----------------|-------------|
| 语言支持 | JDK 21 内置 | 库支持 | 语言内置 |
| 调度 | JVM 管理（ForkJoinPool） | 协程上下文调度器 | Go 运行时（G-M-P模型） |
| 堆栈 | 可动态增长（类似 OS 线程的栈） | 编译期挂起、状态机 | 可动态增长（初始 2KB） |
| 阻塞语义 | 自动 yield（不阻塞平台线程） | 必须使用 `suspend` 函数 | 自动 yield |

Java 虚拟线程的最大优势是**对现有代码的侵入性极小**——几乎不需要修改代码就能享受协程的并发能力，只需将 `new Thread()` 替换为 `Thread.startVirtualThread()`，将 `newFixedThreadPool` 替换为 `newVirtualThreadPerTaskExecutor`。

#### 3. 百万协程背后的操作系统限制

虽然协程可以轻松创建百万级并发，但底层操作系统资源仍有瓶颈：
- **文件描述符（File Descriptor, FD）**：默认 1024 个/进程，需调大 `ulimit -n 65535`
- **端口号（Port Number）**：一个客户端 IP 可用的临时端口约 28k 个
- **内存**：百万虚拟线程约 2-4 GB（每个约 2-4KB），对 16GB JVM 可接受

#### 4. 未来的并发编程趋势

随着 Java 虚拟线程的成熟，传统的 **Reactor 模型**（Netty、WebFlux）和 **异步回调** 范式将逐渐被虚拟线程的**同步阻塞**写法替代——代码更直观、调试更方便，而性能和吞吐量不输异步模型。Java 的并发编程正在经历一场"返回同步"的革命。
