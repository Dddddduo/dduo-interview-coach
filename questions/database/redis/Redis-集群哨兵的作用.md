# Redis 集群、哨兵的作用

### 🧠 联想记忆法（必须最先输出）

**口诀："监投选通主，槽片协重定" —— 10 字记住两大体系**

**哨兵体系（5 字：监投选通主）**
1. **监** — 监控（Monitoring）：哨兵不断 ping 主从节点
2. **投** — 投票（Voting/Raft）：故障时哨兵集群内部 Raft 投票表决
3. **选** — 选举（Election）：从候选从节点中选出一个新主节点
4. **通** — 通知（Notification）：向客户端发布新的主节点地址
5. **主** — 主从切换（Failover）：自动将主库切换到从库

**集群体系（5 字：槽片协重定）**
1. **槽** — 哈希槽（Hash Slot）：全集群 16384 个槽，数据分片核心
2. **片** — 分片（Sharding）：每个节点负责一部分槽位
3. **协** — 协作协议（Gossip）：节点间通过 Gossip 协议交换元数据
4. **重** — 重定向（Redirect）：客户端请求落错节点时返回 MOVED/ASK
5. **定** — 定主从（Replication）：每个主节点挂载若干从节点保证高可用

**场景联想图：**
```
        哨兵 = 保安队长 👮（盯梢 → 报警 → 投票 → 换头头）
        集群 = 分布式仓库 🏭（分库位 → 货架号 → 内部通讯 → 引导取货）
```

## 第一层：核心概念

### 一、Redis 哨兵（Sentinel）

#### 1.1 哨兵是什么

Redis Sentinel（哨兵）是 Redis 官方提供的高可用性（High Availability, HA）解决方案。它的核心职责是：**当主节点（Master）发生故障时，自动将从节点（Slave/Replica）提升为新的主节点，从而保证服务不中断**。

#### 1.2 哨兵的核心功能

| 功能 | 英文术语 | 说明 |
|------|---------|------|
| 监控 | Monitoring | 哨兵定期向所有节点发送 PING 命令，检测节点是否存活 |
| 通知 | Notification | 当被监控节点发生故障时，哨兵向管理员或其他程序发送通知 |
| 自动故障转移 | Automatic Failover | 主节点不可用时，自动将一个从节点提升为主节点 |
| 配置提供 | Configuration Provider | 客户端连接哨兵获取当前主节点地址 |

哨兵本质上是一个运行在特殊模式下的 Redis 服务器，使用 `redis-sentinel` 命令或 `redis-server --sentinel` 启动。

#### 1.3 哨兵集群的必要性

**单个哨兵存在单点故障（Single Point of Failure, SPOF）问题**——如果唯一的哨兵挂了，整个高可用体系就失效了。因此生产环境中**必须部署至少 3 个哨兵节点组成哨兵集群（Sentinel Cluster）**，使用 Raft 一致性算法（Raft Consensus Algorithm）来达成共识，避免"脑裂"（Split-Brain，即多个哨兵对主节点状态判断不一致）。

---

### 二、Redis 集群（Cluster）

#### 2.1 集群是什么

Redis Cluster（集群）是 Redis 3.0 引入的**分布式（Distributed）** 解决方案，它将数据自动分片（Sharding）到多个节点上，同时提供一定程度的高可用性。

#### 2.2 集群的核心特性

- **数据分片**：自动将数据分散到 16384 个哈希槽（Hash Slot）
- **高可用**：每个主节点可以挂载从节点，主节点故障时从节点自动升级
- **去中心化**：所有节点通过 Gossip 协议（Gossip Protocol）通信，没有中心节点
- **线性扩展**：增加节点即可线性提升集群的存储容量和吞吐量

## 第二层：底层原理

### 一、哨兵底层原理

#### 1.1 监控机制：心跳与主观下线

哨兵每秒向所有节点（主、从、其他哨兵）发送 PING 指令：

```
PING → 节点响应 PONG → 正常
PING → 超过 down-after-milliseconds 无响应 → 主观下线（Subjective Down, SDOWN）
```

当一个哨兵认为主节点 SDOWN 后，它会向其他哨兵发起询问。

#### 1.2 故障判定：从 SDOWN 到 ODOWN

当**足够数量（quorum）** 的哨兵都认为主节点不可用时，判定为**客观下线（Objective Down, ODOWN）**：

```
SDOWN（单个哨兵怀疑）→ 投票询问 → 达到 quorum → ODOWN（集群确认）
```

#### 1.3 Raft 投票选举 Leader 哨兵

判定 ODOWN 后，哨兵集群需要进行领导者选举（Leader Election），选出一个哨兵来执行故障转移。这个过程基于 Raft 算法：

1. 每个哨兵向其他哨兵发送 `SENTINEL is-master-down-by-addr` 请求
2. 收到请求的哨兵如果尚未投票，会投给第一个请求者
3. 获得**超过半数（N/2 + 1）** 票的哨兵成为 Leader
4. Leader 负责执行后续的故障转移

#### 1.4 故障转移（Failover）流程

Leader 哨兵执行以下步骤：

1. 从从节点列表中选择一个合适的从节点
   - 优先选择复制的数据最新的（offset 最大）
   - 优先选择 runid 较小的（稳定运行更久）
2. 向选中的从节点发送 `SLAVEOF NO ONE` 命令，使其成为新主节点
3. 向其他从节点发送 `SLAVEOF <新主IP> <新主端口>`，让它们复制新主节点
4. 更新哨兵内部的主节点映射表
5. 当原主节点恢复上线时，哨兵会将其设置为新主节点的从节点

#### 1.5 哨兵配置文件 sentinel.conf 关键配置项

```conf
# 指定哨兵监听的端口
port 26379

# 监控的主节点信息：sentinel monitor <master-name> <ip> <port> <quorum>
# quorum=2 表示至少 2 个哨兵同意才能判定主节点客观下线
sentinel monitor mymaster 127.0.0.1 6379 2

# 主观下线判定时间（毫秒）：30 秒无响应即认为节点可能挂了
sentinel down-after-milliseconds mymaster 30000

# 故障转移超时时间（毫秒）
sentinel failover-timeout mymaster 180000

# 故障转移时，允许同时同步新主节点的从节点数量
# 1 表示逐个同步，避免主节点负载过高
sentinel parallel-syncs mymaster 1

# 保护模式，生产环境建议开启
sentinel auth-pass mymaster <password>

# 通知脚本（可选）：故障转移时调用
sentinel notification-script mymaster /var/redis/notify.sh
```

#### 1.6 为什么哨兵需要奇数个节点？

因为哨兵基于 **Raft 算法** 要求**超过半数（majority）** 才能选举 Leader。以 3 个哨兵为例，允许挂 1 个；2 个哨兵挂 1 个就只剩 50%，无法达成多数。因此生产环境的标配是 **3 个哨兵（或 5 个）** 节点。

### 二、集群底层原理

#### 2.1 哈希槽（Hash Slot）—— 数据分片的核心

Redis Cluster 将整个数据空间划分为 **16384 个哈希槽**，每个键通过 CRC16 算法计算归属槽位：

```
槽位编号 = CRC16(key) % 16384
```

每个集群节点负责管理一部分连续的槽位：

| 节点 | 负责的槽位范围 |
|------|---------------|
| Node A | 0 ~ 5460 |
| Node B | 5461 ~ 10922 |
| Node C | 10923 ~ 16383 |

为什么是 16384 个槽？因为 Redis 作者分析后认为，16384 个槽在集群规模不超过 1000 个节点时，心跳消息体大小适中，网络开销可控。槽位信息用 2KB 的位图（bitmap）即可表示。

#### 2.2 CRC16 算法

```python
# 伪代码：计算键所属的哈希槽
def get_hash_slot(key):
    # 集群模式下支持 hash tag，用 {} 包裹的部分计算 CRC16
    # 例如：{user1000}.name 和 {user1000}.age 会落入同一槽位
    if '{' in key and '}' in key:
        tag = key[key.index('{')+1:key.index('}')]
        return crc16(tag) % 16384
    return crc16(key) % 16384
```

**Hash Tag** 机制允许将相关键强制映射到同一槽位，从而支持多键操作（如 MSET、事务）。

#### 2.3 Gossip 协议（Gossip Protocol）

Redis Cluster **没有中心节点**，所有节点通过 **Gossip 协议** 交换元数据。Gossip 的本质是**节点间周期性随机通信**：

```
每个节点每秒向 5 个随机节点发送 PING 消息
→ 对方回复 PONG 消息携带自身的槽位信息、节点状态
→ 信息在整个集群中像"流言"一样传播
→ 最终所有节点达成一致（最终一致性，Eventual Consistency）
```

Gossip 消息类型：
| 消息类型 | 作用 |
|---------|------|
| MEET | 邀请新节点加入集群 |
| PING | 探活并交换信息 |
| PONG | 响应 PING，同时广播自身信息 |
| FAIL | 广播节点故障信息 |

#### 2.4 MOVED 重定向 vs ASK 重定向

**这是面试高频考点**。两者都涉及客户端请求被重定向，但语义不同：

##### MOVED 重定向 —— 槽位已永久迁移
```bash
# 客户端请求一个不属于当前节点的键
GET mykey
# 服务端响应：槽位 1234 由节点 192.168.1.2:6380 负责
(error) MOVED 1234 192.168.1.2:6380
# 客户端应更新本地槽位映射缓存，并直接重发请求到正确节点
```

**特点**：永久性（Permanent），客户端应缓存新槽位映射。

##### ASK 重定向 —— 槽位正在迁移中
```bash
# 集群正在进行槽位迁移（resharding），键可能还在源节点
GET mykey
# 如果源节点中没有该键，返回 ASK 重定向
(error) ASK 1234 192.168.1.3:6381
# 客户端先发送 ASKING 命令到目标节点，再重发 GET
ASKING
GET mykey
```

**特点**：临时性（Transient），客户端的槽位映射**不更新**，仅本次请求走目标节点。

##### 二者核心区别

| 维度 | MOVED | ASK |
|------|-------|-----|
| 触发时机 | 槽位已永久分配给新节点 | 槽位正在迁移中 |
| 客户端缓存 | **必须更新**槽位映射表 | **不更新**槽位映射表 |
| 数据位置 | 键一定在目标节点 | 键可能在源节点或目标节点 |
| 后续请求 | 直接发往新节点 | 仍需先问源节点 |

#### 2.5 集群节点配置示例

```conf
# redis-cluster-7000.conf — 节点 1
port 7000
cluster-enabled yes               # 开启集群模式
cluster-config-file nodes-7000.conf  # 集群节点配置文件
cluster-node-timeout 5000         # 节点超时时间（毫秒）
appendonly yes
```

启动集群：
```bash
# 创建 6 个节点（3 主 3 从）
redis-cli --cluster create \
  127.0.0.1:7000 127.0.0.1:7001 127.0.0.1:7002 \
  127.0.0.1:7003 127.0.0.1:7004 127.0.0.1:7005 \
  --cluster-replicas 1
# --cluster-replicas 1 表示每个主节点配 1 个从节点
```

## 第三层：实践应用

### 一、哨兵 vs 集群 —— 核心对比

| 对比维度 | Redis Sentinel | Redis Cluster |
|---------|---------------|--------------|
| **定位** | 高可用（HA） | 分布式 + 高可用 |
| **数据分片** | 不支持，所有数据在同一节点 | 支持，16384 个槽自动分片 |
| **自动故障转移** | 支持（哨兵负责） | 支持（集群内部自动完成） |
| **写入能力** | 单点写入，受限于单机内存 | 多点写入，水平扩展 |
| **存储容量** | 受单机内存限制 | 随节点增加线性扩展 |
| **客户端复杂度** | 低，需通过哨兵获取主节点 | 高，需处理 MOVED/ASK 重定向 |
| **节点数要求** | 至少 3 个哨兵 + 1 主 N 从 | 至少 3 主 3 从（6 节点） |
| **一致性** | 主从异步复制，可能丢失数据 | 异步复制，网络分区可能丢数据 |
| **最小可用节点** | 超过半数哨兵存活 | 超过半数主节点存活 |
| **常用部署规模** | 3 哨兵 + 1 主 2 从（5 节点） | 3 主 3 从 ~ 数百节点 |
| **适用场景** | 数据量可控制在单机范围内 | 数据量超过单机容量，需水平扩展 |

### 二、生产部署建议

```bash
# 1. 哨兵模式架构（数据量 < 100G 时推荐）
#    3 台物理机各部署一个哨兵 + 一个 Redis 实例：
#    主机1: Redis Master + Sentinel A
#    主机2: Redis Slave 1 + Sentinel B
#    主机3: Redis Slave 2 + Sentinel C
#    → 任意一台机器挂掉不影响服务

# 2. 集群模式架构（数据量 > 100G 或写入量 > 单机极限时推荐）
#    建议 6 台起步：3 主 3 从
#    主节点平分槽位：0-5460, 5461-10922, 10923-16383
#    每个主节点的从节点部署在不同物理机
#    → 水平扩展能力强，单点故障可自动切换
```

### 三、面试高频追问

**Q：集群模式下执行 MGET 跨槽位操作会怎样？**
A：如果键分布在不同的槽位，Redis Cluster 不支持跨槽位事务或多键操作，除非键都在同一槽位（使用 Hash Tag）。客户端需要分批次请求。

**Q：哨兵模式下客户端如何感知主节点变化？**
A：客户端通过订阅哨兵的 `+switch-master` 频道（Channel）实时接收主节点变更通知，或定期调用 `SENTINEL get-master-addr-by-name` 获取当前主节点。

**Q：数据丢失场景有哪些？**
A：集群主从异步复制导致：主节点挂了但数据还未同步到从节点；哨兵模式下，主节点被判定下线后新的写入丢失。

## 第四层：深入思考

### 一、架构演进路线

```
单机 Redis                    →    哨兵（Sentinel）              →    集群（Cluster）
┌────────────┐                    ┌────────────────────┐           ┌──────────────────┐
│  Master    │                    │  Sentinel A/B/C    │           │  Node1(主) Node2(从)
│  SPOF      │                    │  ↓监控              │           │  Node3(主) Node4(从)
│  不可扩展  │                    │  Master + Slaves    │           │  Node5(主) Node6(从)
└────────────┘                    └────────────────────┘           └──────────────────┘
     ↓ 需要高可用                        ↓ 数据量超单机
     → 引入哨兵                          → 引入集群
```

**演进逻辑**：
1. **单机时代**：数据量小，一个 Redis 实例够用，挂了就挂了
2. **哨兵时代**：业务要求 7×24 高可用，有了哨兵自动切换，但容量受限于单机
3. **集群时代**：数据量爆炸（百 GB 甚至 TB 级别），必须分片，集群天然提供高可用

### 二、哨兵与集群能否共存？

一个常见面试追问：「集群模式下还需要哨兵吗？」

**答案：不需要，集群自带故障转移能力。**

集群模式下，每个主节点挂载从节点，当主节点故障时，集群内部会（通过 Gossip 协议传播故障信息，超过半数主节点确认后）自动将从节点提升。**集群的故障转移是内置的，不需要外部哨兵介入。**

唯一例外：如果你使用的是 **Redis Cluster 的代理层方案（如 Codis、Twemproxy）**，这些代理层可能不处理故障转移，此时可能需要额外组件。

### 三、CAP 理论的取舍

| 组件 | 一致性（Consistency） | 可用性（Availability） | 分区容忍性（Partition Tolerance） |
|------|:---:|:---:|:---:|
| 哨兵模式 | 弱（异步复制） | 高（自动切换） | 一般 |
| 集群模式 | 弱（异步复制） | 高（自动分片+切换） | 强（去中心化） |

Redis 的设计哲学是**优先保证可用性和分区容忍性（AP）**，在一致性上选择最终一致性（Eventual Consistency）。这是它与 ZooKeeper、etcd 等强一致性系统的主要区别，也是面试中常见的对比话题。
