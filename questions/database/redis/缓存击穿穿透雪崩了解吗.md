# 缓存击穿、穿透、雪崩了解吗？请深入讲解三个概念的区别和解决方案

---

## 🧠 联想记忆法

将三个"缓存灾难"想象成**攻击一座城堡（数据库）**的三种方式：

| 概念 | 记忆锚点 | 口诀 |
|------|---------|------|
| **穿透**（Penetration） | 刺客用**透**视眼专找不存在的弱点攻击——**穿透 = 查不存在的数据** | "透=不存在，布隆挡门外" |
| **击穿**（Breakdown） | 攻城锤集中一点**击穿**城墙——**击穿 = 单个热点 key 过期** | "击穿=单点，互斥锁排队" |
| **雪崩**（Avalanche） | 整座城墙同时坍**塌**，如雪**崩**般连锁——**雪崩 = 大量 key 同时过期** | "雪崩=大片，TTL加随机" |

**一句话记忆**："**透**（不存在）→ 布隆过滤器；**击**（单点过期）→ 互斥锁；**雪**（批量过期）→ TTL 随机"。

---

## 📖 深度解答

### 一、核心概念

#### 1. 缓存穿透（Cache Penetration）

**定义**：查询一个**数据库中也不存在**的数据。由于缓存未命中（Cache Miss），每次请求都穿透缓存直达数据库（Database），导致数据库压力剧增。

**典型场景**：恶意攻击者持续请求 `id = -1` 或 `id = "abcd"` 等不可能存在的 key，缓存永远无法命中，数据库被无效查询淹没。

**后果**：轻则 DB 负载升高、响应变慢；重则 DB 连接池耗尽，服务宕机。

#### 2. 缓存击穿（Cache Breakdown / Hotspot Invalid）

**定义**：一个**热点 key（Hot Key）**在过期（Expire）的瞬间，大量并发请求同时涌入，缓存无法提供服务，所有请求直达数据库。

**典型场景**：微博热搜某条新闻的缓存过期，数百万用户同时刷新，数据库瞬间被打满。

**后果**：数据库瞬间承受数倍于平时的 QPS，可能触发连接超时（Connection Timeout）甚至宕机。

#### 3. 缓存雪崩（Cache Avalanche）

**定义**：**大量 key 同时过期**或 **Redis 节点宕机（Downtime）**，导致缓存层面大面积失效（Massive Cache Invalidation），所有请求全部落到数据库，引发连锁崩溃。

**典型场景**：双十一零点，所有商品的缓存 TTL 统一设置为 1 小时，整点同时过期 → 数据库被峰值流量冲垮。或 Redis 主节点宕机，未及时切换，整个缓存层不可用。

**后果**：数据库被瞬间击垮，可能引发服务雪崩——上游服务超时重试 → 更多请求堆积 → 系统整体瘫痪。

---

### 二、底层原理

#### 缓存穿透的根源

缓存系统遵循经典的"**Cache-Aside 模式**"：

```
请求 → 查缓存 → Miss → 查 DB → 回写缓存 → 返回
```

当 key 在 DB 中不存在时，缓存永远不会有该 key，因此每次请求都走完整链路，DB 承受全部流量。攻击者可以利用这个机制，构造大量不存在的 key，让缓存完全失效。

**时间/空间复杂度**：每次穿透需要一次 Redis GET（O(1)）+ 一次 DB 查询（O(1) 或 O(log N)），但 DB 的并发能力远低于 Redis（Redis 可达 10万+ QPS，MySQL 单机约 3000-5000 QPS），穿透带来的 DB 压力放大效应是致命的。

#### 缓存击穿的根源

Redis 的过期机制是**被动过期（Lazy Expiration）** + **主动采样过期（Active Expiration）**：

- **被动过期**：当 key 被访问时，检查是否已到期，到期则删除。
- **主动过期**：每 100ms 随机采样 20 个有过期时间的 key，删除其中已过期的，如果超过 25% 已过期则继续采样。

热点 key 在 TTL 归零时被 Redis 删除（或标记删除），下一个请求发现缓存不存在，回源 DB。如果此时有成千上万个并发请求同时到达，就形成了"**惊群效应（Thundering Herd Problem）**"——所有线程同时发现缓存 miss，同时冲向 DB，DB 在毫秒级被击穿。

#### 缓存雪崩的根源

雪崩分为两种触发模式：

| 模式 | 原因 | 特征 |
|------|------|------|
| **时间同步型** | 批量 key 设置了相同的 TTL | 可预测，周期性发生 |
| **节点故障型** | Redis 实例宕机、网络分区 | 不可预测，突发性 |

二者本质都是**缓存覆盖率从高骤降为 0**，DB 承接全部压力。与击穿的区别在于：击穿是单点失效，雪崩是大面积失效。

### 三、实践应用（含 Java 代码示例）

#### 3.1 缓存穿透解决方案

**方案 A：布隆过滤器（Bloom Filter）**

**原理**：用一个极小的 bit 数组 + 多个哈希函数（Hash Functions）判断 key 是否**可能**存在。不存在则一定不存在，存在则可能存在（有假阳性 False Positive）。

**空间优势**：存储 100 万个元素，1% 假阳性率，只需约 1.2MB 内存。

```java
import com.google.common.hash.BloomFilter;
import com.google.common.hash.Funnels;
import java.nio.charset.Charset;

public class CachePenetrationBloomFilter {
    // 预期数据量 100 万，假阳性率 1%
    private static final BloomFilter<String> bloomFilter =
        BloomFilter.create(Funnels.stringFunnel(Charset.defaultCharset()),
                           1_000_000, 0.01);

    // 初始化时加载所有合法 ID 到布隆过滤器
    public static void init(List<String> allValidIds) {
        allValidIds.forEach(bloomFilter::put);
    }

    public String get(String key) {
        // 布隆过滤器前置判断（O(k)，k为哈希函数个数）
        if (!bloomFilter.mightContain(key)) {
            return null; // 一定不存在，直接返回
        }
        // 可能存在，查缓存
        String value = redis.get(key);
        if (value != null) return value;
        // 查数据库
        value = db.query(key);
        if (value != null) {
            redis.set(key, value, 300);
        }
        return value;
    }
}
```

**方案 B：缓存空值（Cache Null Object）**

对查不到的数据，在缓存中写入一个短过期时间的空值标记，防止反复穿透。

```java
public String getWithNullCache(String key) {
    String value = redis.get(key);
    if (value != null) {
        return "NULL_PLACEHOLDER".equals(value) ? null : value;
    }
    value = db.query(key);
    if (value == null) {
        // 缓存空值，30 秒过期，防止恶意穿透
        redis.set(key, "NULL_PLACEHOLDER", 30);
    } else {
        redis.set(key, value, 300);
    }
    return value;
}
```

**方案 C：参数校验**：在 Controller 层对 `id` 格式、范围做校验，非法请求直接拒绝。

```java
@GetMapping("/user/{id}")
public Result getUser(@PathVariable String id) {
    // 参数校验：非数字ID直接拒绝
    if (!StringUtils.isNumeric(id)) {
        return Result.fail("非法参数");
    }
    // 业务校验：ID范围
    long userId = Long.parseLong(id);
    if (userId <= 0) {
        return Result.fail("ID必须为正数");
    }
    return userService.getUser(userId);
}
```

#### 3.2 缓存击穿解决方案

**方案 A：互斥锁（Mutex Lock）/ 分布式锁**

**核心思想**：只允许一个线程去查 DB 更新缓存，其他线程等待或降级。

```java
public String getWithLock(String key) {
    String value = redis.get(key);
    if (value != null) return value;

    // 分布式锁，只允许一个线程去查 DB
    String lockKey = "lock:" + key;
    String requestId = UUID.randomUUID().toString();
    boolean locked = redis.setnx(lockKey, requestId, 10, TimeUnit.SECONDS);

    if (locked) {
        try {
            // double-check：拿到锁后再次检查缓存（防止等待期间已被更新）
            value = redis.get(key);
            if (value != null) return value;

            value = db.query(key);
            redis.set(key, value, 300);
            return value;
        } finally {
            // 只释放自己加的锁，防止误删其他人的锁
            if (requestId.equals(redis.get(lockKey))) {
                redis.del(lockKey);
            }
        }
    } else {
        // 没拿到锁的线程：等待后重试
        Thread.sleep(50);
        return getWithLock(key); // 递归重试
    }
}
```

**方案 B：逻辑过期（Logical Expiration）**

**核心思想**：热点 key 不设物理 TTL，而是在 value 中嵌入一个逻辑过期时间字段，后台线程异步刷新。用"最终一致性"换取"高可用性"。

```java
public class CacheWrapper<T> {
    private T data;
    private long expireTime; // 逻辑过期时间戳（毫秒）
}

public T getWithLogicalExpire(String key, Function<String, T> dbLoader) {
    CacheWrapper<T> wrapper = redis.get(key);
    if (wrapper == null) {
        // 缓存不存在（如首次），直接回源
        T data = dbLoader.apply(key);
        redis.set(key, new CacheWrapper<>(data, System.currentTimeMillis() + 60000));
        return data;
    }

    if (wrapper.getExpireTime() > System.currentTimeMillis()) {
        return wrapper.getData(); // 逻辑未过期，直接返回
    }

    // 逻辑已过期：加锁后异步刷新（只让一个线程去更新）
    String lockKey = "renew:" + key;
    if (redis.setnx(lockKey, "1", 3, TimeUnit.SECONDS)) {
        // 异步线程去更新缓存
        executor.submit(() -> {
            try {
                T newData = dbLoader.apply(key);
                redis.set(key,
                    new CacheWrapper<>(newData,
                        System.currentTimeMillis() + 60000));
            } finally {
                redis.del(lockKey);
            }
        });
    }

    // 返回过期的旧数据（牺牲一致性，保证可用性）
    return wrapper.getData();
}
```

**方案 C：物理永不过期**：对明确的热点 key（如首页 Banner、热搜榜），不设 TTL，由后台定时任务（Cron Job）或变更事件主动更新。

#### 3.3 缓存雪崩解决方案

**方案 A：TTL + 随机值**

**核心思想**：在基础 TTL 上增加一个随机偏移量，让 key 的过期时间均匀分布，避免同时过期。

```java
public void setWithRandomTtl(String key, String value, int baseTtl) {
    // 基础 TTL 上增加 0~600 秒的随机偏移
    int randomTtl = baseTtl + ThreadLocalRandom.current().nextInt(600);
    redis.set(key, value, randomTtl, TimeUnit.SECONDS);
}

// 批量设置时使用
public void batchSet(Map<String, String> kvMap, int baseTtl) {
    kvMap.forEach((key, value) -> setWithRandomTtl(key, value, baseTtl));
}
```

**方案 B：Redis 高可用架构**

| 架构 | 说明 | 自动故障转移 |
|------|------|------------|
| 主从复制（Replication） | 一主多从，读写分离 | 需手动哨兵 |
| 哨兵（Sentinel） | 监控 + 通知 + 自动故障转移 | ✅ |
| 集群（Cluster） | 分片 + 高可用，无中心节点 | ✅ 内置 |

**方案 C：多级缓存（Multi-Level Cache）**

```
[客户端] → [本地缓存 Caffeine (L1)] → [Redis 缓存 (L2)] → [数据库 (L3)]
```

本地缓存不受 Redis 宕机影响，提供兜底保护。

```java
// 使用 Caffeine 作为一级缓存
Cache<String, String> localCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .build();

public String getWithMultiLevel(String key) {
    // L1: 本地缓存（无网络开销，纳秒级响应）
    String value = localCache.getIfPresent(key);
    if (value != null) return value;

    // L2: Redis（毫秒级网络开销）
    value = redis.get(key);
    if (value != null) {
        localCache.put(key, value);
        return value;
    }

    // L3: 数据库（慢，十几毫秒到百毫秒级）
    value = db.query(key);
    if (value != null) {
        redis.set(key, value,
            300 + ThreadLocalRandom.current().nextInt(600));
        localCache.put(key, value);
    }
    return value;
}
```

**方案 D：限流熔断**：使用 Sentinel 或 Hystrix 对 DB 访问做限流（Rate Limiting）和熔断（Circuit Breaking）。

```java
// Sentinel 限流注解示例
@SentinelResource(value = "dbQuery",
    fallback = "dbQueryFallback",
    blockHandler = "dbQueryBlockHandler")
public String queryDb(String key) {
    return db.query(key);
}

// 熔断降级：返回默认值或缓存旧值
public String dbQueryFallback(String key, Throwable t) {
    return localCache.getIfPresent(key); // 缓存兜底
}
```

### 四、深入思考

#### 4.1 三者的本质联系

三个问题本质上都源于"**缓存和数据库之间的速度差异 + 并发请求的放大效应**"：

| 维度 | 穿透 | 击穿 | 雪崩 |
|------|------|------|------|
| **失效维度** | 空间上的不存在 | 时间上的单点失效 | 时间/空间上的大规模失效 |
| **失效范围** | 始终失效（永不过期） | 短暂失效（TTL 瞬间） | 持续或大面积失效 |
| **并发模型** | 持续低频冲击 | 瞬间高频冲击 | 持续大规模冲击 |

#### 4.2 布隆过滤器的局限性及优化

- **不可删除**：标准布隆过滤器不支持删除元素。可用 **Counting Bloom Filter**（每个 bit 扩展为计数器）或 **Redis Bloom Module**（`BF.ADD` / `BF.EXISTS`）解决。
- **假阳性率**：1% 假阳性率意味 100 个穿透请求中有 1 个漏网。可通过增大 bit 数组降低。
- **动态扩容**：新增合法 key 需要重建布隆过滤器，可使用分片布隆过滤器（Sharded Bloom Filter）或 Redis 的 BF 模块。

#### 4.3 互斥锁 vs 逻辑过期的选择

| 对比维度 | 互斥锁（Mutex） | 逻辑过期（Logical Expire） |
|---------|----------------|--------------------------|
| 一致性（Consistency） | 强一致性（Strong） | 最终一致性（Eventual） |
| 可用性（Availability） | 锁竞争可能导致少量等待 | 无等待，立即返回 |
| 实现复杂度 | 需处理锁超时、锁误删 | 需额外异步线程池 |
| 适用场景 | 对一致性要求高（如库存） | 对响应时间敏感（如首页） |
| 缓冲区压力 | 小（只有单线程查DB） | 小（异步查DB） |

这本质上是一个 **CAP 理论在缓存层面的权衡**——互斥锁选择了 Consistency（C），逻辑过期选择了 Availability（A）。

#### 4.4 雪崩的多层防御体系

单一方案不足以应对雪崩。推荐构建"**预防 + 隔离 + 兜底**"三层防御网：

```
                    ┌─────────────────────┐
    预防层          │ TTL 随机化          │
                    │ 缓存预热（Warming）  │
                    │ Key 分散存储        │
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
    隔离层          │ 多级缓存（L1+L2）    │
                    │ Redis 高可用（哨兵） │
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
    兜底层          │ 限流（Rate Limiting）│
                    │ 熔断（Circuit Break) │
                    │ 降级返回默认值       │
                    └─────────────────────┘
```

#### 4.5 面试升华点

- **Redis 过期策略**：提及 Redis 的两种过期机制（Lazy + Active），分析为何过期瞬间会有并发问题。
- **缓存粒度**：穿透通常配合细粒度缓存（单 key），雪崩常出现在粗粒度缓存（批量 key）的场景。
- **实际运维**：提到用 Redis `MONITOR` 或 `SLOWLOG` 定位问题，或使用 `redis-cli --bigkeys` 分析热点 key。

|------|---------|---------|---------|

## 🗺️ 回答思路

面试中回答本题的**最佳讲述顺序**：

### Step 1：总起（10秒）

> "缓存穿透、击穿、雪崩是分布式缓存系统的三个经典异常场景，本质都是缓存失效后大量请求直达数据库的问题，但各自的成因和解决方案不同。下面我逐一讲解。"

### Step 2：逐一讲解（120秒）

**缓存穿透**：
> "穿透是查询不存在的数据。布隆过滤器用极小的内存开销前置判断 key 是否存在——不存在直接拒绝，避免查 DB。空值缓存是对布隆的补充，针对合法但为空的数据。"

**缓存击穿**：
> "击穿是单点热点 key 过期。我倾向用逻辑过期方案——不设物理 TTL，嵌入逻辑过期时间，后台异步刷新。这样既避免了锁的等待开销，又能保证热点数据始终有旧数据可用。"

**缓存雪崩**：
> "雪崩是大面积缓存失效。最直接的手段是 TTL 加随机值，让过期时间均匀分布。同时我会在架构层面部署多级缓存（Caffeine + Redis），确保 Redis 宕机时本地缓存还能兜底。"

### Step 3：对比总结（20秒）

> "一句话区分三者：穿透是'不存在'，击穿是'单点过期'，雪崩是'批量过期'。穿透用布隆挡在门外，击穿用锁或逻辑过期控制并发，雪崩用 TTL 随机化和多级缓存分散风险。"

### Step 4：升华（30秒）

> "更深一层看，这三个问题都反映了缓存系统的核心矛盾：缓存失效的瞬间，DB 要承接全部流量。解决方案的本质是'削峰填谷'——布隆过滤器和空值缓存是'削峰'，互斥锁是'排队填谷'，多级缓存是'分层削峰'。设计缓存系统时，不能只考虑正常情况，必须针对这三大异常场景做防御性设计。"

### 加分技巧

- **结合项目经验**："我们曾经在双十一大促前，通过分析 Redis 慢查询日志发现了一批 TTL 完全相同的 key，立即用脚本来批量加随机偏移，避免了潜在的雪崩风险。"
- **布隆过滤器参数**："Guava 的 BloomFilter `create` 方法默认 3 个哈希函数，假阳性率 3%。我们通常会传入 0.01（1%），此时每个元素需要约 10 bits 空间。"
- **Redis 4.0+ 特性**："Redis 4.0 引入了 lazy-free 机制（`UNLINK` 命令），可以在主动删除大 key 时避免阻塞主线程。"
- **监控告警**："我们为缓存命中率（Cache Hit Ratio）设置了告警，当命中率突然从 95% 降至 50% 以下时自动告警并触发限流。"
