# 如何保证信息不重复消费 — 消息幂等性方案深度解答

### 🧠 联想记忆法

谐音联想为"唯红状乐观表"——唯有红色状态乐观地看表：

- **状**（状态机）→ 业务状态不可逆

**关联知识**: 这本质上是"分布式系统的三大难题"之一——**Exactly-Once Processing（精确一次处理）**。如果你熟悉 TCP 的 ACK 机制或 HTTP 的幂等性（`PUT` vs `POST`），可以把消息幂等理解为同一逻辑在分布式环境下的延伸。

### 📖 深度解答

#### 1. 核心概念（是什么）

**消息幂等性（Message Idempotency）** 是指：消费者对同一条消息的多次消费产生的结果，与消费一次的结果完全相同。

在分布式消息系统中，这是一个无法回避的问题。消息队列（Message Queue, MQ）为了保证消息**不丢失**（Reliability），普遍采用"至少一次语义"（At-Least-Once Delivery），即生产者（Producer）发送消息后，若未收到 Broker 的 ACK 则会重试；消费者（Consumer）处理完成后若 ACK 因网络原因丢失，Broker 会重新投递。这两个环节的**重试机制（Retry Mechanism）** 共同导致了**重复消息（Duplicate Message）** 的必然性。

关键在于：**MQ 无法在协议层消除重复，只能由消费端通过幂等设计来保证最终一致性（Eventual Consistency）**。

#### 2. 底层原理（为什么）

问题的根因在于分布式系统的两个基础理论：

**（1）FLP 不可能原理（FLP Impossibility）**
在异步通信的分布式系统中，即使只有一个节点故障，也不存在一个确定的算法能保证所有节点达成共识。MQ 的"至少一次"是这一理论在工程上的妥协——系统无法 100% 精确地知道一条消息是否被处理。

**（2）网络不可靠性（Network Unreliability）**
网络分区（Network Partition）和延迟抖动（Latency Jitter）导致 ACK 丢失，使得 MQ Broker 无法区分"消费者已处理但 ACK 丢失"和"消费者尚未处理"这两种状态。Broker 只能保守地选择重新投递。

```
生产者 → [发送消息] → Broker → [投递消息] → 消费者
   ↑                      ↓                      ↓
   └── 重试(未收到ACK)     └── 重试(未收到ACK)     └── 可能重复处理
```

因此，**幂等（Idempotent）** 成为消费端必须承担的责任，而非 MQ 可以提供的保障。

#### 3. 实践应用（怎么用）

以下介绍五种主流方案及其 Java 实现。

##### 方案一：数据库唯一键约束（Unique Key Constraint）

核心思路：利用数据库表的唯一索引（Unique Index）天然防重。当重复消息插入时会抛出 `DuplicateKeyException`，捕获后直接视为已处理。

```java
// 1. 消息记录表（建表时设置唯一键）
CREATE TABLE `message_consume_log` (
  `id` BIGINT AUTO_INCREMENT,
  `message_id` VARCHAR(64) NOT NULL COMMENT '消息唯一ID',
  `biz_key` VARCHAR(64) NOT NULL COMMENT '业务键（如订单号）',
  `status` TINYINT DEFAULT 0,
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY `uk_msg_biz` (`message_id`, `biz_key`)  -- 联合唯一索引
);

// 2. Java 消费逻辑
@Service
public class OrderConsumer {
    @Autowired
    private JdbcTemplate jdbcTemplate;
    
    @Transactional(rollbackFor = Exception.class)
    public void consumeMessage(Message msg) {
        try {
            jdbcTemplate.update(
                "INSERT INTO message_consume_log(message_id, biz_key, status) VALUES(?, ?, 0)",
                msg.getMessageId(), msg.getBizKey()
            );
        } catch (DuplicateKeyException e) {
            log.info("重复消息已忽略: {}", msg.getMessageId());
            return;
        }
        processOrder(msg.getBizKey());
        jdbcTemplate.update(
            "UPDATE message_consume_log SET status = 1 WHERE message_id = ?",
            msg.getMessageId()
        );
    }
}
```

**关键点**：`INSERT` 操作必须和业务操作在同一个数据库事务（Database Transaction）中，否则可能出现 INSERT 成功但业务处理失败的情况。

##### 方案二：Redis 去重（SET NX）

使用 Redis 的 `SET NX` 命令（SET if Not eXists），配合过期时间（TTL）实现轻量级去重。

```java
@Service
public class RedisDedupConsumer {
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String DEDUP_PREFIX = "dedup:msg:";
    private static final long TTL_SECONDS = 7 * 24 * 3600;
    
    public boolean consumeMessage(Message msg) {
        String dedupKey = DEDUP_PREFIX + msg.getMessageId();
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(dedupKey, "1", Duration.ofSeconds(TTL_SECONDS));
        
        if (Boolean.FALSE.equals(success)) {
            log.info("重复消息(Redis命中): {}", msg.getMessageId());
            return true;
        }
        try {
            doBizProcess(msg);
            return true;
        } catch (Exception e) {
            redisTemplate.delete(dedupKey);
            throw e;
        }
    }
}
```

**注意事项**：
- TTL（Time-To-Live，过期时间）需要合理设置，过短可能导致 TTL 内未处理完的消息被重复处理；过长则浪费内存。
- Redis 存在数据丢失风险（宕机未持久化），极端情况下可能降级为重复消费。

##### 方案三：业务状态机（State Machine）

利用业务状态的单向流转特性实现幂等。以订单状态为例：`待支付(WAIT_PAY)` → `已支付(PAID)` → `已发货(SHIPPED)` → `已完成(DONE)`。状态不可逆向。

```java
public enum OrderStatus {
    WAIT_PAY, PAID, SHIPPED, DONE, CANCELLED;
    
    public boolean canTransitionTo(OrderStatus target) {
        switch(this) {
            case WAIT_PAY: return target == PAID || target == CANCELLED;
            case PAID:     return target == SHIPPED || target == CANCELLED;
            case SHIPPED:  return target == DONE;
            case DONE:
            case CANCELLED: return false;
            default: return false;
        }
    }
}

@Service
public class StateMachineConsumer {
    @Autowired
    private OrderRepository orderRepository;
    
    @Transactional
    public void handlePaymentMessage(String orderId, PaymentEvent event) {
        Order order = orderRepository.findById(orderId);
        OrderStatus currentStatus = order.getStatus();
        
        if (!currentStatus.canTransitionTo(OrderStatus.PAID)) {
            log.info("订单状态已是 {}，忽略重复支付消息", currentStatus);
            return;
        }
        order.setStatus(OrderStatus.PAID);
        order.setPaidTime(event.getTimestamp());
        orderRepository.save(order);
    }
}
```

**优势**：这是最优雅的方案——不需要额外的去重存储，完全依托于业务逻辑的不可逆性。

##### 方案四：乐观锁（Optimistic Locking with Version Number）

在数据库表中增加版本号（version）字段，每次更新时检查版本号是否匹配。

```java
CREATE TABLE `account_balance` (
  `account_id` VARCHAR(32) PRIMARY KEY,
  `balance` DECIMAL(18,2) NOT NULL,
  `version` INT NOT NULL DEFAULT 0
);

@Repository
public class AccountRepository {
    @Transactional
    public boolean deductBalance(String accountId, BigDecimal amount, int expectedVersion) {
        int affectedRows = jdbcTemplate.update(
            "UPDATE account_balance SET balance = balance - ?, " +
            "version = version + 1 WHERE account_id = ? AND version = ?",
            amount, accountId, expectedVersion
        );
        return affectedRows > 0;
    }
}
```

##### 方案五：消息表法（Message Table）

在业务数据库中创建一张消息消费记录表，每次消费前先查询该表。

```java
@Service
public class MessageTableConsumer {
    @Autowired
    private MessageLogMapper messageLogMapper;
    
    @Transactional
    public void consume(Message msg) {
        MessageLog log = messageLogMapper.selectByMessageId(msg.getMessageId());
        if (log != null) {
            log.info("已处理过: {}", msg.getMessageId());
            return;
        }
        messageLogMapper.insert(new MessageLog(msg.getMessageId(), msg.getBizKey()));
        doBiz(msg);
    }
}
```

**注意**：SELECT 查不到不代表一定没有重复，因为存在"幻读"（Phantom Read）窗口。该方法通常与唯一索引配合使用。

#### 4. 方案对比与深入思考

##### 方案对比表

| 方案 | 可靠性 | 性能 | 复杂度 | 适用场景 | 风险 |
|------|--------|------|--------|----------|------|
| 唯一键约束 | ★★★★★ | ★★★ | ★ | 数据库直连场景 | 场景受限，需要唯一标识 |
| Redis 去重 | ★★★ | ★★★★★ | ★★ | 高吞吐、可接受少量丢失 | TTL 内宕机丢数据 |
| 状态机 | ★★★★★ | ★★★★★ | ★★★ | 业务状态流转明确 | 无法处理状态不明确的场景 |
| 乐观锁 | ★★★★ | ★★★ | ★★ | 账户、库存等更新场景 | 并发高时大量重试 |
| 消息表 | ★★★★★ | ★★ | ★★ | 对可靠性要求最高 | 性能开销大 |

##### 深入思考

**（1）分布式事务与幂等的关系**：幂等是分布式事务（Distributed Transaction）的基石。在 TCC（Try-Confirm-Cancel）模式中，Confirm 和 Cancel 操作必须幂等。

**（2）雪花算法与消息唯一 ID**：生成全局唯一消息 ID（Globally Unique ID）是实现幂等的前提。业界常用雪花算法（Snowflake Algorithm）。

**（3）最终人工介入**：对于金融级场景，当自动去重方案都失效时，必须保留**对账机制（Reconciliation）**。

**（4）幂等 vs 去重**：**去重（Deduplication）** 是把重复的消息丢弃；**幂等（Idempotency）** 是允许重复消息进入，但保证处理结果一致。
